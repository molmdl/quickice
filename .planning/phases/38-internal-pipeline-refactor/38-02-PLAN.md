<objective>
Refactor `_build_molecule_index` to use metadata-driven guest molecule identification from HydrateConfig instead of pattern matching on atom/residue names.

Purpose: This is PIPE-01 — the core refactoring goal. Pattern matching on atom names (`if atom == "Me"`, `if residue == "THF"`, etc.) is fragile and breaks for new guest types. Metadata-driven identification uses the guest_atom_labels sequence from HydrateConfig to identify guest molecules generically.

Output: `_build_molecule_index` accepts HydrateConfig (or its guest metadata) and identifies molecules by matching atom sequences against known patterns, enabling custom guest support without code changes.
</objective>

<feature>
  <name>Metadata-driven molecule identification in _build_molecule_index</name>
  <files>quickice/structure_generation/hydrate_generator.py, tests/test_build_molecule_index.py</files>
  <behavior>
    Given atom_names, positions, residue_names, residue_seq_nums, and HydrateConfig (with guest_atom_labels and guest_atom_count):
    
    Cases:
    1. Water (TIP4P): atom starts with "OW" + next 3 match ["OW","HW1","HW2","MW"] → MoleculeIndex(i, 4, "water")
    2. 3-site water: atom starts with "O" + next 2 match ["O","H","H"] → MoleculeIndex(i, 3, "water")
    3. Guest molecule: atom sequence starting at i matches guest_atom_labels (config.guest_atom_labels) for guest_atom_count atoms → MoleculeIndex(i, guest_atom_count, config.guest_type)
    4. Guest molecule (residue grouping): If residue_names[i] matches the GenIce2 residue name for this guest type, use residue_seq_nums to group all atoms with same seq num → MoleculeIndex(start, count, config.guest_type)
    5. Ions: "NA"/"Na" → (i, 1, "na"), "CL"/"Cl" → (i, 1, "cl")
    6. Unknown: any unmatched atom → (i, 1, "unknown") with warning
    
    Key design: Guest identification checks guest_atom_labels FIRST (before water), so that guest oxygen atoms (e.g., THF's "O") are not misidentified as water. The first atom of guest_atom_labels serves as the "signature" for quick identification.
    
    For built-in guests like THF where GenIce2 uses residue grouping (same residue seq num), the residue-based grouping is the PREFERRED path. When residue_names are available and match the expected guest residue name, group by residue_seq_nums (as current code does for THF). When residue_names are NOT available (e.g., after interface conversion), fall back to atom-label sequence matching.
    
    The expected GenIce2 residue name for a guest type is the guest_type.upper() (e.g., "THF" for thf, "CH4" for ch4). This is stored in GUEST_MOLECULES[guest_type] but can be derived from guest_type.upper() for simplicity.
  </behavior>
  <implementation>
    Refactor _build_molecule_index to accept a `config` parameter (HydrateConfig) with default None for backward compatibility.
    
    When config is provided:
    1. Extract guest_atom_labels, guest_atom_count, guest_type from config
    2. Build a lookup: first atom of guest_atom_labels → (guest_type, guest_atom_count, full guest_atom_labels list)
    3. In the main loop, check guest signature BEFORE water:
       - If residue_names[i] == guest_type.upper() and residue_seq_nums available: group by residue_seq_nums → MoleculeIndex(start, count, guest_type)
       - Else if atom_names[i] matches first label of guest_atom_labels AND next guest_atom_count-1 atoms match the rest: → MoleculeIndex(i, guest_atom_count, guest_type)
    4. Then check water patterns (unchanged)
    5. Then check ions (unchanged)
    6. Unknown falls through (unchanged)
    
    When config is None (backward compatibility):
    - Fall back to current pattern-matching behavior (unchanged)
    
    Update generate() to pass config=self._config (store config as self._config in generate method) to _build_molecule_index.
    
    Remove the hardcoded "THF" residue check and "Me" united-atom methane check — these are replaced by the generic metadata-driven path.
  </implementation>
</feature>
