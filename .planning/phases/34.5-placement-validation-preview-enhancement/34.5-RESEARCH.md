# Phase 34.5: Placement Validation & Preview Enhancement - Research

**Researched:** 2026-05-08
**Domain:** VTK visualization, placement validation, molecular simulation UI
**Confidence:** HIGH (verified against existing codebase and VTK documentation)

## Summary

Phase 34.5 needs to add placement validation and preview to custom molecule insertion. The existing architecture already has sophisticated overlap checking via scipy cKDTree in CustomMoleculeInserter. The challenge is providing a feasible preview given the REQUIREMENTS.md constraint that "Placement preview before commit" is OUT OF SCOPE for computational reasons on large systems.

Research reveals a **scope-appropriate approach**: Single-molecule preview with bounds validation and overlap checking for the proposed position, NOT full system simulation. This is computationally trivial (checking one molecule against existing atoms) and provides essential user feedback without violating the "expensive for large systems" constraint.

**Primary recommendation:** Implement single-molecule placement validation (bounds check + overlap check) with optional visual preview of the proposed position. Do NOT attempt full system preview with all molecules placed.

### Pattern 1: Single-Molecule Placement Validation
**What:** Validate proposed placement before insertion by checking bounds and overlaps for ONE molecule
**When to use:** Custom placement mode when user specifies position/rotation
**Example:**
```python
# Source: Derived from existing CustomMoleculeInserter._check_overlap() pattern
def validate_single_placement(
    self,
    structure: InterfaceStructure,
    position: tuple[float, float, float],
    rotation: tuple[float, float, float],
) -> ValidationResult:
    """Validate placement for single molecule without full insertion.
    
    Returns validation result with:
    - is_valid: bool
    - within_bounds: bool (position in liquid region)
    - has_overlap: bool (any atom too close to existing atoms)
    - min_distance: float (minimum distance to nearest existing atom)
    - error_message: str | None
    """
    # 1. Check bounds (is position in liquid region?)
    ice_atom_count = structure.ice_atom_count
    water_atom_count = structure.water_atom_count
    liquid_start = ice_atom_count
    liquid_end = ice_atom_count + water_atom_count
    
    liquid_positions = structure.positions[liquid_start:liquid_end]
    min_coords = liquid_positions.min(axis=0)
    max_coords = liquid_positions.max(axis=0)
    
    within_bounds = all(
        min_coords[i] <= position[i] <= max_coords[i]
        for i in range(3)
    )
    
    # 2. Build KDTree from existing atoms (exclude water for solute replacement)
    existing_tree = self._build_existing_atoms_tree(structure, exclude_water=True)
    
    # 3. Rotate template molecule
    center = self.template_positions.mean(axis=0)
    centered_template = self.template_positions - center
    rot_matrix = self._euler_to_rotation_matrix(*rotation)
    rotated = centered_template @ rot_matrix.T
    placed_mol = rotated + np.array(position)
    
    # 4. Check overlap (all-atom check)
    has_overlap = False
    min_distance = float('inf')
    if existing_tree is not None:
        for atom_pos in placed_mol:
            dist = existing_tree.query(atom_pos, k=1)[0]
            min_distance = min(min_distance, dist)
            if dist < self.config.min_separation:
                has_overlap = True
                break
    
    return ValidationResult(
        is_valid=within_bounds and not has_overlap,
        within_bounds=within_bounds,
        has_overlap=has_overlap,
        min_distance=min_distance,
        error_message=None if within_bounds and not has_overlap else
            "Position outside liquid region" if not within_bounds else
            f"Overlap detected (min distance: {min_distance:.3f} nm)"
    )
```

### Pattern 2: Preview Actor Overlay
**What:** Show proposed molecule position in viewer without committing to insertion
**When to use:** After validation, provide visual feedback
**Example:**
```python
# Source: VTK renderer pattern from existing codebase
def show_preview_actor(
    self,
    position: tuple[float, float, float],
    rotation: tuple[float, float, float],
):
    """Show preview molecule at proposed position with distinct styling."""
    # Remove existing preview actor
    if self._preview_actor is not None:
        self.renderer.RemoveActor(self._preview_actor)
    
    # Create molecule at preview position
    preview_positions = self._compute_placed_molecule(position, rotation)
    
    # Create actor with semi-transparent styling
    actor = create_custom_molecule_actor(
        preview_positions,
        self.template_atom_names,
        self.cell,
        moleculetype_name="PREVIEW"
    )
    
    # Make preview semi-transparent (opacity 0.6)
    actor.GetProperty().SetOpacity(0.6)
    
    # Add to renderer
    self.renderer.AddActor(actor)
    self._preview_actor = actor
    
    # Render
    self.render_window.Render()

def clear_preview_actor(self):
    """Remove preview actor from viewer."""
    if self._preview_actor is not None:
        self.renderer.RemoveActor(self._preview_actor)
        self._preview_actor = None
        self.render_window.Render()
```

### Anti-Patterns to Avoid
- **Full system preview:** Placing ALL molecules and rendering entire structure is O(N×M) and violates REQUIREMENTS.md constraint
- **Real-time preview on every change:** Re-validating on every slider drag is expensive; use on-demand (button click) instead
- **Duplicating overlap checking logic:** Reuse existing `_build_existing_atoms_tree()` and `_check_overlap()` from CustomMoleculeInserter

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Overlap detection | Custom distance loops | scipy.spatial.cKDTree | O(N log N) vs O(N²), PBC support |
| Rotation conversion | Manual matrix math | scipy.spatial.transform.Rotation.from_euler | Numerically stable, handles all conventions |
| Molecule rendering | Custom OpenGL | VTK vtkMoleculeMapper | CPK colors, bond detection, hardware acceleration |
| Bounds checking | Manual coordinate loops | numpy array operations | Vectorized, fast |

**Key insight:** CustomMoleculeInserter already has all the placement logic. Extract validation into a reusable method that doesn't modify state.

## Common Pitfalls

### Pitfall 1: Computational Expense Misunderstanding
**What goes wrong:** Implementing full preview that places all N molecules and renders entire system
**Why it happens:** Thinking "preview" means "show the final result"
**How to avoid:** Preview ONE molecule at a time. This is O(M) where M is existing atom count, not O(N×M)
**Warning signs:** Preview takes >1 second, UI freezes during preview

### Pitfall 2: Bounds vs Overlap Confusion
**What goes wrong:** Only checking overlap, not checking if position is in valid region (liquid phase)
**Why it happens:** Overlap checking is obvious, bounds checking is easy to forget
**How to avoid:** Validate BOTH:
  - Position is within liquid region bounds (ice_atom_count to ice_atom_count+water_atom_count)
  - No overlap with existing atoms (ice + guests, exclude water for replacement)
**Warning signs:** Molecule placed in ice region or outside simulation box

### Pitfall 3: State Mutation During Validation
**What goes wrong:** Validation method modifies MoleculetypeRegistry or other state
**Why it happens:** Copy-paste from actual insertion code
**How to avoid:** Validation should be **read-only** - compute placement but don't register with registry
**Warning signs:** Multiple previews create multiple moleculetype entries, registry grows unbounded

### Pitfall 4: Memory Leak with Preview Actors
**What goes wrong:** Preview actors accumulate in renderer, slowing rendering
**Why it happens:** Not removing old preview actor before creating new one
**How to avoid:** Always call `renderer.RemoveActor()` before adding new preview actor
**Warning signs:** Rendering slows down after multiple previews

### Pitfall 5: KDTree Rebuild on Every Check
**What goes wrong:** Rebuilding cKDTree from existing atoms for every validation call
**Why it happens:** Not caching the tree
**How to avoid:** Build tree once when source structure is loaded, cache in CustomMoleculePanel or CustomMoleculeInserter
**Warning signs:** Each validation takes 0.5-1.0 seconds even for simple checks

```python
# Source: Derived from SolutePanel preview pattern
def _on_validate_placement_clicked(self):
    """Validate current position/rotation settings."""
    if not self.gro_path or not self.itp_path:
        QMessageBox.warning(self, "Validation Error", 
            "Please upload GRO and ITP files first")
        return
    
    # Get current position/rotation
    try:
        position = (
            float(self.pos_x_edit.text()),
            float(self.pos_y_edit.text()),
            float(self.pos_z_edit.text())
        )
        rotation = (
            self.rot_alpha_spin.value(),
            self.rot_beta_spin.value(),
            self.rot_gamma_spin.value()
        )
    except ValueError as e:
        QMessageBox.warning(self, "Invalid Input", 
            f"Invalid position value: {e}")
        return
    
    # Create validation inserter (doesn't modify state)
    from quickice.structure_generation.custom_molecule_inserter import (
        CustomMoleculeInserter
    )
    
    config = CustomMoleculeConfig(
        placement_mode="custom",
        gro_path=self.gro_path,
        itp_path=self.itp_path,
        positions=[position],
        rotations=[rotation]
    )
    
    # Get current interface structure from MainWindow
    # (passed via signal or stored reference)
    structure = self._get_current_interface_structure()
    if structure is None:
        QMessageBox.warning(self, "No Structure", 
            "Please generate an interface structure first")
        return
    
    # Validate placement
    inserter = CustomMoleculeInserter(config)
    result = inserter.validate_single_placement(
        structure, position, rotation
    )
    
    # Show result
    if result.is_valid:
        self.validation_label.setText(
            f"✓ Placement valid\n"
            f"  Position: ({position[0]:.2f}, {position[1]:.2f}, {position[2]:.2f}) nm\n"
            f"  Min distance to nearest atom: {result.min_distance:.3f} nm"
        )
        self.validation_label.setStyleSheet("color: green;")
        self.log_message(f"Placement validated: min distance {result.min_distance:.3f} nm")
        
        # Offer to show preview
        reply = QMessageBox.question(
            self, "Show Preview",
            "Placement is valid. Show preview in 3D viewer?",
            QMessageBox.Yes | QMessageBox.No
        )
        if reply == QMessageBox.Yes:
            self._show_preview_requested.emit(position, rotation)
    else:
        self.validation_label.setText(f"✗ {result.error_message}")
        self.validation_label.setStyleSheet("color: red;")
        self.log_message(f"Validation failed: {result.error_message}")
```

## Open Questions

Things that couldn't be fully resolved:

1. **Should preview be automatic or on-demand?**
   - What we know: Real-time preview on every valueChanged would be expensive
   - What's unclear: User preference for automatic vs manual trigger
   - Recommendation: On-demand via "Validate & Preview" button. This matches scientific software patterns where expensive operations are explicit.

2. **Should KDTree be cached across multiple validations?**
   - What we know: Building tree is O(N log N), querying is O(log N)
   - What's unclear: Memory overhead vs speed tradeoff
   - Recommendation: Cache in CustomMoleculePanel when structure is loaded. Invalidate on structure change. This is acceptable because:
     - Structure changes are infrequent (once per tab workflow)
     - Multiple validations are expected (user tries different positions)
     - Memory overhead is modest (positions array already in memory)

3. **Should preview show molecule in context of entire system or isolated?**
   - What we know: Showing entire system (ice + water + guests + preview molecule) provides context
   - What's unclear: Rendering complexity for large systems
   - Recommendation: Show preview in context. CustomMoleculeViewerWidget already has the structure actor. Just add preview actor on top. This is the existing pattern and works well.

## Appendix: Scope Clarification

**REQUIREMENTS.md states:**
> "Placement preview before commit" is OUT OF SCOPE with reason "Computationally expensive for large systems"

**This refers to:**
- Previewing ALL N molecules placed simultaneously
- Full system simulation with all custom molecules
- Rendering O(N×M) atoms where N = number of custom molecules, M = existing atoms

**Phase 34.5 does NOT need this. Instead:**
- Preview ONE molecule at a time (for Custom mode)
- OR show representative placement (for Random mode, first molecule preview)
- Computational cost: O(M) per validation where M = existing atoms
- This is the same cost as the actual insertion validation already implemented

**Feasibility analysis:**
| Operation | Cost | Feasible? |
|-----------|------|-----------|
| Validate single molecule placement | O(M) | ✓ Yes (< 0.1 sec) |
| Render preview actor | O(K) where K = atoms in one molecule | ✓ Yes (instantaneous) |
| Validate all N molecules | O(N×M) | ✗ No (violates constraint) |
| Render all N molecules | O(N×K) | ✗ No (violates constraint) |

**Conclusion:** Single-molecule validation and preview is computationally feasible and does NOT violate the OUT OF SCOPE constraint.
