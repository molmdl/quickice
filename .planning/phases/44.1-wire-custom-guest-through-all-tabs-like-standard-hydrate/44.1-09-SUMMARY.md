## Decisions Made
- **hydrate_config=None default (backward compat):** Existing call sites that don't pass `hydrate_config` keep working unchanged. When the param is None AND the structure has built-in guests (ch4/thf), the helper detects the built-in guest_type from the structure and copies the bundled `{guest_type}_hydrate.itp` — byte-identical to the previous built-in path. Plan 44.1-10 will wire `self._current_hydrate_config` from MainWindow to make the GUI always pass the config explicitly (the forward-looking path).
- **Staging inside the try block (matches hydrate_export.py:147):** If `_stage_hydrate_guest_itps` raises (e.g. FileNotFoundError for a missing custom ITP), the existing broad `except Exception` handler catches it and shows a QMessageBox to the user. AGENTS.md permits broad `except Exception` in GUI code (only `quickice/cli/pipeline.py` forbids it). This also means the helper's FileNotFoundError for a missing custom ITP shows as "Export Error" (slightly different message from the old "Missing Guest ITP" warning, but functionally equivalent — the user sees a critical dialog either way).
- **Explicit `guest_atom_count`/`guest_nmolecules` via `getattr(interface_structure, ..., 0)` to the helper:** The helper's presence gate short-circuits no-guest exports without touching the filesystem. The `getattr` with default 0 is defensive (interface_structure always has these attrs after `assemble_slab`, but the default makes the call safe for any structure). The helper itself also falls back to structure attrs when the kwargs are None, but the explicit `getattr` matches the plan's spec verbatim and makes the gating self-documenting at the call site.
- **Removed the entire broken block (not partial):** The old `_detect_guest_type_from_structure + shutil.copy + QMessageBox.warning` block (export.py:958-972 in the pre-edit file) is gone. The helper handles both the custom (transform+write etoh.itp) AND built-in (shutil.copy ch4_hydrate.itp) paths. Keeping any part of the old block would be dead code (the helper already stages the built-in ITP, so the old `shutil.copy({guest_type}_hydrate.itp)` would either copy the same file twice or fail for custom).
- **Kept the tip4p-ice.itp copy (water ITP) separate:** The helper only stages GUEST ITPs (custom etoh.itp OR built-in {guest_type}_hydrate.itp); the water ITP is structural and always copied. Matches the plan's instruction.
- **Test exercises the EXPORTER METHOD (not the writers directly):** The plan explicitly requires `exporter.export_interface_gromacs(iface, hydrate_config=config)` under the `patch(...)` context managers, NOT calling `write_interface_gro_file`/`write_interface_top_file` directly. The whole point of the plan is to verify the exporter threads `hydrate_config` through to the writers. The existing 41-10 test calls the writers directly (wrong model for the exporter-method path) — the new test is the right model.
- **Real generation pipeline (not synthetic data) in the test:** The plan says "Generate a custom ethanol hydrate (etoh_e2e, ...) hydrate.to_candidate() -> assemble_slab(...) -> InterfaceStructure (no crash — plans 02/03/05 fixed this)". This exercises the actual config that MainWindow will pass (44.1-10). GenIce2 sI 1x1x1 + `assemble_slab` is fast (~0.04s) so no module-scoped fixture amortization is needed — the test generates inline. The resulting slab is substantial (ice_nmolecules=1656, water_nmolecules=2197, guest_nmolecules=288, guest_atom_count=2592 — ~18k atoms) but grompp handles it in ~1s.
- **QFileDialog called ONCE (not multiple times):** The plan's note "may be called multiple times ... use side_effect" was a defensive hedge. Reading the actual `export_interface_gromacs` method shows `QFileDialog.getSaveFileName` is called ONCE (for the .gro path); the .top path is derived from `path.with_name(path.stem + '.top')` (no second dialog). So a single `return_value` patch is sufficient — no `side_effect` needed.
- **@gmx_skipif on the WHOLE test (matches 41-10/41-11 convention):** The plan's wording "file-consistency always runs" is slightly ambiguous but the established repo pattern skips the whole test when gmx is absent. gmx IS on PATH for this repo so the test runs (file-consistency + grompp both execute).
