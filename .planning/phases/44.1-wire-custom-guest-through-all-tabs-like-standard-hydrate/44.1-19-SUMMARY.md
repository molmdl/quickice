## Decisions Made
- **Parametrized over 4 exporters (not one giant test):** The plan's primary spec named a single `test_custom_guest_gui_full_chain_grompp` test but explicitly allowed splitting into 4 parametrized cases if one test is too heavy. Parametrization gives per-exporter failure isolation with `-x` (one exporter failing doesn't mask the other 3), and the base function name still satisfies the must-have `contains` check. The module-scoped fixture amortizes the shared GenIce2 + inserter setup across the 4 cases (the plan's amortization requirement). 4 grompp calls (~3s total) is not too heavy.
- **Fresh interface per inserter (generate-once, derive-many):** The ion inserter mutates `iface.molecule_index` (`ion_inserter.py:259`) when the interface has no molecule_index (real GenIce2 data has an empty one). A single shared interface would cross-contaminate the solute exporter's `solute.interface_structure.molecule_index` reference (same object) after the ion inserter runs. The fixture generates the hydrate (GenIce2) ONCE, then derives a fresh `assemble_slab(candidate, config)` interface per inserter — deterministic with `seed=42` so all are identical, and each inserter gets its own unmutated copy. `assemble_slab` is cheap (no GenIce2 re-run), so 4 derivations are negligible.
- **Renamed custom-molecule copy in module-scoped temp dir:** The custom-molecule case reuses the `cmet.gro`/`cmet.itp` (moleculetype `MOL`) renamed-copy workaround from plan 44.1-13 to avoid the `etoh.itp` filename collision between the cage guest (staged as `MOL_H` via `etoh.itp`) and the custom molecule (copied with atomtypes commented). The renamed files live in `tmp_path_factory.mktemp("custom_molecule_data")` (module-scoped) since the `CustomMoleculeInserter` needs the files to exist at fixture-setup time (function-scoped `tmp_path` is not yet available inside a module fixture).
- **Added explicit `#include` assertion (beyond existing per-exporter tests):** The plan's must-have states "the `.top` references `MOL_H` (not `GUE`) + `#include` the transformed `etoh.itp` (not `guest.itp`)". The existing per-exporter tests check `MOL_H` in `[molecules]`/`.gro` residues + `GUE` absent but do NOT explicitly assert the `#include` directives. Added `parse_top_includes` check that `etoh.itp` is `#include`d and `guest.itp` is NOT — directly verifies the plan's key assertion.

## Deviations from Plan

None - plan executed exactly as written. The plan explicitly anticipated the parametrization split ("If running all 4 exporters in one test is too heavy, split into 4 parametrized cases") and the module-scoped fixture ("use module-scoped fixture to amortize GenIce2 + inserter calls"), so both design choices are within plan scope. The per-inserter fresh-interface isolation is a correctness safeguard (preventing the ion inserter's `iface.molecule_index` mutation from cross-contaminating the solute exporter's reference), not a plan deviation — it implements the plan's "each inserter returns a NEW structure (V-17 compliant)" intent at the fixture level.

## Issues Encountered
- **Ion inserter mutates the source interface's `molecule_index`:** Discovered via grep (`structure.molecule_index = mol_index` at `ion_inserter.py:259`) while designing the module-scoped fixture. This is a pre-existing V-17 partial-violation in the ion inserter (it mutates an input attribute rather than being fully pure), but it only triggers when the input has no `molecule_index` (the InterfaceStructure-from-GenIce2 case). Resolved by giving each inserter its own fresh interface (generate-once, derive-many) so no inserter's mutation affects another's input or a stored structure reference. Did NOT fix the ion inserter itself (out of scope for a test-only plan; the mutation is a known pre-existing condition, and the existing per-exporter 44.1-15 test independently calls the ion inserter on its own interface so it's not exposed there).
