# Phase 44.1 Plan 01: Pitfall 6 Engine Relaxation Summary

**Relaxed HydrateConfig.__post_init__ Pitfall 6 check to track residue_name→guest_type, allowing the same custom guest in multiple cages (aggregating like ch4) while still rejecting different guest_types sharing a residue name**

## Accomplishments
- Relaxed the Pitfall 6 over-restriction in `HydrateConfig.__post_init__` so the same custom guest (same `guest_type`) can be assigned to multiple cages — it aggregates into ONE `_H` moleculetype exactly like ch4-in-all-cages
- Retained rejection of the genuinely unsafe case: two DIFFERENT custom `guest_type`s sharing the same `guest_residue_name` still raises `ValueError`
- Added a new guard (Open Question Q4): the same `guest_type` across cages must use a SINGLE `guest_residue_name` (one moleculetype → one residue name in the _H path)
- Added 2 new tests; the existing `test_duplicate_custom_residue_name_rejected` (unsafe case) is unchanged and green

## Files Created/Modified

- `tests/test_hydrate_config_custom.py` - Added `test_same_custom_guest_in_two_cages_allowed` (safe case, no raise) and `test_same_guest_type_different_resname_rejected` (new guard raises with match="different")

## Decisions Made
- **Pitfall 6 semantics:** Track `residue_name → guest_type` instead of `residue_name → cage_key`. Reject only when a DIFFERENT `guest_type` claims an already-seen residue name (genuine collision). Same `guest_type` in multiple cages is allowed (aggregates into one moleculetype like ch4). This is the prerequisite for removing the 44-02 GUI auto-clear mitigation (plan 18).
- **New guard (Q4):** Same `guest_type` across cages must use the SAME `guest_residue_name`. One moleculetype (`mol_type`) maps to one residue name in the _H hydrate path; mismatched residue names for the same guest_type are rejected with a "different" message. Added after the collision check, before the dict assignment.
