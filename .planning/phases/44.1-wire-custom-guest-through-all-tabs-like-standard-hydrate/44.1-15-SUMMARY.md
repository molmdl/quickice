When the param is None AND the structure has built-in guests (ch4/thf), the helper detects the built-in guest_type from `ion_structure.molecule_index` and copies the bundled `{guest_type}_hydrate.itp` — byte-identical to the previous built-in path. Plan 44.1-16 will wire `self._current_hydrate_config` from MainWindow to make the GUI always pass the config explicitly.
- **Pass ion_structure ITSELF (not interface_structure) to the staging helper:** Anticipated by the plan's context note ('IonStructure has NO interface_structure field — pass the ion_structure itself'). IonStructure (types.py:826-876) is FLATTENED — it carries its own `guest_atom_count`/`guest_nmolecules` (populated by IonInserter.replace_water_with_ions from the source interface, ion_inserter.py:243/287/312/538) and its own `molecule_index` with "guest" entries (ion_inserter.py:150-153). This matches the CustomMoleculeStructure pattern (44.1-13 — also flattened) and diverges from the SoluteStructure pattern (44.1-11 — which passes interface_structure because SoluteStructure delegates to it). No Rule 3 deviation needed — the plan's snippet already specified the correct pattern.
- **Staging inside the try block (matches 44.1-09 + 44.1-11 + 44.1-13 + hydrate_export.py:147):** If `_stage_hydrate_guest_itps` raises (e.g. FileNotFoundError for a missing custom ITP), the existing broad `except Exception` handler catches it and shows a QMessageBox. AGENTS.md permits broad `except Exception` in GUI code.
- **Guest-write branch first-entry fallback (gro writer):** The IonInserter builds `molecule_index` with literal `mol_type == "guest"` (ion_inserter.py:150-153), NOT the custom guest's actual mol_type (like "etoh_e2e"). So `custom_by_moltype.get(mol.mol_type)` returns None (since "guest" is not in custom_by_moltype). The fallback `next(iter(custom_by_moltype.values()))` resolves the first/only custom_guest_info entry's residue name. Same pattern as the solute (44.1-11) and custom-molecule (44.1-13) writers. No `reorder_guest_atoms` in the custom branch (custom guest atoms are already in ITP canonical order).
- **write_ion_top_file [molecules] line reused (no separate custom block):** The existing `f.write(f"{guest_res_name:<17s}{guest_count}\n")` already uses `guest_res_name`. When `custom_active`, `guest_res_name` is set to `custom_guest_info['residue_name']` (MOL_H) in the detection block, so the `[molecules]` line writes `MOL_H` automatically. The `#include` section uses `if custom_active: ... elif guest_count > 0 and guest_type:` so the custom `#include` takes precedence.
- **Custom guest atomtype merge AFTER ch4/thf blocks (not elif):** `needs_ch4_atomtypes`/`needs_thf_atomtypes` are based on `guest_type` (None when custom_active). The custom guest atomtype merge is an ADDITIONAL block relying on `_written_atomtypes` dedup to prevent duplicates with the water/ion atomtypes. Verified by grompp rc==0.
- **Test exercises the EXPORTER METHOD (not the writers directly):** The plan explicitly requires `with patch(...) exporter.export_ion_gromacs(ion_structure, hydrate_config=config)` instead of calling write_ion_gro/top_file directly. This is the WHOLE POINT of the plan — verify the exporter threads hydrate_config through to the writers.
- **QFileDialog called ONCE (single return_value patch suffices):** Confirmed the plan 09/11/13 note. No `side_effect` needed.
