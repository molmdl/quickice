## Decisions Made
- **Pocket-only change (no consolidation):** Per the plan, the identical change pattern was applied to pocket.py only. piece.py has its own `_detect_guest_atoms` function and is fixed by separate plan 07. Consolidating the three helpers would be scope creep and risk breaking the other mode mid-wave.
- **guest_atom_count threaded as kwarg with default None:** Both `_detect_guest_atoms` and `_count_guest_molecules` keep their existing positional args first; `guest_atom_count: int | None = None` is appended so any existing callers (tests, other modes) keep working unchanged. The new value is only populated on the hydrate path where candidate.metadata carries `guest_atom_counts`.
- **Lookup by _guest_type (single source of truth):** `_guest_atom_count = guest_atom_counts.get(_guest_type) if _guest_type else None` reuses the same `_guest_type` already extracted from `guest_type_counts`. This guarantees the atom count matches the identified guest type (no key-mismatch risk) and stays None when there is no guest.
- **Built-in path stays byte-identical:** For ch4/thf, `count_guest_atoms` returns the explicit constant (CH4=5, THF=13) and ignores `guest_atom_count` when `guest_type in ("ch4", "thf", None)` (44.1-03 logic). The `guest_atoms_per_mol` heuristic also yields 5 for ch4 (first atom "C", second not "O") and 13 for thf (first atom "O", len//nmolecules). So threading `guest_atom_count=5` for ch4 changes nothing — verified by `test_assemble_pocket_builtin_ch4_regression`.
- **Pocket InterfaceConfig uses pocket_diameter + pocket_shape (not slab fields):** The plan's probe example used slab-only `ice_thickness`/`water_thickness` and omitted `pocket_diameter`; per the plan's own instruction to "check the InterfaceConfig fields used in existing pocket tests and adjust", the probe and tests use `mode="pocket", box_x=3.0, box_y=3.0, box_z=3.0, seed=42, pocket_diameter=2.0, pocket_shape="sphere"` (matching `test_pocket_invariants.py`).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed guest_atoms_per_mol heuristic that fragmented custom multi-atom molecules**

- **Found during:** Task 1 (verification probe)
- **Issue:** The plan asked to apply the "IDENTICAL change pattern as plan 05 (slab)" — threading `guest_atom_count` through `_detect_guest_atoms` + `_count_guest_molecules`. However, after applying only those 3 changes, the verification probe failed: `guest_atom_count=1778` equalled `guest_nmolecules=1778` instead of `9 * guest_nmolecules`. Root cause: pocket.py's tiling section has a `guest_atoms_per_mol` heuristic (lines 372-388) that defaults to `1` for custom guests whose first atom is not Me/C/O (ethanol's labels start with "H"). This `guest_atoms_per_mol` is used in TWO places the slab fix didn't touch: (a) `tile_structure(..., atoms_per_molecule=guest_atoms_per_mol)` returns `tiled_guest_nmolecules = atom_count` (not molecule count); (b) the cavity-distance filtering uses `guest_o_idx = np.arange(0, len(...), guest_atoms_per_mol)` and `for mol_idx in keep_mols: start = mol_idx * guest_atoms_per_mol` to keep whole molecules — with stride 1 this keeps individual atoms, fragmenting 9-atom ethanol molecules. Slab avoided this because slab does no cavity filtering and recomputes `guest_nmolecules` from position length at the end; pocket does neither.
- **Fix:** Extended the `guest_atoms_per_mol` detection to prefer the threaded `_guest_atom_count` (already in scope in `assemble_pocket`) when it is not None, falling back to the existing first-atom heuristic for built-in guests. This fixes both the molecule count (`tiled_guest_nmolecules` now = atom_count / 9) and the cavity-filtering molecule boundaries (arange stride = 9, molecule slices of 9 atoms).
- **Files modified:** `quickice/structure_generation/modes/pocket.py` (guest_atoms_per_mol block, lines 371-388)
- **Commit:** `bcf3768` (included in Task 1 commit)
- **Byte-identical for built-ins:** For ch4 the heuristic yields 5 (first atom "C", second not "O") and `_guest_atom_count` is also 5; for thf the heuristic yields 13 (first atom "O", len//nmolecules) and `_guest_atom_count` is also 13. So the built-in path is unchanged — verified by `test_assemble_pocket_builtin_ch4_regression`.
