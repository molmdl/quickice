# Phase 44.1: Wire Custom Guest Through All Tabs Like Standard Hydrate - Research

**Researched:** 2026-07-08
**Domain:** Hydrate custom-guest metadata propagation through the Interface → Custom → Solute → Ion → Export tab chain (GUI + CLI)
**Confidence:** HIGH (claims verified by reading source + running live probes against the `quickice` conda env)

## Summary

A custom-guest hydrate (e.g. ethanol, 9 atoms, residue `MOL`) **cannot flow past the Hydrate tab today** — it crashes the moment it enters any Interface mode (slab/pocket/piece). This is a more severe bug than the ROADMAP describes: the ROADMAP frames 44.1 as "Interface tab **export** broken," but live probing shows interface **generation** itself crashes (`IndexError: index 256 is out of bounds` in `slab.py:184` / `pocket.py` / `piece.py`). Root cause: `HydrateStructure.to_candidate()` (types.py:1168) drops `guest_descriptors` (which carry each guest's `atom_count`), so the interface modes fall back to `count_guest_atoms` heuristic in `quickice/utils/molecule_utils.py:21`, which only knows `ch4` (5) and `thf` (13). For a 9-atom ethanol guest the heuristic miscounts (returns 1/2/13 randomly), producing out-of-bounds indices. **Three independent defect clusters** must be fixed for end-to-end flow: (1) the upstream generation crash (to_candidate + count_guest_atoms + 3 interface modes), (2) the four GUI exporters + four CLI export branches that use `_detect_guest_type_from_structure` + `shutil.copy({guest_type}_hydrate.itp)` (returns None / wrong built-in for custom → broken `.top`), and (3) the Pitfall 6 engine over-restriction (`types.py:711-720`) that blocks the SAFE case of the same custom guest in two cages.

**Primary recommendation:** Fix in four vertical layers — (a) Pitfall 6 engine relaxation (~3 lines), (b) `to_candidate()` carries `guest_descriptors` in metadata + `count_guest_atoms` accepts an explicit `guest_atom_count` + the 3 interface modes thread it (unblocks generation), (c) all 4 GUI exporters + the CLI export branches receive the hydrate config and build `custom_guest_info` via the existing `_build_custom_guest_info(config)` (pipeline.py:32) — **no new fields on InterfaceStructure/SoluteStructure/IonStructure are strictly required** because the hydrate config is the source of ITP paths and persists on `MainWindow._current_hydrate_config` (main_window.py:749, never cleared) and `CLIPipeline._hydrate_result.config` (pipeline.py:383), and (d) remove the 44-02 `_enforce_single_custom_cage` auto-clear mitigation (hydrate_panel.py:636) now that the engine allows the same custom guest in 2 cages. Mirror the 42-08 structure-driven ITP staging pattern (`hydrate_export.py:253-316`) in a shared helper so all 4 GUI exporters stage custom-guest ITPs identically. The user wants many small plans — a 23-plan decomposition across 4 dependency waves is recommended below.

### Pattern 1: Config-Driven custom_guest_info (recommended primary)
**What:** Build `custom_guest_info` from the hydrate config's `cage_guest_assignments` via the existing `_build_custom_guest_info(config)`, and pass it to the writers + ITP staging. The structure's `guest_atom_count`/`guest_nmolecules` only gates whether guests are present.
**When to use:** All 4 GUI exporters + all CLI export branches. The hydrate config is available in both paths (GUI: `MainWindow._current_hydrate_config` set at main_window.py:749, never cleared; CLI: `CLIPipeline._hydrate_result.config` at pipeline.py:383).
**Why safe here (vs 42-08 desync):** The interface/custom/solute/ion structures are built FROM the hydrate. You cannot change the interface without regenerating from the hydrate, so config and structure are inherently in sync. The 42-08 desync (change lattice without regen) only applies to the hydrate-export path, which already uses structure-driven staging.

```python
# Source: mirrors pipeline.py:32-70 (_build_custom_guest_info) + hydrate_export.py:253-316
from quickice.output.gromacs_writer import write_interface_gro_file, write_interface_top_file, transform_guest_itp
# _build_custom_guest_info should be MOVED to a shared module and imported by both GUI + CLI
custom_guest_info = _build_custom_guest_info(hydrate_config)  # None for all-built-in; list[dict] for custom
write_interface_gro_file(iface, str(path), custom_guest_info=custom_guest_info)
write_interface_top_file(iface, str(top_path), custom_guest_info=custom_guest_info)
# Stage custom ITPs (transform with _H suffix) — mirror hydrate_export.py:353-359
for ci in (custom_guest_info or []):
    content = Path(ci["itp_path"]).read_text()
    transformed = transform_guest_itp(content, ci["residue_name"].removesuffix("_H"), suffix="_H")
    path.with_name(Path(ci["itp_path"]).name).write_text(transformed)
```

### Pattern 2: Structure-Driven ITP Staging (42-08 — reference, optional hardening)
**What:** Iterate `structure.molecule_index` for unique guest mol_types; use `structure.guest_descriptors` for residue_name + `config.cage_guest_assignments` for ITP path. This is the hydrate-export pattern at `hydrate_export.py:253-316`.
**When to use:** Only if the planner chooses the structure-driven hardening (adds `guest_descriptors` to InterfaceStructure + propagates through inserters). NOT required for the minimal fix.
**Caveat:** InterfaceStructure built by slab.py has `molecule_index=[]` (slab.py:620-632 does NOT pass molecule_index). So for the interface path, the "iterate structure.molecule_index" approach does NOT work — you'd iterate `guest_descriptors` instead. This is a meaningful difference from the hydrate path.

### Anti-Patterns to Avoid
- **Extending `_detect_guest_type_from_structure` to return custom info:** It's a heuristic for built-ins only (ch4/thf/co2/h2). Custom guests have no heuristic signature. Branch on "does the hydrate config have a custom assignment?" instead. (export.py:26-65)
- **Adding `guest_itp_path` to `GuestDescriptor`:** GuestDescriptor is the WHAT (residue_name, atom_count, is_custom). The ITP path is the WHERE and lives on `CageGuestAssignment`/`HydrateConfig`. Mixing them breaks the 42-08 separation. Keep ITP paths in config.
- **Hardcoding `0.0299` or `4`:** AGENTS.md constraints — use `WATER_VOLUME_NM3` / `WATER_ATOMS_PER_MOLECULE` from types.py.
- **Bare `except Exception` in `quickice/cli/pipeline.py`:** AGENTS.md — use `except (ValueError, OSError)`. GUI code (export.py) may use broad catches.
- **Mutating input structures in inserters:** AGENTS.md V-17 — all inserters return NEW objects. When propagating `guest_descriptors` (if chosen), copy the list.

## Pitfall 6 Fix

### Exact change (types.py:711-720)

Current (rejects ANY duplicate `guest_residue_name` across custom assignments — blocks the safe same-guest-in-2-cages case):

```python

seen_residue_names: dict[str, str] = {}  # residue_name -> first cage_key
...
# Pitfall 6: distinct guest_residue_name across custom assignments.
if assignment.guest_residue_name in seen_residue_names:
    raise ValueError(
        f"Duplicate guest_residue_name '{assignment.guest_residue_name}' "
        f"across custom cage assignments: cage '{seen_residue_names[assignment.guest_residue_name]}' "
        f"and cage '{cage_key}' both use this residue name. Custom hydrate "
        f"guests must have distinct guest_residue_name (the _H hydrate path "
        f"does not disambiguate)."
    )
seen_residue_names[assignment.guest_residue_name] = cage_key
```

Fixed (~3-line semantic change: track `guest_residue_name → guest_type`, only raise when a DIFFERENT `guest_type` claims an already-seen residue name):

```python

_seen_residue_name_to_guest_type: dict[str, str] = {}
...
# Pitfall 6: distinct guest_residue_name across DIFFERENT custom guest_types.
# Same custom guest_type in multiple cages is SAFE (aggregates to one
# {residue_name}_H moleculetype, exactly like ch4 in small+large); different
# guest_types sharing a residue name is UNSAFE (two moleculetypes would both

    raise ValueError(
        f"Duplicate guest_residue_name '{assignment.guest_residue_name}' "
        f"across DIFFERENT custom guest_types: '{existing_gt}' and "
        f"'{assignment.guest_type}' both use this residue name. Custom hydrate "
        f"guests of DIFFERENT types must have distinct guest_residue_name "
        f"(the _H hydrate path does not disambiguate)."
    )
_seen_residue_name_to_guest_type[assignment.guest_residue_name] = assignment.guest_type
```

## to_candidate() + InterfaceStructure Propagation

### The upstream crash (verified by live probe — HIGH confidence)
Running a custom ethanol hydrate through `to_candidate()` → `assemble_slab` raises:
```
IndexError: index 256 is out of bounds for axis 0 with size 256  (slab.py:184)
```
Same crash in `assemble_pocket` and `assemble_piece`. Root cause chain:
1. `to_candidate()` (types.py:1217-1231) builds `metadata={"guest_type_counts": {"etoh_e2e": 8}, ...}` but **drops `guest_descriptors`** (which carry `atom_count=9`).
2. `slab.py:175` `_guest_type = next(iter(guest_type_counts), None)` → `"etoh_e2e"` (custom slug).
3. `slab.py:178` `_detect_guest_atoms(atom_names, atoms_per_mol=4, guest_type="etoh_e2e")`.
4. `count_guest_atoms(atom_names, i, guest_type="etoh_e2e")` (`molecule_utils.py:53-58`) — custom slug is not `ch4`/`thf`, so it **falls through to the heuristic** which returns 1/2/13 randomly for ethanol atoms (H,C,H,H,C,H,H,O,H). Confirmed by probe: `count_guest_atoms(['H','C','H','H','C','H','H','O','H'], 2, guest_type='etoh_e2e')` returns `13` (thinks the O is THF). The accumulated indices overshoot the array → IndexError.

**Built-in ch4 (5 atoms, C+4H) does NOT crash** — the heuristic correctly returns 5. Verified: `ch4` slab generation succeeds with `guest_nmolecules=288, guest_atom_count=1440`. This is why the bug was missed: all existing tests use ch4/thf or synthetic InterfaceStructures.

### Recommended fix (to_candidate carries descriptors + count_guest_atoms accepts atom_count)

**Step 1 — `to_candidate()` carries `guest_descriptors` (types.py:1217-1231):**
Add to the returned `Candidate.metadata`:
```python
metadata={
    "lattice_type": lattice_id,
    "lattice_description": self.lattice_info.description,
    "water_count": self.water_count,
    "guest_count": self.guest_count,
    "guest_type_counts": guest_type_counts,
    "original_hydrate": True,
    # NEW (44.1): carry guest descriptors so interface modes can identify
    # custom guest atoms precisely (atom_count) instead of the ch4/thf heuristic.
    "guest_descriptors": list(self.guest_descriptors),
    # NEW (44.1): fast lookup mol_type -> atom_count for count_guest_atoms.
    "guest_atom_counts": {gd.mol_type: gd.atom_count for gd in self.guest_descriptors},
},
```
`GuestDescriptor` is a dataclass (types.py:469) and `Candidate.metadata` is `dict[str, Any]` (types.py:251), so carrying the list directly is safe.

**Step 2 — `count_guest_atoms` accepts explicit `guest_atom_count` (molecule_utils.py:21):**
```python
def count_guest_atoms(atom_names, start, guest_type=None, guest_atom_count=None):
    if start >= len(atom_names):
        return 0
    # Explicit atom count (custom guests) takes priority over heuristic.
    if guest_atom_count is not None and guest_type not in ("ch4", "thf", None):
        return guest_atom_count
    # existing ch4/thf explicit + heuristic fallback unchanged...
```
Verified by probe: `patched_count(custom_atoms, i, guest_type='etoh_e2e', guest_atom_count=9)` correctly yields 8 molecules × 9 atoms = 72, no out-of-bounds.

**Step 3 — 3 interface modes thread `guest_atom_count` (slab.py:170-192, pocket.py:154-167, piece.py:200-220):**
Each mode currently does:
```python
guest_type_counts = candidate.metadata.get("guest_type_counts", {})
_guest_type = next(iter(guest_type_counts), None) if guest_type_counts else None
water_indices, guest_indices = _detect_guest_atoms(candidate.atom_names, atoms_per_mol, guest_type=_guest_type)
```
Change to also look up the atom count:
```python
guest_atom_counts = candidate.metadata.get("guest_atom_counts", {})  # NEW
_guest_atom_count = guest_atom_counts.get(_guest_type) if _guest_type else None  # NEW
water_indices, guest_indices = _detect_guest_atoms(
    candidate.atom_names, atoms_per_mol,
    guest_type=_guest_type, guest_atom_count=_guest_atom_count,  # NEW
)
```
And `_detect_guest_atoms` (slab.py:24, pocket.py:24, piece.py:31) + `_count_guest_molecules` (slab.py:92, pocket.py:84, piece.py) each gain a `guest_atom_count=None` param threaded to `count_guest_atoms`. **These `_detect_guest_atoms` functions are duplicated 3×** (tech debt) — keep them separate for this phase (consolidation is scope creep) but apply the identical ~3-line change to each.

**Note:** `_detect_guest_atoms` and `_count_guest_molecules` are duplicated in slab.py:24/92, pocket.py:24/84, piece.py:31/102. Each pair needs the `guest_atom_count` param. This is 3 small parallel plans (or 1 plan touching all 3).

### Should InterfaceStructure gain `guest_descriptors`? (Open question — planner decides)

**Option A (recommended — minimal):** Do NOT add `guest_descriptors` to `InterfaceStructure`/`SoluteStructure`/`IonStructure`/`CustomMoleculeStructure`. Exporters build `custom_guest_info` purely from the hydrate config (`_build_custom_guest_info(config)`). The structure's `guest_atom_count`/`guest_nmolecules` (now correct after the generation fix) only gate presence. **Zero inserter changes needed.** This works because the hydrate config persists and is in sync with the interface (you can't change the interface without regenerating from the hydrate).

**Option B (optional hardening — more robust, more plans):** Add `guest_descriptors: list[GuestDescriptor] = field(default_factory=list)` to `InterfaceStructure` (types.py:426), populate it from `candidate.metadata["guest_descriptors"]` in the 3 interface modes, and propagate it through `SoluteStructure`/`IonStructure`/`CustomMoleculeStructure` (each inserter must copy it forward — `solute_inserter.py:784/831/961`, `ion_inserter.py:235/279/331/591`, `custom_molecule_inserter.py:774/948`). Then exporters mirror 42-08 exactly: structure gives residue_name, config gives ITP path. **~6 extra plans** for inserter propagation. Only worth it if the planner wants max robustness against future desync.

**Recommendation:** Option A for this phase (meets the goal: custom guest flows through all tabs). Option B can be a follow-up. The ROADMAP explicitly allows either ("carries custom guest descriptors **or** the exporter reconstructs `custom_guest_info` from ... the original `HydrateConfig`").

## GUI Exporter Unification

### The shared broken pattern (4 exporters, verified)
All 4 GUI exporters in `export.py` use the identical broken pattern for hydrate-guest ITP staging:
```python
guest_type = _detect_guest_type_from_structure(structure)   # returns None for custom (or wrong built-in)
if guest_type:
    guest_itp_source = _get_hydrate_guest_itp_path(guest_type)  # FileNotFoundError for custom (no {custom}_hydrate.itp)
    shutil.copy(guest_itp_source, path.with_name(f"{guest_type}_hydrate.itp"))
```
Locations: `InterfaceGROMACSExporter` (export.py:959-965), `SoluteGROMACSExporter` (export.py:181-187), `CustomMoleculeGROMACSExporter` (export.py:310-316), `IonGROMACSExporter` (export.py:410-416). Plus the writers are called with NO `custom_guest_info` (export.py:942-947 for interface; the solute/custom/ion writers don't take it but the ITP staging is the bug).

**`_detect_guest_type_from_structure` returns None for custom guests** because `detect_guest_type_from_atoms` (gromacs_writer.py:1433) only matches ch4/thf/co2/h2 atom signatures. **For a custom guest with ~5 atoms it may wrongly return "ch4"** (heuristic `atoms_per_guest = guest_atom_count // guest_nmolecules; return "thf" if >=10 else "ch4"` at export.py:62-63), causing the exporter to copy `ch4_hydrate.itp` — wrong ITP, broken `.top`.

### Recommended shared helper (DRY — mirror hydrate_export.py:147-359)
Add a module-level helper in `quickice/gui/export.py` (or a new `quickice/output/guest_info.py` shared with CLI) that all 4 exporters call:

```python

    Args:

    custom_guest_info = _build_custom_guest_info(hydrate_config)  # None or list[dict]

    if not (getattr(structure, 'guest_atom_count', 0) > 0 and getattr(structure, 'guest_nmolecules', 0) > 0):
```

## Common Pitfalls

### Pitfall 1: Interface GENERATION crashes (not just export) for custom guests
**What goes wrong:** `assemble_slab`/`assemble_pocket`/`assemble_piece` raise `IndexError` for any custom guest whose atom names aren't ch4/thf.
**Why:** `to_candidate()` drops `guest_descriptors.atom_count`; `count_guest_atoms` heuristic miscounts custom atoms.
**How to avoid:** Fix `to_candidate()` + `count_guest_atoms` (§5) BEFORE touching exporters. The exporters can't be tested until generation works.
**Warning signs:** Any test that calls `generate_interface(custom_hydrate.to_candidate(), ...)` crashes. No existing test exercises this (test_e2e_custom_guest_cli_grompp.py uses a SYNTHETIC InterfaceStructure — bypasses generation).

### Pitfall 2: `_detect_guest_type_from_structure` returns wrong built-in for custom
**What goes wrong:** A custom guest with ~5 atoms → heuristic returns "ch4" (export.py:62-63) → exporter copies `ch4_hydrate.itp` → `.top` references `CH4_H` but `.gro` has `MOL_H` → grompp fatal "residue MOL_H not found".
**Why:** The atom-count heuristic (`atoms_per_guest = guest_atom_count // guest_nmolecules`) only distinguishes "thf" (>=10) from "ch4" (<10).
**How to avoid:** Branch on the hydrate config FIRST (does it have a custom assignment?). Only call `_detect_guest_type_from_structure` for the all-built-in path. Do NOT extend the heuristic.

### Pitfall 3: GUI→CLI import or CLI→Qt import violations
**What goes wrong:** Moving `_build_custom_guest_info` to `quickice/gui/` would force the CLI to import a GUI module (PySide6) → breaks headless CLI. Or the GUI importing from `quickice/cli/pipeline.py` pulls CLI argparse into GUI.
**Why:** AGENTS.md: "No Qt/PySide6/VTK imports in `quickice/cli/`" and the dual-path separation.
**How to avoid:** Move `_build_custom_guest_info` to a NEUTRAL module (`quickice/output/guest_info.py` — no Qt, no argparse). Both GUI and CLI import from there.

### Pitfall 4: Forgetting `comb-rule=2` in any new/edited ITP handling
**What goes wrong:** Custom guest ITP transform must preserve comb-rule=2 (AMBER/GAFF2).
**Why:** AGENTS.md hard constraint.
**How to avoid:** `transform_guest_itp` already preserves it; do NOT write custom ITP copy logic — reuse `transform_guest_itp` + `copy_custom_guest_itp` (itp_helpers.py:201).

### Pitfall 5: Regression of built-in ch4/thf through all tabs
**What goes wrong:** Built-in ch4/thf hydrate → interface → export stops working after the refactor.
**Why:** The new config-driven branch must fall back to the built-in path when `hydrate_config is None` or has no custom assignment.
**How to avoid:** Every exporter change: `if hydrate_config and any(a.is_custom_guest for a in hydrate_config.cage_guest_assignments.values()): <custom path> else: <existing built-in path>`. Default `hydrate_config=None` → built-in. Add a regression e2e test (ch4 through all tabs) as a guard.
**Warning signs:** `test_ion_hydrate_fix.py`, `test_ion_path1_guest_attrs.py`, `test_e2e_*` ch4/thf chains fail.

### Pitfall 6: Removing auto-clear before engine fix lands
**What goes wrong:** Remove `_enforce_single_custom_cage` before the types.py Pitfall 6 fix → selecting custom in 2 cages raises `ValueError` from `__post_init__`, uncaught at main_window.py:742/748 (no try/except).
**Why:** The auto-clear was the 44-02 mitigation BECAUSE the engine rejected it.
**How to avoid:** Land the engine fix (§4) FIRST, then remove the auto-clear. Order matters.

## Risks & Regressions

1. **Pitfall 6 relaxation may allow an unsafe case the engine doesn't catch.** The new check only blocks DIFFERENT guest_types with the same residue_name. Edge case: two custom guests with DIFFERENT residue names but the SAME mol_type slug? That's impossible — `guest_type` IS the slug; same slug + different residue_name would mean the generator registers one moleculetype with two residue names. **Verify** the generator's ExitStack dedup (`hydrate_generator.py`) collapses same mol_type to one moleculetype regardless of residue_name. If two cages have the same custom guest_type but the user somehow set different residue_names, the engine should ALSO reject that. **Recommend adding a guard:** same `guest_type` across cages must have the SAME `guest_residue_name` (else raise). Add this to the Pitfall 6 fix plan.

2. **(CONFIRMED, not a risk — a scope fact):** `write_solute_top_file` / `write_custom_molecule_top_file` / `write_ion_top_file` do NOT accept `custom_guest_info` (verified gromacs_writer.py:2404/2865/3360). They emit `guest_res_name="GUE"` + `#include "guest.itp"` for custom guests → broken `.top`. Plans 12/14/16 MUST extend each dedicated top writer (mirror `write_interface_top_file:1536-1603`). Backward-compat: `custom_guest_info=None` default keeps the built-in ch4/thf path identical (the existing `test_e2e_*` ch4/thf chains stay green). See Open Question 1 for the extension pattern.

3. **Built-in ch4/thf through all tabs regressing.** Every exporter change MUST keep the `hydrate_config is None` (or no custom assignment) path identical to today. The regression guard (plan 23) is mandatory.

4. **`_current_hydrate_config` stale after the user clears/regenerates the hydrate.** `MainWindow._current_hydrate_config` is set at main_window.py:749 and never cleared. If the user generates hydrate A (custom), then hydrate B (ch4), `_current_hydrate_config` becomes B. The OLD interface result (from A) would then export with B's config → wrong. **This is the same class as the 42-08 desync.** Mitigation: Option B (structure-driven guest_descriptors) is the robust fix. For Option A, document the limitation: regenerate the interface after changing the hydrate. Low risk for 44.1 (the user flow is generate-once-export).

5. **CLI: `self._hydrate_result` is None when the user starts from an ice candidate (no hydrate).** Then `getattr(self._hydrate_result, "config", None)` → None → built-in path. Safe. But if the user did `--interface` without `--hydrate`, `_ice_candidate` is the ice candidate (not from a hydrate) and `_hydrate_result` is None — the interface export correctly uses the built-in path. Verified safe.

6. **`count_guest_atoms` change could affect built-in ch4/thf.** The new `guest_atom_count` param only fires when `guest_type not in ("ch4","thf",None)`. Built-in guests pass `guest_type="ch4"/"thf"` → existing explicit path → unchanged. The heuristic fallback (guest_type=None) is unchanged. Safe.

7. **Removing `_enforce_single_custom_cage` before the engine fix** → uncaught ValueError. Plan 19 MUST be ordered after plan 1. Enforce in the plan dependencies.
