**1. [Rule 3 - Blocking] Pass `custom_structure` (not `interface_structure`) to `_stage_hydrate_guest_itps`**

- **Found during:** Task 2's verification (`pytest tests/test_output/test_gromacs_export_custom.py` after wiring the exporter following the plan's snippet)
- **Issue:** The plan's Task 2 snippet said to reach guest presence via the interface: `_iface = getattr(custom_structure, 'interface_structure', custom_structure)` + `guest_atom_count=getattr(_iface, 'guest_atom_count', 0)`. But `CustomMoleculeStructure` (unlike `SoluteStructure`) carries its OWN `guest_atom_count`/`guest_nmolecules` + `molecule_index` (propagated from the interface during insertion, types.py:1044-1045, custom_molecule_inserter.py:782-783/730/744). The `interface_structure` is the SOURCE interface — for real GenIce2 data it has an EMPTY `molecule_index` and (in synthetic test data) `guest_atom_count=0`. Passing `interface_structure` short-circuits the helper's presence gate (`guest_atom_count=0` → returns `(None, {})` → no guest ITP staged) and the built-in detection finds no "guest" entries in the interface's `molecule_index`. The old code used `_detect_guest_type_from_structure(custom_structure)` which read `custom_structure.molecule_index` directly (NOT interface_structure).
- **Fix:** Pass `custom_structure` as the `structure` parameter to `_stage_hydrate_guest_itps` and use `custom_structure.guest_atom_count`/`guest_nmolecules` as the explicit kwargs (not `interface_structure`'s). The helper's presence gate + regression fix (guest_nmolecules recovery from molecule_index) + built-in detection all read `custom_structure`'s attrs/molecule_index, which carry the guest entries. This mirrors the old `_detect_guest_type_from_structure(custom_structure)` behavior.

## Issues Encountered
- **Known limitation surfaced (not fixed — out of scope):** When the cage guest and the custom molecule use the SAME ITP filename (e.g. both `etoh.itp`), the exporter's custom-molecule ITP copy (atomtypes commented, original moleculetype) OVERWRITES the staged cage-guest ITP (moleculetype `MOL_H`), and the `.top` has a DUPLICATE `#include "etoh.itp"`. This is a pre-existing collision latent in the custom-molecule export path that only manifests with custom cage guests (a Phase 44.1 scenario — before 44.1, cage guests were only built-in ch4/thf with different ITP filenames). The e2e test avoids it by using a renamed copy (`cmet.itp`). A future plan could: (a) deduplicate `#include` directives in the writer (skip the custom-molecule `#include` if the same filename is already `#include`d by the cage-guest branch), and (b) order the staging so the cage-guest ITP (`MOL_H`) is staged AFTER the custom-molecule ITP copy (or use a different filename for one of them). Out of scope for 44.1-13 (which is about the writers + exporter wiring, not ITP collision handling).
