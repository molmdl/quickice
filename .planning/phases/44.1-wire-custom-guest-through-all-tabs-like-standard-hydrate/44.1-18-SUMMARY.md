## Decisions Made
- **Removed the method entirely (not left as dead code):** Dead mitigation methods are UX-limiting landmines — a future caller could re-wire the call and silently re-introduce the auto-clear. The plan's verification (`grep -rn _enforce_single_custom_cage quickice/ tests/` returns nothing) enforces zero textual residue, so the method name was also removed from the updated explanatory comments/docstrings (referred to as "the 44-02 auto-clear mitigation" instead of naming the removed method).
- **Kept _on_cage_guest_changed (the handler itself):** It still routes per-cage combo changes to refresh the ff label + lattice info display + emit configuration_changed. Only the auto-clear logic inside it was removed. The lambda at the per-cage combo connection site still calls it.
- **Updated comments/docstrings without naming the removed method:** To satisfy the grep-returns-nothing verification, the updated explanatory text refers to "the 44-02 auto-clear mitigation" and "the mitigation was removed in 44.1-18" without the literal `_enforce_single_custom_cage` identifier. The rationale (engine relaxation in 44.1-01, residue_name->guest_type check only rejects genuine collisions) is preserved for future readers.
- **Renamed the test (not just rewrote assertions):** The old name `test_pitfall6_auto_clears_second_cage` documents behavior that no longer exists ("auto_clears"). The new name `test_same_custom_in_two_cages_gui_kept` documents the new behavior ("kept"). Kept the test class + fixtures unchanged per the plan.
