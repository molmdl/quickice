```markdown
## Custom Guest ITP Requirements (for Hydrate Cage Guests)

When uploading a custom guest molecule for hydrate cage placement (GUI-only for v4.7), the `.gro` and `.itp` files must meet specific requirements enforced by QuickIce's validation pipeline:

### 1. comb-rule = 2 (Mandatory)

The ITP `[ defaults ]` section must specify `comb-rule = 2` (Lorentz-Berthelot combining rules, AMBER/GAFF2 convention). 

- **Rejected:** If `[ defaults ]` contains `comb-rule = 1`, the upload fails with: "comb-rule must be 2 (Lorentz-Berthelot), got comb-rule=1"
- **Accepted:** If `[ defaults ]` is absent, the upload succeeds (the main `.top` file supplies `comb-rule = 2`)

This is enforced by `parse_itp_defaults_comb_rule()` in `itp_parser.py`.

### 2. Residue Base Name ≤ 3 Characters

The GRO file format uses fixed-width columns (6-10) for residue names, allowing a maximum of 5 characters. QuickIce appends an `_H` suffix to hydrate guest residue names (see rule 3 below), so the **base name must be ≤ 3 characters** to keep the total ≤ 5 chars.

- **Accepted:** `MOL` (3 chars) → `MOL_H` (5 chars) ✓
- **Rejected:** `ETHAN` (5 chars) → `ETHAN_H` (7 chars) ✗ — error: "residue name exceeds 3 chars"

This is enforced by `validate_gro_residue_name()` in `types.py`.

### 3. `_H` Suffix Convention

Hydrate cage guests use the `_H` suffix; liquid solutes use the `_L` suffix. The `transform_guest_itp()` function (in `cli/itp_helpers.py`) rewrites the `[ atoms ]` section residue name from `{base}` to `{base}_H` during export. The `MoleculetypeRegistry` (in `moleculetype_registry.py`) produces the registered moleculetype name with the suffix.

Example: A custom guest with residue name `MOL` becomes `MOL_H` in the exported `.gro` and `.top` files.

### 4. `[ atomtypes ]` Commented Out + Merged

The `[ atomtypes ]` section in the uploaded `.itp` is **commented out** during export (via `transform_guest_itp`), and the atom type definitions are **merged into the main `.top`** file's `[ atomtypes ]` section via `_merge_custom_atomtypes()`. This prevents duplicate atomtype conflicts between the custom guest and the bundled force field parameters.

### 5. Distinct from Tab-3 Liquid Custom Molecule

The hydrate custom guest (Tab 1) is distinct from the liquid custom molecule (Tab 3):

| Property | Hydrate Custom Guest (Tab 1) | Liquid Custom Molecule (Tab 3) |
|----------|-------------------------------|--------------------------------|
| Suffix | `_H` | None |
| Residue name limit | ≤ 3 chars (base) | ≤ 5 chars (no suffix) |
| comb-rule requirement | comb-rule = 2 mandatory | Inherited from main `.top` |
| Placement | Hydrate cages (GenIce2 bridge) | Liquid region (random/custom positions) |
| CLI flag | None (GUI-only for v4.7) | `--custom-gro` + `--custom-itp` (requires `--interface`) |

> **Note:** Custom guest in hydrate is a GUI-only feature for v4.7. The CLI supports built-in CH₄/THF guests only (via `--cage-guest`).
```
