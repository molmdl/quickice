### GRO/ITP Residue Name Convention
**What:** GRO files often use generic residue names ("MOL", "UNK") while ITP files define specific moleculetype names ("etoh", "methanol").

**When it occurs:**
- Sobtop-generated ITP files (like etoh.itp)
- Common in computational chemistry workflows
- GRO residue name is 5-character field in columns 6-10

**Example from etoh files:**
```
# etoh.gro (line 3)
    1MOL      H    1   0.208   0.048  -0.000
    ^^^^
    residue name = "MOL" (generic)

# etoh.itp (line 13)
[ moleculetype ]
; name          nrexcl
etoh       3
^^^^
moleculetype name = "etoh" (specific)
```

**Validation logic** (molecule_validator.py lines 148-160):
```python
if gro_residue_name and itp_residue_name:
    if gro_residue_name != itp_residue_name:
        residue_name_mismatch = True
        warnings.append(...)
```

### Anti-Patterns to Avoid
- **Disabling buttons unnecessarily:** Buttons should remain active after successful operations to allow repeated actions
- **Overly strict validation warnings:** "MOL" is a legitimate generic residue name that should not trigger warnings
- **Hiding bounds information:** Users need to know valid placement ranges before attempting manual placement

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|

**Key insight:** Reuse existing validation and parsing infrastructure. All core functionality already exists in the codebase.

## Common Pitfalls

### Pitfall 1: Button State Not Persisting After Validation
**What goes wrong:** Validate button becomes disabled after successful validation and preview
**Why it happens:** Button state may be modified in multiple places (file upload, mode change, validation, exception handlers)
**How to avoid:** Ensure validation success path does not modify button state; only error paths should disable
**Warning signs:** Button grayed out after green validation message, user cannot re-validate same position

### Pitfall 2: False Positive Residue Name Mismatch Warning
**What goes wrong:** Validator warns about "MOL" ≠ "etoh" even though this is correct behavior
**Why it happens:** Validator treats all residue name differences as potential mismatches
**How to avoid:** Suppress warning when GRO residue name is a known generic name ("MOL", "UNK", "LIG")
**Warning signs:** User sees warning dialog for valid etoh.itp/etoh.gro pair, confusion about file validity

### Pitfall 3: Missing Bounds Information for Manual Placement
**What goes wrong:** Users don't know valid xyz range for custom placement, get "Position outside liquid region" error
**Why it happens:** Liquid bounds calculated dynamically but never shown to user
**How to avoid:** Display liquid region bounds in Custom mode UI, update when interface structure changes
**Warning signs:** Users guessing position values, repeated validation failures

### Pitfall 4: No Volume Preview for Random Placement
**What goes wrong:** Users don't know how many molecules can fit in liquid region
**Why it happens:** Random mode only has molecule count input, no volume/occupancy feedback
**How to avoid:** Calculate and display liquid volume, estimate maximum molecules based on min_separation
**Warning signs:** Insertion failures with "Failed to place molecule X after 1000 attempts", user frustration

### Issue 2: False Positive Warning for GRO Resname "MOL"

**Confirmed:**
- etoh.gro uses "MOL" as residue name (verified in file content)
- etoh.itp uses "etoh" as moleculetype name (verified in file content)
- Validator compares these and flags mismatch (lines 148-160 in molecule_validator.py)
- This is CORRECT behavior in computational chemistry workflows

**Fix strategy:**
1. Define set of generic residue names: `{"MOL", "UNK", "LIG", "XXX", "RES"}`
2. Skip warning if GRO residue name is in generic set
3. Update warning message to explain this is expected

### Issue 3: Missing Liquid XYZ Range Documentation

**Current state:**
- Liquid bounds calculated in CustomMoleculeInserter.validate_single_placement()
- Bounds used for validation but never shown to user
- Users must guess valid position values

**Fix strategy:**
1. Add method to CustomMoleculePanel to calculate and display liquid bounds
2. Update bounds when interface structure changes
3. Show in Custom mode UI: "Liquid region: X=(min, max), Y=(min, max), Z=(min, max)"
4. Add to tooltip for position inputs

### Issue 4: Missing Volume Preview for Random Concentration

**Current state:**
- Random mode has molecule_count_spin input
- No indication of liquid volume or how many molecules fit
- Users may request too many molecules and get insertion failures

**Fix strategy:**
1. Calculate liquid volume from liquid_positions bounds
2. Estimate maximum molecules based on molecule size and min_separation
3. Display volume and suggested molecule count range
4. Add tooltip explaining estimation method

### Liquid Region Bounds Calculation
```python
# From custom_molecule_inserter.py lines 123-125
liquid_positions = structure.positions[ice_atom_count:ice_atom_count + water_atom_count]
min_coords = liquid_positions.min(axis=0)  # shape: (3,)
max_coords = liquid_positions.max(axis=0)  # shape: (3,)
```

### Volume Estimation Method
```python
# Approximate liquid volume
liquid_volume = np.prod(max_coords - min_coords)  # nm³

# Estimate max molecules (very rough approximation)
# Assumes min_separation is center-to-center distance
# Real packing is more complex due to molecule shape
molecule_radius = min_separation / 2  # nm
molecule_volume = (4/3) * np.pi * molecule_radius**3  # nm³
packing_fraction = 0.64  # random close packing
max_molecules_approx = int(liquid_volume * packing_fraction / molecule_volume)
```
