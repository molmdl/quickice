### Gaps Summary

**1. scorer.py orthorhombic path lacks boundary clamping (cross-cutting robustness gap)**

Plan 05 fixed the floating-point edge case `np.mod(-tiny, L) → exactly L` in overlap_resolver.py by adding boundary clamping (lines 69-76). The **same edge case exists unpatched** in scorer.py's orthorhombic path (lines 72-80). I empirically confirmed that `np.mod(-1e-16, 1.5)` returns exactly `1.5`, which cKDTree rejects with ValueError: "Some input data are greater than the size of the periodic box."

The scorer.py orthorhombic path does:
```python
o_wrapped[:, dim] = np.mod(o_wrapped[:, dim], cell_dims[dim])  # line 77
tree = cKDTree(o_wrapped, boxsize=box_list)                      # line 80
```

But does NOT add the boundary clamping that overlap_resolver.py has:
```python
on_boundary = ice_o_wrapped[:, dim] >= box_dims_nm[dim]          # overlap_resolver line 73
ice_o_wrapped[on_boundary, dim] -= box_dims_nm[dim]              # overlap_resolver line 74
```

**Practical impact:** The edge case is less likely to manifest in scorer.py than in overlap_resolver.py (which deals with tiling positions). Candidate positions from the generation system are typically well-formed. However, it IS a real vulnerability that violates the Plan 05 must-have "All coordinates passed to cKDTree(boxsize=) are strictly within [0, boxsize)" and contradicts the phase goal of "ensure scientific reliability."

**Fix required:** Add the same 4-line boundary clamping block after line 77 in scorer.py, before the cKDTree construction at line 80. This is a minimal, safe, physically correct fix (PBC boundary x==L is equivalent to x==0).
