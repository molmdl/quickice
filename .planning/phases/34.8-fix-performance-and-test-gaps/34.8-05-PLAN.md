Purpose: The PERF-02 boxsize optimization (Plan 34.8-01) changed overlap_resolver.py to use `cKDTree(boxsize=)`, which strictly requires all data within [0, boxsize). The fix: after np.mod wrapping, clamp any values exactly equal to boxsize down to 0 (they are at the PBC boundary).

In `detect_overlaps()` (lines 59-67), after the `np.mod` wrapping loop, add a clamping step that handles the floating-point edge case where `np.mod(-tiny_value, L)` returns exactly `L` instead of `L - tiny_value`.

The root cause: For positions that are slightly negative (like -1e-16 nm, which happens when triclinic tiling produces molecules whose COM wraps to near-zero fractional coordinates but individual atoms extend slightly below zero), `np.mod(-1e-16, 13.406...)` returns exactly `13.406...` because the subtraction `13.406... - 1e-16` rounds to exactly `13.406...` in float64 (the 1e-16 is below the representable precision for values ~13.4). cKDTree requires data STRICTLY less than boxsize, so it rejects this value.

The fix is simple and physically correct: any coordinate that equals exactly `boxsize` after wrapping is at the PBC boundary and should wrap to 0. Add this code AFTER the existing `np.mod` loop (after line 67):

```python
    # Handle floating-point edge case: np.mod(-tiny, L) can return exactly L
    # due to float64 precision limits. cKDTree requires data < boxsize,
    # so values exactly on the PBC boundary must wrap to 0.
    for dim in range(3):
        on_boundary = ice_o_wrapped[:, dim] >= box_dims_nm[dim]
        ice_o_wrapped[on_boundary, dim] -= box_dims_nm[dim]
        on_boundary = water_o_wrapped[:, dim] >= box_dims_nm[dim]
        water_o_wrapped[on_boundary, dim] -= box_dims_nm[dim]
```

This is physically correct: a position at exactly the PBC boundary (x == L) is the same as x == 0 in periodic boundary conditions. The subtraction `L - L = 0` is exact in float64.

Do NOT modify any other part of detect_overlaps or the overlap_resolver module. The fix is localized to the wrapping step only.
