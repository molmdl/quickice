# Phase 34.3: Tab Order Swap - Research

**Researched:** 2026-05-07
**Domain:** PySide6 tab management, UI workflow design
**Confidence:** HIGH

## Summary

This phase swaps the order of Solute Insertion (Tab 5) and Custom Molecule Upload (Tab 4) tabs to enable the desired workflow: Custom → Solute. Users should upload custom molecules first (now Tab 4), then use them as a source for solute insertion (now Tab 5). The swap requires updating the TabIndex enum values, reordering the addTab() calls in MainWindow, and verifying no hardcoded tab indices exist. The TabIndex enum pattern (created in Phase 32) makes this change straightforward and safe, as all references use named constants rather than magic numbers.

**Primary recommendation:** Update TabIndex enum to swap SOLUTE (3→4) and CUSTOM (4→3), then swap addTab() order in main_window.py. The existing TabIndex pattern eliminates risk of broken references.

## Architecture Patterns

### Pattern 1: TabIndex IntEnum for Type Safety

**What:** Named integer constants prevent hardcoded tab index bugs

**Why it works:**
```python
# GOOD: Type-safe, readable, refactorable
if current_idx == TabIndex.SOLUTE:
    self._on_export_solute_gromacs()

# BAD: Magic number, fragile, unclear meaning
if current_idx == 3:  # What is tab 3?!
    self._on_export_solute_gromacs()
```

**Current TabIndex definition (quickice/gui/constants.py):**

### Pattern 3: Workflow-Driven Tab Order

**What:** Tab order should reflect typical user workflow

**Desired workflow (Phase 34.3 goal):**
```
User workflow:
1. Generate Interface (Tab 3)
2. Upload Custom Molecules (Tab 4) ← First: prepare custom molecules
3. Insert Custom Molecules (Tab 4)
4. Insert Solutes (Tab 5) ← Second: can use custom molecules as source
5. Insert Ions (Tab 6) ← Third: can use solute structure as source
```

**Why this order matters:**
- Custom Molecule tab prepares molecules that can be used as solute sources
- Solute Insertion can now reference custom molecules uploaded earlier
- Enables Custom → Solute → Ion workflow chain

**Previous order (Phase 34):**
- Tab 4: Solute Insertion (can't reference custom molecules from Tab 5)
- Tab 5: Custom Molecule Upload (must come AFTER solutes - backward workflow)

### Anti-Patterns to Avoid

- **Hardcoded tab indices:** Using magic numbers (0, 1, 2, 3) instead of TabIndex constants
- **Mismatched enum and addTab order:** TabIndex.SOLUTE=3 but solute_panel added as 4th tab
- **Comments out of sync:** Comments saying "Tab 4" when code shows Tab 5

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tab index constants | Integer variables or magic numbers | TabIndex IntEnum | Type-safe, refactorable, self-documenting |
| Tab reordering logic | insertTab/removeTab shuffling | Swap addTab() order + enum values | Simpler, clearer, less error-prone |
| Cross-tab references | Hardcoded index checks | TabIndex constants | Works regardless of tab position |

**Key insight:** Phase 32's investment in TabIndex enum pays off here. No complex refactoring needed.

## Common Pitfalls

### Pitfall 1: Enum and addTab Order Mismatch

**What goes wrong:** TabIndex.SOLUTE=3 but solute_panel added as 5th tab (index 4)

**Why it happens:** Updating enum without updating addTab() calls, or vice versa

**How to avoid:** Always update both together:
1. Change TabIndex enum values
2. Change addTab() call order
3. Verify index==TabIndex match with debug print

**Warning signs:**
- Tab switches to wrong tab
- Export shortcut exports wrong structure
- Source dropdown routes to wrong tab

**Example violation:**
```python
# In constants.py
SOLUTE = 3  # Changed to 3

# In main_window.py
self.tab_widget.addTab(self.solute_panel, "Solute Insertion")  # Added 5th (index 4)!

# Result: TabIndex.SOLUTE (3) points to Custom tab!
```

### Pitfall 2: GROMACS Export Order Confusion

**What goes wrong:** Thinking tab order affects GROMACS molecule ordering

**Why it happens:** Misunderstanding that GROMACS-01 requirement is about export logic, not UI tab order

**How to avoid:** Understand that:
- **Tab order** = UI workflow order (which tab is Tab 4, Tab 5, etc.)
- **GROMACS molecule order** = Export output order (SOL → guests → solutes → custom → ions)
- These are **independent** systems

**Example confusion:**
```
WRONG: "If I swap tabs, GROMACS export will put solutes before custom molecules"
RIGHT: "GROMACS export ordering is hardcoded in export functions, not affected by tab positions"
```

### Pitfall 3: Forgetting Comment Updates

**What goes wrong:** Comments still say "Tab 4: Solute" after swapping

**Why it happens:** Changing code but not updating documentation/comments

**How to avoid:** Update all comments in:
1. TabIndex enum docstring (tab order comment block)
2. main_window.py _setup_ui() docstring (planned tab order comment)
3. Any inline comments near tab references

**Warning signs:**
- Comments contradict code
- Future developers confused by stale comments
- ARCHITECTURE.md shows outdated order

### GROMACS Export Molecule Ordering (Unaffected)

**Clarification:** GROMACS-01 requirement specifies export ordering:

```
GROMACS molecule order (fixed in export functions):
SOL (ice + water) → hydrate guests → liquid solutes → custom molecules → ions
```

This is **NOT** affected by tab order. Tab order is UI workflow order. GROMACS molecule order is export output order. They are independent.

**Example:**
- Tab 4: Custom Molecule (index 3)
- Tab 5: Solute Insertion (index 4)
- GROMACS export still outputs: SOL → ... → solutes → custom molecules → ions

The export functions have hardcoded ordering logic that does not depend on tab positions.
