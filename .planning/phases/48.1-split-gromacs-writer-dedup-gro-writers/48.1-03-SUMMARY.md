## Decisions Made
- **NA/CL helper format adjusted from plan placeholder to match source byte-for-byte** — The plan's placeholder template `f"{res_num:5d}NA       NA{atom_num_wrapped:5d}"` didn't match the source's actual column alignment. The source (write_ion_gro_file:1515-1517/1526-1528) uses `f"{res_num_wrapped:5d}NA   "` (resname field = "NA" + 3 trailing spaces = 5 chars) + `f"   NA{atom_num_wrapped:5d}"` (atom-name field = 3 leading spaces + "NA" = 5 chars). The plan explicitly anticipated this: "READ the exact format string from write_ion_gro_file BEFORE finalizing — the column alignment for 'NA' residue name and 'NA' atom name must match byte-for-byte. The placeholder format above may need adjustment." So this is an expected adjustment, documented in the helper docstring and locked by the test's column-alignment assertions (lines[0][5:10] == "NA   ", lines[0][10:15] == "   NA").
- **_write_gro_box_vectors documented as 5-of-6 (NOT all 6)** — Research §3's diff matrix claimed "Box vector write (`f"{cell[0,0]:10.5f}..."`) | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | Byte-identical (6 of 6)". But direct source read revealed write_custom_molecule_gro_file:2149 uses a DIVERGENT 3-value diagonal format: `f"{custom_structure.cell[0, 0]:10.5f}{custom_structure.cell[1, 1]:10.5f}{custom_structure.cell[2, 2]:10.5f}\n"` (only the diagonal, not the full 9-value triclinic). Corrected the helper docstring to say "Used by: 5 of 6" and documented the custom writer's divergence (DO NOT use this helper for that writer). The helper itself writes the correct 9-value triclinic format used by the 5 compatible writers. This is a pre-existing divergence, not introduced by the refactor.
- **compute_mw_position imported from _shared to satisfy key_links dependency pattern** — The helpers take `mw_pos` as a caller-computed parameter (they do NOT call compute_mw_position internally — the caller decides whether MW is computed from O,H,H 3-atom input or pulled from a 4-atom hydrated structure). The import establishes the `_gro_format → _shared` dependency link required by the plan's key_links must_have. It's marked `# noqa: F401` since it's re-exported for downstream per-structure modules' convenience, not called by the helpers.
- **TestWriteGroHeader parametrized over 3 n_atoms values to reach exactly 12 unit tests** — The plan's test structure listed 10 test classes with "12 unit tests" total. Parametrizing the header test over (42 basic, 99999 5-digit edge, 100000 overflow edge) yields 3 test cases, plus 1 test each for the other 9 helpers = 12 total. The 100000 case verifies `:5d` expands to 6 digits (GROMACS wraps atom numbers at 100000, but the header count is written as-is).
- **Helpers are pure formatters — caller owns all per-structure logic** — Helpers take `res_name` as a pre-validated parameter (caller calls validate_gro_residue_name), `res_num` as an already-wrapped value (caller does % 100000), and `atom_num_counter` as a 1-element list for in-place mutation. Helpers never call validate_gro_residue_name, never do the custom_active gate, never reorder atoms. This keeps the per-structure divergence (e.g., write_interface_gro_file uses `if custom_guest_info:` while the other 4 use `custom_active = (custom_guest_info is not None and len(...) > 0 and ...)`) in the per-structure writer, per research §7 risk #3.

### Auto-fixed Issues

**1. [Rule 1 - Bug] NA/CL helper format adjusted from plan placeholder to match source byte-for-byte**
- **Found during:** Task 1 (creating _format_na_ion / _format_cl_ion)
- **Issue:** The plan's placeholder template `f"{res_num:5d}NA       NA{atom_num_wrapped:5d}"` did not match the actual source format at write_ion_gro_file:1515-1517/1526-1528. The source uses two separate f-string fragments: `f"{res_num_wrapped:5d}NA   "` (resname field = "NA" + 3 trailing spaces) + `f"   NA{atom_num_wrapped:5d}"` (atom-name field = 3 leading spaces + "NA"). The plan's single-fragment "NA       NA" had 7 spaces between the two NA tokens (11 chars), but the source's two-fragment form produces 3 + 3 = 6 spaces (with the resname and atom-name fields each exactly 5 chars). Using the plan's placeholder verbatim would have produced byte-DIFFERENT output and broken ion grompp.
- **Fix:** Read the source at write_ion_gro_file:1515-1517 (NA) and :1526-1528 (CL), copied the two-fragment f-string byte-for-byte into `_format_na_ion` and `_format_cl_ion`. The helper docstring notes the adjustment. The unit tests assert exact column alignment (lines[0][5:10] == "NA   ", lines[0][10:15] == "   NA") to lock the format.
- **Files modified:** quickice/output/_gro_format.py (helpers 7 + 8)
- **Verification:** TestFormatNaIon and TestFormatClIon pass — helper output byte-identical to the inline source. All 14 byte-equivalence tests still pass (writers untouched, so no end-to-end change yet).
- **Committed in:** f522fb6b (Task 1 commit)

**2. [Rule 1 - Bug] _write_gro_box_vectors divergence documented (research §3 oversight corrected)**

- **Issue:** Research §3's diff matrix claimed "Box vector write | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | Byte-identical (6 of 6)" — i.e., all 6 GRO writers use the 9-value triclinic box format.
