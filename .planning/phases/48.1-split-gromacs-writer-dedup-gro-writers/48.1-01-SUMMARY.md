## Decisions Made
- **Use synthetic fixtures instead of GenIce2-based fixtures** — GenIce2's dipole optimization introduces run-to-run nondeterminism that `seed=42` does NOT control (verified: two identical `IceStructureGenerator(...).generate_all(1)` calls produce different position arrays). This breaks the byte-equivalence contract — the SHA256 baseline would change every run. Fix: use SYNTHETIC fixtures (hardcoded positions, no GenIce2) — the same pattern used by `tests/test_scancode_bugs_inserter_perf_rebuild.py` for its SHA256 golden snapshots.
- **ion path has 4 files (not 5)** — The plan said "5 files (ion.gro, ion.top, na.itp, cl.itp, tip4p-ice.itp)" but `na.itp` and `cl.itp` do NOT exist as separate files. `write_ion_itp` (at `quickice/structure_generation/gromacs_ion_export.py:87`) generates a single `ion.itp` containing BOTH the NA and CL `[moleculetype]` definitions. Actual: 4 files (ion.gro, ion.top, ion.itp, tip4p-ice.itp).
- **Box sizes ≥ 3.0 nm for all synthetic fixtures** — grompp's PBC rule requires the 1.0 nm cutoff to be less than half the shortest box vector (need shortest edge ≥ 2.0 nm). Using 3.0 nm gives a comfortable margin.
- **Hydrate uses MoleculetypeRegistry** with `register_hydrate_guest("CH4")` → `"CH4_H"` (matches `HydrateGROMACSExporter` built-in path; `write_multi_molecule_top_file` registry branch produces `"CH4_H"` in `[molecules]`).
- **Solute uses MoleculetypeRegistry** with `register_liquid_solute("CH4")` → `"CH4_L"` (matches `SoluteInserter` registry pattern; `write_solute_top_file` uses registry for the solute moleculetype name).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] GenIce2 nondeterminism breaks byte-equivalence contract**
- **Found during:** Task 2 (GREEN) — first baseline capture + verification
- **Issue:** The plan assumed `seed=42` would make GenIce2 deterministic, but GenIce2's dipole optimization introduces run-to-run nondeterminism that the seed does NOT control. Verified: two identical `IceStructureGenerator(phase_info, 192).generate_all(1)` calls produce different `positions` arrays (SHA256 `f6d8ca2e...` vs `d3295b42...`). This caused 6 of 7 byte-equivalence tests to FAIL on the first GREEN run (the baseline SHA256s captured by the capture script didn't match the SHA256s computed by the test run, because the GenIce2-generated fixtures differed between the two processes).
- **Fix:** Replaced ALL GenIce2-based fixtures with SYNTHETIC fixtures (hardcoded positions, no GenIce2 calls). Built helper functions `_water_3atom`, `_water_4atom`, `_ch4_5atom`, `_ethanol_9atom` that generate deterministic molecule positions. The synthetic fixtures are minimal (4-22 atoms per path) but exercise every writer code path: ice 3→4 MW expansion, water 4-atom pass-through, guest detect+reorder, ion NA/CL formatting, custom molecule, solute, custom_guest. Box sizes ≥ 3.0 nm on every edge satisfy grompp's PBC rule.
- **Files modified:** `tests/test_phase_48_1_gro_top_byte_equivalence.py` (fixtures section rewritten), `tests/_phase_48_1_capture_baseline.py` (fixture builders rewritten to mirror the synthetic fixtures)
- **Verification:** 14 PASSED in 4 consecutive runs (0.37-0.39s each — much faster than the 1.7s GenIce2-based version). All SHA256s stable across runs.
- **Committed in:** 5cfc9db6 (GREEN commit)
