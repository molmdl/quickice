## Decisions Made
- **Applied `_wrap_aux_positions` to interface's solute/custom-molecule position wrapping** — Plan instruction #9 says "Use the `_wrap_aux_positions` helper IF the inline code matches the helper's pattern. If the inline code uses `wrap_positions_into_box(positions, cell)` directly (full PBC wrap), keep that call — don't force the diagonal-modulo helper." The interface's solute/custom-molecule position wrapping (lines 329-334 of pre-refactor source) uses `% np.diag(iface.cell)` which is exactly the helper's pattern. The `hasattr(iface, '...') and ... is not None and len(...) > 0` guards stay at the call site because the helper does NOT abstract attribute presence (older/incomplete InterfaceStructure instances may lack the attribute entirely). Output bytes unchanged.
- **Kept the `wrap_positions_into_box` fallback inline and untouched** — Lines 318-323 of pre-refactor source (when `iface.molecule_index` is empty, fall back to atom-by-atom wrapping via `wrap_positions_into_box(iface.positions, iface.cell)`) is a DIFFERENT concern from the solute/custom-molecule position wrapping. Plan instruction #9 explicitly says to keep this call inline (don't force the diagonal-modulo helper when the inline code uses the full PBC wrap).
- **Moved header writes into the `lines` list (single `f.writelines` flush per writer)** — Plan offered two options: (a) move header into the lines list (call `_write_gro_header(lines, ..., n_atoms)` BEFORE the molecule loop, then `f.writelines(lines)` writes header + atoms in one call) OR (b) keep header as `f.write(...)` calls and only use the helper for the molecule loop. Chose (a) — the natural pattern for the helper. Output bytes are identical (just the order of `f.write` calls changes; the file content is the same). This establishes the pattern for the other 4 writers (48.1-05/06).
- **PRESERVED the `if custom_guest_info:` gate VERBATIM in `write_interface_gro_file`** — Research §7 risk #3 + plan instruction "CRITICAL: Preserve the `if custom_guest_info:` gate VERBATIM. Do NOT 'unify' it with the `custom_active = ...` pattern. The difference is intentional (interface path is older). Unifying would change byte-output for the empty-list case (`[]` is truthy-falsy differently)." Added an inline comment at the gate (lines 422-431 of the refactored source) explaining the divergence so 48.1-05/06 (which touch the other 4 GRO writers) don't accidentally "fix" it. The 14 byte-equivalence tests confirm the gate preservation (custom_guest path SHA256 unchanged).
- **All 10 helpers imported at the top of `gromacs_writer.py`** — Single import block (lines 56-65 of the refactored source) covering all 10 helpers from `_gro_format`.
