## Decisions Made
- **Sub-split _shared.py into aggregator + 6 domain modules** (SC-1 ≤ 800 lines) — AST analysis confirmed the 21 shared helpers sum to 901 lines of function code; adding constants (~86 lines), imports (~20), logger, and spacing yields ~1050 lines, exceeding the 800-line target. The sub-split distributes content across 6 domain-specific private modules plus a thin aggregator. All 7 new files stay well under 800 lines (largest is _itp.py at 342).
- **Aggregator re-export pattern** — _shared.py imports from all 6 sub-modules and re-exports. Downstream callers (gromacs_writer, _gro_format in Wave 2a, future per-structure writers in Waves 3-8) import ONLY from _shared. This gives plans 03/08-13 a stable import path that's agnostic to the internal sub-split.
- **validate_gro_residue_name lives in _itp.py** (not directly in _shared.py as the plan note suggested) — its primary consumer transform_guest_itp is also in _itp.py. Defining it in _shared would create a circular import (_shared → _itp → _shared). Co-locating it with transform_guest_itp preserves byte-identical function bodies while breaking the cycle. _shared re-exports it from _itp.
- **Scan-all-files test pattern** (Path('quickice/output').glob('*.py')) for source-text grep invariants — robust to future module moves. Applied to both test_tip4p_ice_lj_values.py (per plan) and test_scancode_bugs_constants.py (Rule 3 deviation).
- **gromacs_writer.py stays as a flat file** (NOT package conversion) — preserves the test_tip4p_ice_lj_values.py file-path reads (Path("quickice/output/gromacs_writer.py").read_text() still works; the re-export shim doesn't contain forbidden values but the scan-all-files pattern catches them in sub-modules anyway).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] validate_gro_residue_name placement (circular import)**
- **Found during:** Task 1 (creating _itp.py)
- **Issue:** The plan note said "validate_gro_residue_name is defined DIRECTLY in _shared.py (not imported from a sub-module)". But transform_guest_itp (in _itp.py) calls validate_gro_residue_name at runtime. If validate_gro_residue_name lives in _shared.py, then _itp.py would need to import it from _shared → circular import (_shared → _itp → _shared). The plan's "sub-modules MUST NOT import from _shared" rule conflicts with the plan's "validate_gro_residue_name defined directly in _shared" note.
- **Fix:** Placed validate_gro_residue_name in _itp.py (same module as its primary consumer transform_guest_itp). _shared.py re-exports it from _itp.py. All function bodies stay byte-identical to the pre-refactor source. The 12 inline writers in gromacs_writer.py get it via `from quickice.output._shared import validate_gro_residue_name` (works via re-export).
- **Files modified:** quickice/output/_itp.py (validate_gro_residue_name defined here), quickice/output/_shared.py (re-exports from _itp)
- **Verification:** All 14 byte-equivalence tests pass; all 78 PBC/GRO-resname/ITP tests pass; all 18 e2e grompp tests pass
- **Committed in:** baa04c37 (Task 1 commit)

**2. [Rule 3 - Blocking] test_scancode_bugs_constants.py source-text grep tests broke**

## Issues Encountered
- **Parallel bash race during injection test:** The manual injection test (inject "# 0.31668e-3 was the bug" into _constants.py, verify test FAILS, revert) had a race condition — I sent `cp` (backup) and `echo >>` (inject) as separate bash calls in the same message, which ran in parallel. The cp backed up the file AFTER the echo appended to it, so the backup was dirty. Fixed by using the Edit tool to remove the injected line from both _constants.py and _shared.py (cleaner than cp-based restore). The injection test itself confirmed the scan-all-files pattern catches regressions in both _constants.py and _shared.py (both tests FAILED as expected).
- **gromacs_writer.py git add refused:** Plain `git add` for gromacs_writer.py refused with "paths are ignored" because the output/ directory is gitignored. Used `git add -f` to force-stage (same precedent as STATE [41-02] and the 7 new sub-modules in Task 1).
