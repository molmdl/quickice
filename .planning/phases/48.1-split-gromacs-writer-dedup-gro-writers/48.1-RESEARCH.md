# Phase 48.1 Research: Split gromacs_writer.py + dedup GRO writers

**Researched:** 2026-07-19
**Domain:** GROMACS export module refactor (Python, NumPy, structure-generation dataclasses)
**Confidence:** HIGH (all facts verified by direct read of current source at HEAD `d224d074`; `wc -l`, `grep`, AST parse, git log)

## Summary

`quickice/output/gromacs_writer.py` is currently **4139 lines** (not 4067 as the roadmap says — the line count grew after Group 1/5/8 cleanup added triclinic PBC branches, extracted TIP4P_ICE constants, and replaced ad-hoc `type()` objects with `MoleculeIndex`). It contains **33 top-level functions**: 6 GRO writer functions (1739 lines, 42% of file), 6 TOP writer functions (1313 lines, 32%), and 17 shared helpers + module-level constants (~1085 lines, 26%). Group 1/5/8 cleanup from `74fc4b72` is **already applied** to the current source — the split must preserve those fixes (triclinic PBC at lines 310-410, `TIP4P_ICE_*` constants at 55-68, `MoleculeIndex` at all 7 former ad-hoc sites).

The 6 GRO writers share **~590 lines of near-byte-identical code** (SOL ice 3→4 expansion, SOL water pass-through, guest detect+reorder+format, custom/solute format, GRO header/box-vector write). Extracting these into a private `_gro_format.py` module saves ~390 lines and eliminates the SOLUTE-PBC-class bug factory (the historical bug: wrapping existed in 2 writers but was missing from the 3rd — exactly the duplication risk TD-01 flags).

**Primary recommendation:** Split into **per-structure sibling modules** (`ice_writer.py`, `interface_writer.py`, `multi_molecule_writer.py`, `ion_writer.py`, `custom_writer.py`, `solute_writer.py`) plus a private `_shared.py` (constants + cross-cutting helpers) and a private `_gro_format.py` (DRY-extracted GRO formatting helpers). Keep `gromacs_writer.py` as a **thin re-export shim** so all 67 import sites (34 distinct symbols, including private `_merge_custom_atomtypes` / `_check_custom_atomtype_conflict`) continue to work unchanged. **Use the sibling-modules approach, NOT package conversion** — `tests/test_tip4p_ice_lj_values.py:273,278,283` reads `Path("quickice/output/gromacs_writer.py").read_text()` to grep for forbidden values; converting to a package would break that test.

Verify with the established **SHA256 golden-snapshot pattern** (precedent: `tests/test_scancode_bugs_inserter_perf_rebuild.py:109` uses `hashlib.sha256(arr.tobytes()).hexdigest()` for byte-equivalence). Apply the same pattern to `.gro`/`.top`/`.itp` file bytes: snapshot before refactor → re-export after → assert SHA256 matches. `gmx` is available at `/data/nglokwan/ompi_plumed-gmx/plumed-gromacs2023.5-gpu/bin/gmx` (GROMACS 2023.5-plumed_2.9.3), so `gmx grompp` validation can run (not skipped).

## 2. Module Boundary Options

### Option A: By output type (2 main files)
```
quickice/output/
├── gromacs_writer.py    # thin re-export shim (~60 lines)
├── gro_writers.py       # 6 GRO writers (~1740 lines) — EXCEEDS 800-line target
├── top_writers.py       # 6 TOP writers (~1310 lines) — EXCEEDS 800-line target
└── _shared.py           # constants + 17 helpers (~1085 lines) — EXCEEDS 800-line target
```
**Pros:** Simple 2-axis split (GRO vs TOP).
**Cons:** All 3 substantive files exceed the 800-line target. `gro_writers.py` would be 1740 lines — barely better than the monolith. **Reject.**

### Option B: By structure type (6 writer files + shared)
```
quickice/output/
├── gromacs_writer.py         # thin re-export shim (~60 lines)
├── _shared.py                # constants + 17 helpers (~1085 lines) — EXCEEDS 800-line target
├── ice_writer.py             # ~210 lines (write_gro_file, write_top_file, get_tip4p_itp_path, compute_mw_position)
├── interface_writer.py      # ~640 lines (write_interface_gro_file, write_interface_top_file, _get_molecule_atoms, detect_guest_type_from_atoms)
├── multi_molecule_writer.py # ~355 lines (write_multi_molecule_gro_file, write_multi_molecule_top_file)
├── ion_writer.py             # ~760 lines (write_ion_gro_file, write_ion_top_file) — near limit
├── custom_writer.py          # ~530 lines (write_custom_molecule_gro_file, write_custom_molecule_top_file)
└── solute_writer.py          # ~730 lines (write_solute_gro_file, write_solute_top_file) — near limit
```
**Pros:** Each per-structure file groups its GRO+TOP pair (coherent editing unit). Matches the roadmap's example split (`ion_writer.py`, `shared.py`).
**Cons:** `_shared.py` exceeds 800 lines (must be sub-split). `ion_writer.py` (760) and `solute_writer.py` (730) are near the limit BEFORE DRY extraction — after DRY they shrink to ~450 each. **Acceptable with DRY extraction.**

### Option C: Hybrid (per-structure + DRY-extracted GRO format helpers) — **RECOMMENDED**
```
quickice/output/
├── gromacs_writer.py            # thin re-export shim (~70 lines) — preserves all 34 imported symbols
├── _shared.py                   # ~600 lines: constants, atomtype helpers, PBC, ITP transform, guest detection, TIP4P
├── _gro_format.py               # ~250 lines: DRY-extracted GRO formatting helpers (TD-01)
├── ice_writer.py                # ~150 lines: write_gro_file, write_top_file
├── interface_writer.py          # ~400 lines: write_interface_gro_file, write_interface_top_file
├── multi_molecule_writer.py     # ~200 lines: write_multi_molecule_gro_file, write_multi_molecule_top_file
├── ion_writer.py                # ~450 lines: write_ion_gro_file, write_ion_top_file (after DRY, down from 760)
├── custom_writer.py             # ~350 lines: write_custom_molecule_gro_file, write_custom_molecule_top_file (after DRY)
└── solute_writer.py             # ~450 lines: write_solute_gro_file, write_solute_top_file (after DRY)
```

**Why recommended:**
1. **Every file under 800 lines** (largest: `ion_writer.py` and `solute_writer.py` at ~450 after DRY).
2. **Each per-structure file is a coherent editing unit** (GRO + TOP pair together — matches how the code is actually maintained).
3. **`_gro_format.py` is the TD-01 deliverable** — extracted DRY helpers go here, private (underscore prefix), shared across the 6 GRO writers.
4. **`_shared.py` holds the cross-cutting concerns** (constants, PBC, ITP transform, guest detection) — also private.
5. **`gromacs_writer.py` stays as a thin re-export shim** — preserves all 67 import sites (34 distinct symbols) without a single import rewrite. Critical for not breaking `tests/test_tip4p_ice_lj_values.py:273,278,283` which reads `Path("quickice/output/gromacs_writer.py").read_text()`.

**Reject package conversion (e.g. `gromacs_writer/__init__.py`):** `tests/test_tip4p_ice_lj_values.py` reads `quickice/output/gromacs_writer.py` as a TEXT FILE to grep for forbidden values (`"0.31668e-3"`, `"0.88216e-6"`, comb-rule=1 pattern). Converting `gromacs_writer.py` to a package (directory + `__init__.py`) would make `Path("quickice/output/gromacs_writer.py").read_text()` raise `IsADirectoryError`. **Use sibling modules, keep `gromacs_writer.py` as a flat file.**

```python
    Used by: write_gro_file, write_interface_gro_file, write_ion_gro_file,

        f"{res_num:5d}SOL    HW2{atom_num_wrapped:5d}{h2_pos[0]:8.3f}..."

    """Write the 9-value triclinic box vector line.
    Used by: all 6 GRO writers (identical format).

    """

def _wrap_aux_positions(positions: np.ndarray | None, cell: np.ndarray) -> np.ndarray | None:
    """PBC-wrap solute/custom-molecule positions via diagonal modulo (AN-03 fix).
    Used by: write_interface_gro_file, write_ion_gro_file, write_solute_gro_file (3 of 6).
    Pattern: `positions % np.diag(cell)` (the AN-03 fix pattern).
    """
```

### TOP writers: same approach viable? — **Partially, with caution**

TOP writers are **more divergent** than GRO writers:
- `write_top_file` (ice-only, 58 lines): Writes `[defaults]`, `[atomtypes]` (3 water types), `[moleculetype] SOL`, `[atoms]`, `[settles]`, `[virtual_sites3]`, `[exclusions]`, `[system]`, `[molecules]`. Self-contained, minimal duplication.
- `write_interface_top_file` (199 lines): Adds guest atomtype blocks (CH4/THF/CO2/H2 from `GAFF2_ATOMTYPES`), `#include` directives, guest `[molecules]` entries.
- `write_multi_molecule_top_file` (205 lines): Generic — iterates `molecule_index`, emits `#include` per mol_type, `[molecules]` entries.
- `write_ion_top_file` (319 lines): Adds `NA`, `CL` `[molecules]` + `#include "na.itp"`, `#include "cl.itp"`.
- `write_custom_molecule_top_file` (232 lines): Adds custom molecule `[molecules]` + `#include` for custom ITP.
- `write_solute_top_file` (300 lines): Adds solute `[molecules]` + `#include` for solute ITP.

**Shared patterns across TOP writers (DRY candidates):**
- `[defaults]` block (comb-rule=2 — `1 2 yes 0.5 0.8333`) — byte-identical across all 6 TOP writers. ~10 lines × 6 = ~60 lines dedup-able.
- `[atomtypes]` block emission (water types always; CH4/THF/CO2/H2 conditional; NA/CL conditional; custom conditional) — already partially shared via `_write_atomtypes_block` / `_format_atomtype_line` (lines 122-181). The per-writer logic that decides WHICH atomtypes to emit is divergent.
- `[system]` + `[molecules]` section write — similar structure but different content per writer.
- `#include` directive emission — same pattern, different file lists.

**Recommendation for TOP writers:**
- **Extract the `[defaults]` block** into `_shared._write_top_defaults(f)` — ~10 lines saved × 6 = ~50 lines.
- **Leave the rest per-structure** — the divergence in `[atomtypes]`/`[molecules]`/`#include` logic makes further DRY extraction cost more readability than it saves. The TOP writers are not the source of the historical SOLUTE-PBC-class bug (that was GRO-only).
- The bulk of the TOP-writer shared work (`_write_atomtypes_block`, `_format_atomtype_line`, `_check_custom_atomtype_conflict`, `_merge_custom_atomtypes`) is **already extracted** into shared helpers at lines 122-292. **No further TOP-writer DRY extraction is needed for this phase.**

**Key insight:** The TD-01 scope is "6 near-identical GRO writer functions." TOP writers are NOT in the TD-01 scope. Focus DRY on GRO; for TOP, only extract the `[defaults]` block (a 10-line freebie).

### Patterns to reuse for the snapshot approach

The `74fc4b72` commit message confirms: "All fixes verified with golden-snapshot byte-equivalence tests (PERF-01/02/03) and env-var equivalence tests (VTK-DUP)." The SHA256 golden-snapshot pattern is the **established verification approach** for refactors in this repo. The PERF-01/02/03 commits (`87b07779`, `a438e1cb`, `03c515b2`) each added a `test_*_golden_snapshot` test that captures the pre-fix SHA256 and asserts the post-fix code produces the same SHA256.

**Reuse this pattern verbatim** for the GRO/TOP byte-equivalence checks. The pattern is:
1. Add a test that captures the expected SHA256 as a hardcoded string in the test file.
2. The test runs the code, computes the actual SHA256, asserts they match.
3. If the refactor changes any byte, the SHA256 mismatches and the test fails with a clear message.

For file bytes (not numpy arrays), the helper is:
```python
def _file_sha256(path: Path) -> str:
    return hashlib.sha256(path.read_bytes()).hexdigest()
```
