# Architecture

**Analysis Date:** 2026-04-13

## Pattern Overview

**Overall:** Pipeline + MVVM hybrid

**Key Characteristics:**
- CLI path: Sequential pipeline (validation → phase mapping → structure generation → ranking → output)
- GUI path: MVVM (Model-ViewModel-View) with background workers on QThread
- Core domain logic is shared between CLI and GUI — neither depends on the other
- Data flows through typed dataclasses (`Candidate`, `GenerationResult`, `RankedCandidate`, `RankingResult`, `OutputResult`, `InterfaceStructure`)
- Phase mapping uses curve-based boundary evaluation (not polygon containment) for scientific accuracy
- All internal coordinates in nanometers (nm); TIP3P ice generated by GenIce2 normalized to TIP4P-ICE at GROMACS export time

## Layers

**Presentation (CLI):**
- Purpose: Parse CLI arguments, display results to terminal
- Location: `quickice/cli/`
- Contains: Argument parser, CLI-specific validators
- Depends on: `quickice.validation`, `quickice.main`
- Used by: `quickice.py` entry point

**Presentation (GUI):**
- Purpose: Interactive ice structure generation and visualization
- Location: `quickice/gui/`
- Contains: PySide6 widgets, VTK viewers, matplotlib canvas, MVVM ViewModel, QThread workers
- Depends on: All core modules (`phase_mapping`, `structure_generation`, `ranking`, `output`)
- Used by: `python -m quickice.gui` entry point

**Orchestration:**
- Purpose: Wire pipeline stages together into a single execution flow
- Location: `quickice/main.py` (CLI), `quickice/gui/viewmodel.py` + `quickice/gui/workers.py` (GUI)
- Contains: Sequential call chain from phase lookup through output
- Depends on: All pipeline modules
- Used by: CLI entry point, GUI ViewModel

**Domain — Phase Mapping:**
- Purpose: Determine which ice polymorph is stable at given T,P conditions
- Location: `quickice/phase_mapping/`
- Contains: Curve-based lookup, IAPWS melting equations, solid-solid boundary interpolation, triple point data, density calculators
- Depends on: `iapws` (for Liquid-Vapor boundary only)
- Used by: `quickice/main.py`, `quickice/gui/workers.py`, `quickice/output/phase_diagram.py`, `quickice/gui/phase_diagram_widget.py`

**Domain — Structure Generation:**
- Purpose: Generate ice crystal structures using GenIce2 and build ice-water interfaces
- Location: `quickice/structure_generation/`
- Contains: GenIce2 wrapper, phase-to-lattice mapper, supercell calculator, interface builder (slab/pocket/piece modes), water filler, overlap resolver
- Depends on: `genice2`, `quickice.phase_mapping` (for density metadata)
- Used by: `quickice/main.py`, `quickice/gui/workers.py`

**Domain — Ranking:**
- Purpose: Score and rank generated ice structure candidates
- Location: `quickice/ranking/`
- Contains: Energy scorer (O-O distance heuristic), density scorer, diversity scorer, weighted combination
- Depends on: `quickice.structure_generation.types` (Candidate type), `scipy` (cKDTree)
- Used by: `quickice/main.py`, `quickice/gui/workers.py`

**Domain — Output:**
- Purpose: Write results to files (PDB, GROMACS, phase diagrams)
- Location: `quickice/output/`
- Contains: PDB writer, GROMACS writer, phase diagram generator, structure validator (spglib), orchestrator
- Depends on: `quickice.ranking.types`, `quickice.phase_mapping` (for diagram boundary curves), `spglib`, `matplotlib`, `shapely`
- Used by: `quickice/main.py`, `quickice/gui/export.py`

**Validation:**
- Purpose: Validate user inputs (temperature, pressure, molecule count, box dimensions)
- Location: `quickice/validation/`
- Contains: CLI validators (raise `ArgumentTypeError`)
- Depends on: None (standalone)
- Used by: `quickice/cli/parser.py`, `quickice/gui/validators.py` (separate GUI-specific validators)

**Static Data:**
- Purpose: Bundled reference files (GROMACS topology templates)
- Location: `quickice/data/`
- Contains: `tip4p-ice.itp` (TIP4P-ICE topology), `tip4p.gro` (water template for interface generation)
- Depends on: None
- Used by: `quickice/output/gromacs_writer.py`, `quickice/structure_generation/water_filler.py`

## Data Flow

**CLI Ice Generation Pipeline:**

1. User runs `python quickice.py -T 300 -P 100 -N 96`
2. `quickice/cli/parser.py` parses and validates arguments via `quickice/validation/validators.py`
3. `quickice/main.py:main()` calls `lookup_phase(T, P)` from `quickice/phase_mapping/lookup.py`
   - Hierarchical curve evaluation checks boundaries from high-P to low-P phases
   - Returns phase_info dict: `{"phase_id", "phase_name", "density", "temperature", "pressure"}`
   - Ice Ih density calculated via IAPWS R10-06(2009) in `quickice/phase_mapping/ice_ih_density.py`
4. `generate_candidates(phase_info, nmolecules)` in `quickice/structure_generation/generator.py`
   - Creates `IceStructureGenerator`, maps phase to GenIce lattice name via `quickice/structure_generation/mapper.py`
   - Calculates supercell (n×n×n) to reach target molecule count
   - Generates 10 candidates with sequential seeds using GenIce2 + TIP3P water model
   - Returns `GenerationResult` with list of `Candidate` dataclasses
5. `rank_candidates(candidates)` in `quickice/ranking/scorer.py`
   - Scores each candidate: energy (O-O distance deviation), density (deviation from expected), diversity (seed uniqueness)
   - Normalizes scores to 0-1, combines with equal weights (lower combined = better)
   - Returns `RankingResult` with `RankedCandidate` list sorted by combined_score
6. `output_ranked_candidates(ranking_result)` in `quickice/output/orchestrator.py`
   - Writes top 10 PDB files with CRYST1 records via `quickice/output/pdb_writer.py`
   - Validates each structure (space group via spglib, atomic overlap via cKDTree) — warnings only, non-fatal
   - Generates phase diagram PNG/SVG via `quickice/output/phase_diagram.py`
7. Optional: `quickice/output/gromacs_writer.py` writes `.gro`, `.top`, `.itp` files
   - Converts 3-atom TIP3P (O, H, H) to 4-atom TIP4P-ICE (OW, HW1, HW2, MW) at export time
   - MW virtual site computed from O and H positions using `TIP4P_ICE_ALPHA = 0.13458335`

**CLI Interface Generation Pipeline:**

1. User runs `python quickice.py --interface --mode slab -T 250 -P 100 -x 5 -y 5 -z 10 --ice-thickness 3 --water-thickness 4`
2. Steps 2-4 same as above (phase lookup + candidate generation with fixed 256 molecules)
3. `quickice/structure_generation/interface_builder.py` → `validate_interface_config()` validates config and candidate
   - Checks box dimensions, mode-specific params, Ice II rejection (rhombohedral constraint)
4. `generate_interface(candidate, config)` routes to mode assembly:
   - `modes/slab.py` → `assemble_slab()`: ice | water | ice sandwich along Z
   - `modes/pocket.py` → `assemble_pocket()`: water cavity carved from ice matrix
   - `modes/piece.py` → `assemble_piece()`: ice crystal fragment centered in water box
5. Each mode uses `quickice/structure_generation/water_filler.py` for water placement and `quickice/structure_generation/overlap_resolver.py` for PBC-aware overlap removal
6. Returns `InterfaceStructure` with combined ice+water atoms, `ice_atom_count` boundary marker, generation report
7. Optional: `quickice/output/gromacs_writer.py` → `write_interface_gro_file()`, `write_interface_top_file()`

**GUI Ice Generation Flow (MVVM):**

1. User enters T, P in `InputPanel` or clicks `PhaseDiagramPanel` → clicks Generate
2. `MainWindow._on_generate_clicked()` → `MainViewModel.start_generation(T, P, nmol)`
3. ViewModel creates `GenerationWorker` (QObject), moves to `QThread`, connects signals, starts thread
4. Worker runs same pipeline as CLI (lookup → generate → rank), emitting progress/status signals
5. ViewModel forwards signals: `generation_progress(int)`, `generation_status(str)`, `ranked_candidates_ready(object)`
6. View updates: `ProgressPanel`, `InfoPanel` log, `DualViewerWidget` loads candidates via VTK
7. Phase diagram shows user's (T,P) marker; bidirectional binding: input fields ↔ diagram coordinates

**GUI Interface Generation Flow (Tab 2):**

1. User selects candidate from Tab 1 dropdown in `InterfacePanel`, configures mode parameters
2. `MainWindow._on_interface_generate(candidate_index)` → creates `InterfaceConfig.from_dict()`
3. `MainViewModel.start_interface_generation(candidate, config)` → `InterfaceGenerationWorker`
4. Worker calls `generate_interface(candidate, config)` → returns `InterfaceStructure`
5. Signal chain: `interface_generation_complete` → `InterfaceViewerWidget` displays structure
6. Export: `InterfaceGROMACSExporter` writes interface `.gro`/`.top`/`.itp` files

**State Management:**
- CLI: Stateless pipeline — each run is independent
- GUI: ViewModel holds `_last_ranking_result` and `_last_interface_result` for cross-tab access and export
- No persistent state between sessions — all state is transient (in-memory)
- Tab 2 reads Tab 1 candidates via `MainViewModel.get_last_ranking_result()`

## Key Abstractions

**Candidate (`quickice/structure_generation/types.py`):**
- Purpose: Represents a single generated ice structure with atomic positions, cell vectors, metadata
- Examples: Created by `IceStructureGenerator._generate_single()`, consumed by ranking and output
- Pattern: Dataclass with numpy arrays (`positions`, `cell`), `atom_names` list, `metadata` dict
- Note: `positions` and `cell` are in nm; cell stored as ROW vectors (position @ cell for fractional → Cartesian)

**GenerationResult (`quickice/structure_generation/types.py`):**
- Purpose: Bundles multiple Candidates with generation metadata (requested vs actual molecule count)
- Examples: Return type of `generate_candidates()`
- Pattern: Dataclass container with `was_rounded` flag for crystal symmetry rounding

**RankedCandidate (`quickice/ranking/types.py`):**
- Purpose: A Candidate with scoring breakdown and rank
- Examples: Output of `rank_candidates()`, input to PDB writer and VTK viewer
- Pattern: Dataclass wrapping a Candidate reference + `energy_score`, `density_score`, `diversity_score`, `combined_score`, `rank`

**InterfaceConfig (`quickice/structure_generation/types.py`):**
- Purpose: Configuration dataclass capturing all interface generation parameters from UI
- Examples: Created by `InterfacePanel.get_configuration()`, validated by `validate_interface_config()`
- Pattern: Validated dataclass with `__post_init__` for overlap_threshold range check, `from_dict()` classmethod for UI→domain conversion

**InterfaceStructure (`quickice/structure_generation/types.py`):**
- Purpose: Combined ice + water atomic positions with phase boundary marker
- Examples: Output of `generate_interface()`, input to VTK `InterfaceViewerWidget`
- Pattern: Dataclass with `ice_atom_count` marking the ice/water split index; ice atoms come FIRST, then water atoms

**Phase Info Dict (`quickice/phase_mapping/lookup.py`):**
- Purpose: Result of phase lookup containing `phase_id`, `phase_name`, `density`, `temperature`, `pressure`
- Examples: Return type of `lookup_phase()`, passed to `generate_candidates()`
- Pattern: Plain dict with known keys (not a dataclass — legacy from polygon-based design)

**ScoringConfig (`quickice/ranking/types.py`):**
- Purpose: Configuration for scoring parameters (ideal O-O distance, cutoff)
- Examples: Passed to `energy_score()` and `rank_candidates()`
- Pattern: Dataclass with sensible defaults (0.276 nm ideal O-O, 0.35 nm cutoff)

**Custom Exception Hierarchy:**
- Purpose: Domain-specific errors with context (T, P, mode, phase_id)
- Examples: `quickice/phase_mapping/errors.py`, `quickice/structure_generation/errors.py`
- Pattern: Base error per module with specialized subclasses; `UnknownPhaseError` carries `temperature`/`pressure` attrs; `InterfaceGenerationError` carries `mode` attr and prefixes message with `[mode]`

## Entry Points

**CLI Entry Point:**
- Location: `quickice.py` (project root)
- Triggers: `python quickice.py --temperature 300 --pressure 100 --nmolecules 96`
- Responsibilities: Imports `quickice.main:main()`, calls `sys.exit(main())`

**GUI Entry Point:**
- Location: `quickice/gui/__main__.py`
- Triggers: `python -m quickice.gui`
- Responsibilities: Imports `run_app()`, creates `QApplication`, `MainWindow`, starts Qt event loop

**Main Module Entry Point:**
- Location: `quickice/main.py:main()`
- Triggers: Called by `quickice.py`
- Responsibilities: Parses args, runs full pipeline (lookup → generate → rank → output), handles errors, returns exit code

**Direct Script Execution:**
- Location: `quickice/output/phase_diagram.py` (has `if __name__ == "__main__"` block)
- Triggers: `python quickice/output/phase_diagram.py [T] [P]`
- Responsibilities: Standalone phase diagram generation for testing

## Error Handling

**Strategy:** Fail-fast with typed exceptions; each module defines its own error hierarchy

**Patterns:**
- Domain errors use custom exception hierarchies:
  - `PhaseMappingError` → `UnknownPhaseError` (`quickice/phase_mapping/errors.py`) — carries `temperature`, `pressure` attrs
  - `StructureGenerationError` → `UnsupportedPhaseError`, `InterfaceGenerationError` (`quickice/structure_generation/errors.py`) — `InterfaceGenerationError` carries `mode` attr
- CLI catches exceptions in `quickice/main.py:main()`: `UnknownPhaseError` (exit 1), `InterfaceGenerationError` (exit 1), generic `Exception` (exit 1); prints to `sys.stderr`
- GUI workers catch exceptions in `run()` methods, emit `error` signal with string message
- GUI `MainWindow` shows `QMessageBox.critical()` for generation errors
- Validation errors: CLI uses `ArgumentTypeError` (argparse convention), GUI returns `(bool, str)` tuples for inline labels
- Interface generation validates config BEFORE starting expensive operations (`validate_interface_config()`) — fail-fast with descriptive error messages including "How to fix" instructions
- Structure validation (spglib) and overlap checking are non-fatal — logged as warnings via Python `logging`, not errors
- Output orchestrator catches per-candidate failures, logs warnings, continues with remaining candidates

## Cross-Cutting Concerns

**Logging:** Python `logging` module used in `quickice/output/orchestrator.py` and `quickice/output/phase_diagram.py` for non-fatal issues. GUI uses `InfoPanel` log display fed by ViewModel `generation_status` and `interface_generation_status` signals.

**Validation:** Dual validation systems — CLI validators raise `ArgumentTypeError` (`quickice/validation/validators.py`), GUI validators return `(bool, str)` tuples (`quickice/gui/validators.py`). Same rules, different error propagation. GUI molecule count max is 216 (vs CLI 100000).

**Unit Conversions:** Positions and cell vectors are consistently in nanometers (nm) internally. PDB writer converts to Ångströms (`* 10.0`). GROMACS writer keeps nm. VTK viewer converts nm → Å for rendering. `InterfaceConfig.overlap_threshold` validates nm range [0.1, 1.0] to catch Å/nm unit mismatches. Conversion helpers in `quickice/structure_generation/overlap_resolver.py`: `angstrom_to_nm()`, `nm_to_angstrom()`.

**Thread Safety:** GenIce2 uses global `np.random` state (not thread-safe). `IceStructureGenerator._generate_single()` saves/restores global state around each call. QuickIce is designed for sequential execution; concurrent generation is explicitly unsupported. GUI uses QThread worker-object pattern (NOT QThread subclassing). Workers import modules inside `run()` to avoid blocking main thread. Cancellation via `QThread.requestInterruption()` + `wait(100ms)` timeout.

**VTK Availability:** VTK rendering may fail in SSH/X11 forwarding environments. Both `quickice/gui/view.py` and `quickice/gui/interface_panel.py` detect `_VTK_AVAILABLE` at import time and degrade gracefully (show log message instead of 3D viewer). `QUICKICE_FORCE_VTK=true` env var overrides detection.

**Security:** Output path traversal protection in `quickice/output/orchestrator.py` — output directory must be within current working directory.
