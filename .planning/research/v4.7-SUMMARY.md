# Project Research Summary: QuickIce v4.7

**Project:** QuickIce v4.7 — Extended Hydrate Generation (v4.6 free wins + P3 fix + v4.7 custom guest in hydrate)
**Domain:** Molecular simulation structure preparation — clathrate hydrate + filled ice generation
**Researched:** 2026-06-27
**Confidence:** HIGH

## Executive Summary

QuickIce v4.7 extends the hydrate generation pipeline from 2 guest types on 3 lattices (sI/sII/sH with CH₄/THF) to support 5+ new guest molecules (CO₂, H₂, ethane, custom), 9+ new lattice types (filled ices C0/C1/C2/Ih, Ice XVI/XVII, sT'), 4 water models (TIP4P-ICE/TIP4P-2005/TIP3P/SPC/E), mixed cage occupancy, and custom guest upload. The core finding from all four research files is that **GenIce2's existing API supports every feature except sT' guest placement and custom guest module registration** — both have verified workarounds. The recommended approach is an extend-don't-replace strategy: extend `HydrateConfig` with new fields, refactor `_build_molecule_index()` to be metadata-driven, and inject custom guests via `sys.modules` before calling GenIce2.

The key risks are: (1) **GRO residue name overflow** — Python's `:<5s` format does NOT truncate long names, it silently corrupts the fixed-width GRO format by overflowing into the atom name column; (2) **`_build_molecule_index()` is the single-point bottleneck** — it hardcodes TIP4P water recognition (residue "ICE", atoms OW/HW1/HW2/MW) and fails for ALL non-TIP4P water models and custom guests; (3) **Thread safety gap** — `sys.modules` injection for custom guests must happen outside the existing `_genice_lock` scope, creating a race condition in QThread. All three have clear mitigations: validate residue names at write time, refactor molecule_index to use config-provided metadata, and register custom guest modules in the main thread before starting HydrateWorker.

## Key Findings

### Recommended Stack

No new dependencies required. v4.7 uses the existing `environment.yml` stack exclusively. GenIce2 2.2.13.1 provides all lattice/guest/water-model plugins via its Python API. The `genice2-cif` plugin is already installed but **deferred** to v7.0 (needs validation layer). All claims verified by running actual Python code in the `quickice` conda environment (Python 3.14.3).

**Core technologies (unchanged):**
- **GenIce2 2.2.13.1**: Ice/hydrate structure generation — API-exposed for all v4.7 features
- **scipy cKDTree**: Collision detection — working, no changes needed
- **VTK 9.5.2**: Multi-actor rendering — working, element map extension only
- **PySide6 6.10.2**: MVVM GUI — working, new combo boxes only

**New GenIce2 APIs to use (verified):**
- `safe_import('lattice', name).Lattice()` — dynamic lattice loading (c0te, c1te, c2te, ice1hte, sTprime, 16, 17)
- `safe_import('molecule', name).Molecule()` — dynamic water model + guest loading
- `parse_guest(guests, spec_string)` — mixed occupancy syntax (e.g., `'12=co2*0.6+me*0.4'`)
- `GenIce(lat, spot_guests={idx: 'name'})` — per-cage guest assignment for filled ices
- `sys.modules['genice2.molecules.{name}']` injection — custom guest registration

- Empty hydrate (Ice XVI / Ice XVII) — mechanical properties studies

**Should have (differentiators):**
- Custom guest .gro+.itp upload with auto-cage fitting — user-defined molecules in any cage
- GUI-driven mixed occupancy sliders — huge UX win over CLI `-g` flags
- Binary clathrate presets (one-click "CH₄+CO₂ sI") — common workflow shortcut

### Architecture Approach

v4.7 extends the existing dual-path architecture (CLI + GUI → shared core) with **additive changes to existing components**. The central architectural change is shifting from **pattern-matching-based molecule identification** to **metadata-driven identification**: instead of `_build_molecule_index()` guessing molecules from atom/residue names, HydrateConfig carries guest metadata (name, atom labels, atom count, ITP path) through the generation→export pipeline. This unblocks custom guests AND fixes the P3 export issue where `detect_guest_type_from_atoms()` returns `None` for unknown molecules.

**Major components and their v4.7 changes:**
1. **`types.py`** — Add GUEST_MOLECULES entries (co2, h2, et), HYDRATE_LATTICES entries (filled ices + exotic ices), WATER_MODELS dict, and new HydrateConfig fields (water_model, depol_mode, guest_metadata)
2. **`hydrate_generator.py`** — Refactor `_build_molecule_index()` for metadata-driven recognition; add water_model and depol parameters to `_run_via_api()`; add `spot_guests` support for filled ices; add `inject_custom_guest()` for sys.modules registration
3. **`gromacs_writer.py`** — Add CO₂/H₂/ethane atomtype blocks and ITP includes; fix P3 by using `mol_type` from molecule_index instead of re-detecting; add GRO residue name validation; apply `comment_out_atomtypes_in_itp()` to ALL guest ITPs during export
4. **`moleculetype_registry.py`** — Add CO2_H, H2_H, ET_H to RESERVED_NAMES; minimal changes (already handles arbitrary names)

### Critical Pitfalls

1. **GRO residue name overflow** — `:<5s` does NOT truncate, it overflows into atom name column. Add `validate_residue_name()` at EVERY writer entry point (6+ functions). For custom guests, base name must be ≤3 chars (to fit `_H` suffix within 5-char GRO limit).
2. **`_build_molecule_index()` breaks for non-TIP4P water** — TIP3P/SPC/E use residue name "SOL" (not "ICE") and atoms O/H/H (not OW/HW1/HW2/MW). Refactor to use config-provided water model metadata before exposing water model selector.
3. **Thread safety gap for custom guest injection** — `sys.modules` registration must happen in the main thread before `HydrateWorker.start()`, not inside `run()`. Extend `_genice_lock` scope or use per-run UUID-prefixed names.
4. **detect_guest_type_from_atoms returns None for custom guests** — Pass `mol_type` from `MoleculeIndex` directly to GROMACS writer instead of re-detecting from atom names. This is the P3 fix.
5. **GUEST_MOLECULES dict rejects custom types** — `HydrateConfig.__post_init__` validates `guest_type not in GUEST_MOLECULES`. Add "custom" entry or separate validation path.
6. **ITP A/B format misinterpretation** — `parse_itp_atomtypes()` assumes sigma/epsilon format. A/B format produces wrong LJ parameters by orders of magnitude. Detect and REJECT (do not auto-convert).
7. **Filled ices use `spot_guests`, not `-g`** — Two different guest placement APIs. `_run_via_api()` must detect lattice type and use the correct mechanism.

## Open Questions Requiring User Input

1. **ITP parameter provenance** — Which parameterization source for CO₂ in hydrates? Zhang-Sprik is common but has variants. Which source for H₂? Which for ethane cross-parameters with TIP4P-ICE? This affects Phase 1 ITP file creation.

2. **Custom guest thread safety strategy** — Three options: (a) UUID-prefixed names (`qice_<uuid>_<name>`), (b) QMutex lock around registration+generation, (c) enforce serial generation. Option (a) is cleanest but pollutes sys.modules. Option (c) is simplest but limits GUI responsiveness. Which approach?

3. **sT' guest placement approach** — sTprime has NO cage definitions in GenIce2. Options: (a) water-only generation in v4.7, defer guests to v5+; (b) build custom lattice plugin with cage positions; (c) post-hoc void detection + manual placement. Option (a) is recommended but needs user buy-in.

4. **Water model scope for v4.7** — Start with 4 models (TIP4P-ICE, TIP4P/2005, TIP3P, SPC/E) or include TIP5P? TIP5P is common for structure reproduction but needs its own ITP file and 5-site molecule_index handling.

5. **C1 hydrogen ordering** — c1te is H-ordered (`fixed` attribute has 72 pairs). Default `asis=False` allows GenIce2 to rearrange H-bonds, potentially violating C1 ordering. Should v4.7 auto-set `asis=True` for c1te, or just warn the user?

6. **Custom guest + v5.5 molecule library overlap** — v4.7 (custom guest from user files) and v5.5 (pre-built molecule library) both place guest molecules in hydrate cages. Should v5.5's curated molecules feed into v4.7's bridge, or use a separate code path?

7. **Mixed occupancy UI complexity** — How many guest types per cage type? GenIce2 supports arbitrary mixing (`12=co2*0.3+me*0.3+h2*0.4`), but the UI gets unwieldy beyond 2 guests. Limit to 2 per cage type?

8. **Ion ITP files** — `na.itp` and `cl.itp` are currently missing. Are these needed for v4.7, or can they wait until the ion insertion pipeline is fully tested? They're needed for any export path that includes ions.

- **ITP parameter provenance:** CO₂, H₂, and ethane ITP files need validated LJ parameters. Zhang-Sprik (CO₂) and Vehkamäki (H₂) are referenced in research but specific parameter values need extraction from literature and cross-validation with existing QuickIce force field conventions.
- **Thread safety design:** The sys.modules injection approach works in single-threaded testing but has NOT been tested in QThread context. Design the thread-safety strategy before Phase 6 implementation.
- **Cage diameter table for validation:** Cage cavity diameters used in size validation are from cross-verified training data but not primary-sourced from crystallographic tables. Need to verify against Jeffrey (1984) or Sloan & Koh (2007).
- **sT' void structure:** If sT' guest placement is ever needed, the channel void positions must be extracted from the water framework geometry. No existing tool does this — would require novel Voronoi analysis or manual crystallographic reference.
- **Custom guest rotation:** GenIce2 applies random rotations to guest molecules via `Molecule.rotation()`. Custom guest modules must implement this method. The default implementation (identity rotation) produces physically unrealistic non-rotated guests. Need to implement rotation using `scipy.spatial.transform.Rotation`.
