# Architecture Patterns: QuickIce v4.7 Extended Hydrate Generation

**Domain:** Scientific MD simulation structure preparation
**Researched:** 2026-06-27
**Mode:** Ecosystem (architecture dimension)
**Confidence:** HIGH (verified by running actual Python code against conda env)

## Executive Summary

v4.7 extends QuickIce's hydrate generation from 2 guest types (CH4, THF) on 3 lattice types (sI, sII, sH) to support new guests (CO2, H2, custom molecules), new lattice types (sIII, sIV, sV, sVII), a water model selector, and filled ice generation. The existing MVVM architecture is **well-suited** for these extensions — most v4.7 features are additive changes to existing components rather than architectural redesigns.

The most critical architectural verification is the **sys.modules injection** pattern for custom guests: I verified this works end-to-end by creating a custom Xenon molecule module, injecting it into `sys.modules['genice2.molecules.xenon']`, and generating a complete sI hydrate with 8 xenon guests. GenIce2's `safe_import('molecule', 'xenon')` correctly finds the injected module.

The main architectural gaps are: (1) `_build_molecule_index()` has no way to recognize custom guest atom patterns, (2) `detect_guest_type_from_atoms()` returns `None` for custom guests causing "UNK" residue names in export, and (3) `HydrateConfig` hard-validates against the 2-entry `GUEST_MOLECULES` dict. All three are **local modifications**, not redesigns.

## Recommended Architecture

### Integration Strategy: Extend, Don't Replace

All v4.7 features integrate into the **existing dual-path architecture** (CLI + GUI → shared core). The key principle: **extend the data flow, don't fork it**.

```
                              ┌─────────────────┐
                              │  HydrateConfig   │ ← EXTEND: add guest_type="co2"|"h2"|"custom", water_model, lattice_type="sIII"...
                              └────────┬────────┘
                                       │
                              ┌────────▼────────┐
                              │ HydrateStructure │ ← EXTEND: add guest_metadata dict for custom guests
                              │    Generator     │
                              └────────┬────────┘
                                       │
                              ┌────────▼────────┐
                              │  HydrateResult   │ ← EXTEND: add guest_atom_patterns, water_model_name
                              │   (dataclass)    │
                              └────────┬────────┘
                                       │
                    ┌──────────────────┼──────────────────┐
                    │                  │                    │
           ┌────────▼───────┐ ┌───────▼──────┐ ┌────────▼────────┐
           │  VTK Renderer   │ │ GROMACS      │ │ Moleculetype   │
           │ (hydrate_       │ │ Writer       │ │ Registry       │
           │  renderer.py)   │ │ (gromacs_    │ │ (registry.py)  │
           │                 │ │  writer.py)  │ │                │
           └────────────────┘ └──────────────┘ └─────────────────┘
           EXTEND:             EXTEND:           EXTEND:
           - element map       - new guest ITPs  - register new guests
           - custom guest viz  - custom ITP path  - custom guest names
```

### Data Flow Changes

**Current flow:**
```
HydratePanel → HydrateConfig(lattice, guest, occupancy, supercell)
  → HydrateWorker → HydrateStructureGenerator.generate(config)
    → GenIce2 API → GRO string → parse → _build_molecule_index
  → HydrateResult → VTK renderer + GROMACS writer
```

**v4.7 extended flow:**
```
HydratePanel → HydrateConfig(lattice, guest, occupancy, supercell, water_model, custom_guest_spec)
  → HydrateWorker → HydrateStructureGenerator.generate(config)
    → if custom_guest: inject sys.modules['genice2.molecules.X'] with custom Molecule class
    → GenIce2 API with water_model → GRO string → parse → _build_molecule_index(guest_metadata=...)
  → HydrateResult(guest_metadata={mol_type: atom_count, ...}) → VTK renderer + GROMACS writer
```

Key change: **guest metadata flows from config through generation to export**, enabling the molecule_index builder and GROMACS writer to handle arbitrary guests.

### 2. Custom Guest Bridge (sys.modules Injection)

**Status: VERIFIED WORKING by running actual Python code**

This is the most novel v4.7 feature. I verified the complete end-to-end flow:

```python
# Test result: PASSED
# Step 1: Create custom molecule module
custom_mod = types.ModuleType('genice2.molecules.xenon')
class Molecule(genice2.molecules.Molecule):
    def __init__(self):
        super().__init__()
        self.sites_ = np.array([[0.0, 0.0, 0.0]])
        self.labels_ = ['Xe']
        self.name_ = 'XENON'
custom_mod.Molecule = Molecule
sys.modules['genice2.molecules.xenon'] = custom_mod

# Step 2: GenIce2 safe_import finds it
result = safe_import('molecule', 'xenon')  # PASSES

# Step 3: Generate hydrate with custom guest
# parse_guest(guests, '12=xenon*1.0') + '14=xenon*1.0'
# GenIce generates sI hydrate with 8 xenon guests → 192 atoms
# Output contains: "47XENON   Xe  185   0.600   0.600   0.600"
```

**Architecture for custom guest injection:**

```
User provides: .itp file (topology) + .gro file (reference geometry)
                    │
                    ▼
          ┌─────────────────────┐
          │ CustomGuestBuilder   │  ← NEW component
          │                     │
          │ Input: ITP file     │
          │ Process:            │
          │  1. Parse ITP →     │
          │     atom_types,     │
          │     atom_names,     │
          │     charges, bonds  │
          │  2. Read GRO →      │
          │     positions       │
          │  3. Create GenIce2  │
          │     Molecule class   │
          │     dynamically      │
          │  4. Inject into     │
          │     sys.modules      │
          │                     │
          │ Output:             │
          │  - guest_name str   │
          │  - atom_labels list │
          │  - n_atoms int      │
          │  - ITP path for     │
          │    GROMACS export   │
          └─────────────────────┘
```

**Key constraint:** The `genice2.molecules.Molecule` base class requires:
- `self.sites_`: np.ndarray of shape (N, 3) — relative atom positions
- `self.labels_`: list of str — atom names (e.g., ['C', 'H', 'H', 'H', 'H'])
- `self.name_`: str — molecule name for GRO residue naming

**Integration points requiring modification:**

| File | Change | Complexity |
|------|--------|------------|
| `types.py` | Add `custom_guest_name`, `custom_itp_path`, `custom_gro_path` fields to HydrateConfig | Low |
| `hydrate_generator.py` | Add `inject_custom_guest()` method; modify `_run_via_api()` to use injected module | Medium |
| `hydrate_generator.py` | **CRITICAL**: Modify `_build_molecule_index()` to use guest metadata from config instead of pattern matching | High |
| `gromacs_writer.py` | Handle custom guest residue name from GenIce2 `name_` attribute; include custom ITP; write custom atomtypes | Medium |
| `hydrate_panel.py` | Add custom guest upload UI (similar to Tab 3 Custom panel) | Medium |

**The _build_molecule_index problem:**

Currently, `_build_molecule_index()` uses **pattern matching** on atom names and residue names to identify molecules. This fails for custom guests:

```
# Test result for custom xenon:
# Unknown atom type 'Xe' at index 184 — tracked as single-atom 'unknown' molecule
# mol_type = "unknown" instead of "xenon"
```

**Solution: Pass guest metadata from config to molecule_index builder.**

```python
# Proposed: HydrateConfig gets guest_metadata field
@dataclass
class HydrateConfig:
    # ... existing fields ...
    guest_metadata: dict[str, Any] | None = None
    # guest_metadata = {
    #   "custom_guest_name": "xenon",    # mol_type name
    #   "custom_guest_atoms": ["Xe"],    # expected atom labels
    #   "custom_guest_atoms_per_mol": 1, # atoms per guest molecule
    #   "custom_itp_path": Path(...),    # for GROMACS export
    # }
```

In `_build_molecule_index()`, add a branch at the top that uses the metadata:
```python
if config.guest_metadata and residue_name == config.guest_metadata["custom_guest_name"].upper():
    n_atoms = config.guest_metadata["custom_guest_atoms_per_mol"]
    molecule_index.append(MoleculeIndex(i, n_atoms, config.guest_metadata["custom_guest_name"]))
    i += n_atoms
    continue
```

| tip3p | SOL | O, H, H | 3 | 3-site SPC-like |

| Component | Impact | Change Required |
|-----------|--------|----------------|

**Recommended initial scope:** Support tip4p (default), tip3p, spce only. 5-site+ requires ITP files and is rarely used.

### 4. Filled Ices

**Status: Needs a new code path, NOT just a config change**

**Key finding from running code:** Ice lattices (Ih, Ic, etc.) do NOT have named cages. The `parse_guest()` mechanism only works for clathrate hydrate lattices.

```python
# Test result: Ice Ih with guests via parse_guest()
# parse_guest(guests, '0=ch4*1.0') → 64 atoms = 16 water molecules × 4 atoms
# No CH4 guests actually placed. "Cage type 0" doesn't exist in ice Ih.
```

Filled ices (e.g., ice Ih with interstitial H2, He@ice) require **post-generation insertion** of guest molecules into identified interstitial voids. This is architecturally different from hydrate cage occupancy.

**Two approaches:**

**Approach A: Reuse existing solute inserter** (Recommended)
- Generate ice structure first (existing path)
- Identify interstitial void positions using Voronoi analysis or distance-based detection
- Insert guests into voids using existing `SoluteInserter` with modifications
- Pros: Minimal new code; reuses tested overlap detection
- Cons: Void detection is new code; placement may not match experimental crystallography

**Approach B: New GenIce2 lattice modules**
- Create custom GenIce2 lattice modules that encode filled ice positions (like `sI` encodes cage positions)
- This would require reverse-engineering filled ice crystal structures
- Pros: Exact crystallographic positions
- Cons: High effort per lattice type; doesn't scale

**Recommended: Approach A** with a new `FilledIceInserter` component that:
1. Takes an ice Candidate and guest molecule info
2. Finds interstitial voids (center of largest gap between water molecules)
3. Inserts guests avoiding overlap (reusing solute_inserter.py patterns)
4. Returns a structure with water + guest molecule_index

### 5. P3 Export Fix (Per-Molecule mol_type MW Detection)

**Status: Local fix in gromacs_writer.py**

**Current problem:** For custom/unknown guests, `detect_guest_type_from_atoms()` returns `None`, causing:
- Residue name falls back to "UNK" / "UNK_H"
- No ITP include for the guest
- No atomtypes written for the guest
- `[ molecules ]` section uses "UNK" name

**Solution:** Pass the guest type name from `HydrateResult.molecule_index` (where `mol_type` is known) to the GROMACS writer, instead of re-detecting from atom names.

```python
# CURRENT: gromacs_writer.py re-detects guest type from atom names
guest_type = detect_guest_type_from_atoms(guest_atom_names)  # Returns None for custom

# PROPOSED: Use mol_type from molecule_index directly
# In write_multi_molecule_top_file() and write_ion_top_file():
for mol in molecule_index:
    if mol.mol_type not in ["water", "ice", "na", "cl"]:
        guest_type = mol.mol_type  # Use directly, don't re-detect
```

This requires:
1. Ensure `_build_molecule_index()` correctly tags custom guests (not "unknown")
2. Pass `custom_itp_path` through the data flow (HydrateConfig → HydrateResult → InterfaceStructure → IonStructure → writer)
3. Use the custom ITP path for `#include` directives

### 6. ITP Transformation Pipeline

**Status: Existing infrastructure is sufficient**

The ITP handling architecture for v4.7 custom guests follows the **same pattern** as v4.5 custom molecules in Tab 3:

```
User provides custom_guest.itp
        │
        ▼
┌──────────────────────┐
│ parse_itp_file()      │  ← EXISTING (itp_parser.py)
│ Returns:              │
│  - molecule_name      │
│  - atom_count         │
│  - atom_types         │
│  - atom_names         │
│  - charges            │
│  - has_atomtypes      │
└───────────┬──────────┘
            │
            ▼
┌──────────────────────┐
│ parse_itp_atomtypes() │  ← EXISTING (gromacs_writer.py)
│ Returns: atomtypes    │
│ as string tuples      │
└───────────┬──────────┘
            │
            ▼
┌────────────────────────────┐
│ comment_out_atomtypes_     │  ← EXISTING (gromacs_writer.py)
│ in_itp()                   │
│ Removes [atomtypes] from   │
│ ITP → types go in .top     │
└───────────┬────────────────┘
            │
            ▼
┌────────────────────────────┐
│ write_*_top_file()         │
│ Writes:                    │
│ 1. [atomtypes] with custom │
│    types (dedup + conflict)│
│ 2. #include "custom.itp"   │
│ 3. [molecules] with name  │
└────────────────────────────┘
```

**No new ITP handling code needed.** The existing pipeline handles:
- Parsing atomtypes from custom ITP files (`parse_itp_atomtypes`)
- Commenting out atomtypes in ITP before including in .top (`comment_out_atomtypes_in_itp`)
- Deduplication of shared atomtypes (`_write_atomtypes_block`, `_check_custom_atomtype_conflict`)
- Conflict detection for same-name atomtypes with different LJ params (`_lj_params_match`)

**One gap:** For hydrate custom guests, we need a `_hydrate` variant ITP file (with `_H` moleculetype name). This follows the existing pattern:
- `ch4.itp` → moleculetype `ch4`
- `ch4_hydrate.itp` → moleculetype `CH4_H`
- Custom: `user_mol.itp` → moleculetype `MOL`
- Custom hydrate: `user_mol_hydrate.itp` → moleculetype `MOL_H`

We can generate the `_hydrate` variant **programmatically** by:
1. Reading the user's ITP file
2. Renaming the `[moleculetype]` name to add `_H` suffix
3. Writing the modified file alongside the original

## Patterns to Follow

### Pattern 1: Config-Driven Guest Metadata Flow

**What:** Pass guest metadata from HydrateConfig through generation to export, instead of re-detecting from atom names.

**When:** Any time a custom or non-standard guest is used.

**Example:**
```python
@dataclass
class HydrateConfig:
    # ... existing fields ...
    water_model: str = "tip4p"
    guest_metadata: dict[str, Any] | None = None
    # guest_metadata keys:
    #   "guest_name": str          — GenIce2 molecule name (e.g., "xenon")
    #   "guest_res_name": str      — GRO residue name (e.g., "XENON")
    #   "guest_atom_labels": list  — expected atom names (e.g., ["Xe"])
    #   "guest_atoms_per_mol": int — atom count per molecule (e.g., 1)
    #   "guest_itp_path": str      — path to ITP file for GROMACS export
```

### Pattern 2: sys.modules Injection for Custom Guests

**What:** Create a dynamic GenIce2-compatible Molecule class from user ITP/GRO, inject it into sys.modules, then call safe_import.

**When:** Custom guest generation.

**Example:**
```python
def inject_custom_guest(guest_name: str, atom_positions: np.ndarray, 
                        atom_labels: list[str], mol_name: str) -> None:
    """Inject a custom molecule into GenIce2's module namespace."""
    import types, sys, genice2.molecules
    
    mod = types.ModuleType(f'genice2.molecules.{guest_name}')
    
    class Molecule(genice2.molecules.Molecule):
        def __init__(self):
            super().__init__()
            self.sites_ = atom_positions.copy()
            self.labels_ = list(atom_labels)
            self.name_ = mol_name
    
    mod.Molecule = Molecule
    sys.modules[f'genice2.molecules.{guest_name}'] = mod
```

### Pattern 3: ITP Name Suffix Convention

**What:** Hydrate guests use `_H` suffix, liquid solutes use `_L` suffix in GROMACS moleculetype names.

**When:** Any GROMACS export with guests/solutes.

**Example:**
```python
# Registry handles this automatically:
registry.register_hydrate_guest('XENON')  # → 'XENON_H'
registry.register_liquid_solute('XENON')   # → 'XENON_L'
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: Re-Detecting Guest Type from Atom Names at Export

**What:** `detect_guest_type_from_atoms()` inspects atom names to guess the guest type. This fails for custom guests.

**Why bad:** Custom guests have unknown atom patterns → returns None → "UNK" residue name → broken GROMACS topology.

**Instead:** Use `mol_type` from `molecule_index` directly. The generator already knows what guest was requested — pass that information forward.

### Anti-Pattern 2: Hardcoding Water Model in Generator

**What:** `safe_import('molecule', 'tip4p').Molecule()` is hardcoded at line 164 of hydrate_generator.py.

**Why bad:** Prevents TIP3P, SPC/E, TIP5P models.

**Instead:** Use `config.water_model` parameter: `safe_import('molecule', config.water_model).Molecule()`

### Anti-Pattern 3: Pattern-Matching-Only Molecule Index

**What:** `_build_molecule_index()` uses if/elif chains on atom names and residue names to identify molecules. Custom guests fall to "unknown" handler.

**Why bad:** Doesn't scale to arbitrary guest molecules. Every new guest requires a new pattern branch.

**Instead:** When guest metadata is available from config, use it to identify and group guest atoms directly (by residue name + known atom count), falling back to pattern matching only when metadata is absent.

## Scalability Considerations

| Concern | At 2 guest types | At 10+ guest types | At custom guests |
|---------|------------------|--------------------|-------------------|
| GUEST_MOLECULES dict | Small, manageable | Large but still dict | Needs "custom" entry |
| `_build_molecule_index` | if/elif chain works | Chain gets unwieldy | Must use metadata |
| ITP files | 4 files (ch4/thf × hydrate/liquid) | 20+ files | User-provided |
| atomtypes dedup | Simple | Need full dedup system | Need conflict detection (existing) |
| GUI guest combo | 2 items | Dropdown with search | "Custom..." option |

## Phase-Specific Research Flags

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|---------------|------------|
| Custom guest injection | sys.modules pollution (old injections linger) | Clean up sys.modules after generation, or namespace with UUID |
| Custom guest ITP | User ITP may have [atomtypes] that conflict with existing | `_check_custom_atomtype_conflict()` already handles this |
| Water model selector | Non-TIP4P models need different ITP files (tip3p.itp, spce.itp) | Must create these ITP files; can't reuse tip4p-ice.itp |
| Water model 3-site | _build_molecule_index needs 3-atom water pattern (O, H, H not OW, HW1, HW2, MW) | Add 3-site branch to _build_molecule_index |
| Filled ice void detection | Voronoi-based void finding is complex | Start with simple distance-based approach: find centers of gaps > threshold |
| sH cage type 12_1 | GenIce2 sH uses "12_1" cage type notation, different from "12" | Parse cage types generically, not hardcoded "12"/"14"/"16" |
| New lattice occupancy | sIII has 3 cage types (12, 14, 15) — need 3 occupancy controls | Add per-cage-type occupancy UI or use "small/medium/large" pattern |
| CO2 guest in sI | CO2 is larger than CH4 — may not fit in small cages | GUEST_MOLECULES entry should list cage compatibility |
