# Domain Pitfalls: QuickIce v4.7 Extended Hydrate Generation

**Domain:** Molecular simulation — custom guest hydrate structure generation + GROMACS export
**Researched:** 2026-06-27
**Codebase version:** v4.5 (1,032 tests passing)
**Overall confidence:** HIGH (all pitfalls verified against current source code)

---

## Critical Pitfalls

Mistakes that cause silent data corruption, GROMACS rejection, or runtime crashes.

### Pitfall 1: GRO Residue Name Overflow — NOT Silent Truncation

**What goes wrong:** When a residue name exceeds 5 characters, the GRO file is **silently corrupted** — not truncated. Python's `:<5s` format specifier pads shorter strings to 5 chars but **does NOT truncate** longer strings. This overflows into the atom name column, breaking the fixed-width GRO format.

**Why it happens:** GRO format is fixed-width columns (residue name = cols 5–10, atom name = cols 10–15). Every writer in `gromacs_writer.py` uses `{res_name:<5s}` formatting, which:
- Pads "CH4" → "CH4  " (correct, 5 chars)
- Leaves "CUSTOM_NAME_H" → "CUSTOM_NAME_H" (13 chars, corrupts atom name column)

**Consequences:** GROMACS `grompp` reads wrong atom names, wrong atom types, or crashes with "Invalid line in coordinate file." The corruption is invisible in the .gro file — it looks like normal text, but column alignment is broken.

**Current code evidence (v4.5):**
```python
# gromacs_writer.py line 1391 (write_multi_molecule_gro_file)
f"{res_num:5d}{res_name:<5s}"  # Does NOT truncate >5 char names

# Same pattern in:
# - write_interface_gro_file line 1036
# - write_ion_gro_file lines 1819, 1849, 1882
# - write_custom_molecule_gro_file line 2362
# - write_solute_gro_file line 2856
```

**Prevention:**
1. Add `validate_residue_name()` function that rejects names >5 chars BEFORE writing
2. Apply validation at EVERY writer entry point (6+ functions)
3. For custom guests, compute max base name length as `5 - len(suffix)` (e.g., `_H` → base ≤3 chars)

**Detection:** Run `grompp` on exported files as integration test. Or check `len(res_name) > 5` at write time.

**Phase:** Phase 1 — must be fixed BEFORE any custom guest ITP is accepted

### Pitfall 2: GenIce2 `safe_import` Cannot Find Custom Guest Molecules

**What goes wrong:** GenIce2's `safe_import('molecule', guest_name)` searches for plugins in three places:
1. Local directory (`molecules.{name}` module)
2. System path (`genice2.molecules.{name}`)
3. Entry points (`genice2_molecule` group via `importlib.metadata`)

User-provided custom guest molecules have none of these. `safe_import` raises `AssertionError: Nonexistent or failed to load the module`.

**Why it happens:** GenIce2's plugin architecture requires pre-installed packages with proper entry points. QuickIce cannot install pip packages at runtime (conda-only constraint in AGENTS.md).

**Consequences:** Custom guests are impossible to use with GenIce2's `generate_ice()` API. The generation call `ice.generate_ice(formatter=formatter, water=water, guests=guests)` will fail when GenIce2 tries to load the guest molecule plugin.

**Current code evidence (v4.5):**
```python
# hydrate_generator.py line 151
lattice = safe_import('lattice', lattice_name).Lattice()
# line 164
water = safe_import('molecule', 'tip4p').Molecule()
# guests dict at lines 171-205 only uses GenIce2-recognized names (ch4, thf)
```

**Available built-in GenIce2 molecule plugins:** ch4, thf, co2, H2, et (ethane), me (united-atom methane), uathf, uathf6, and water models (tip3p, tip4p, tip5p, spce, etc.)

**Prevention:**
1. **Option A (Recommended):** Pre-register custom molecules in `sys.modules` before calling GenIce2. Create a synthetic `genice2.molecules.{name}` module with a `Molecule` class implementing `rotation()`, `atoms`, `name`. This is the approach GenIce2's own "local folder" mechanism uses.
2. **Option B:** Only support GenIce2 built-in guests for cage occupancy. For custom guests, place them AFTER generation using QuickIce's existing `CustomMoleculeInserter` (which doesn't use GenIce2 at all).
3. **Option C (Rejected):** Install genice2-{name} packages at runtime — violates AGENTS.md conda-only constraint.

**Detection:** `safe_import('molecule', guest_name)` raises `AssertionError` if guest not found.

**Phase:** Phase 1 — custom guest registration mechanism is foundational

### Pitfall 3: Custom Guest ITP A/B Format → Sigma/Epsilon Misinterpretation

**What goes wrong:** GROMACS `[ atomtypes ]` supports two formats:
- **Format A (sigma/epsilon):** Used by AMBER/GAFF2/OPLS-AA with comb-rule=2. Columns: name, bond_type, at.num, mass, charge, ptype, sigma, epsilon
- **Format B (A/B):** Used by some OPLS-AA and older force fields. Columns: name, bond_type, at.num, mass, charge, ptype, A, B (where A=4εσ⁶, B=4εσ¹²)

Current `parse_itp_atomtypes()` assumes sigma/epsilon format and has NO detection for A/B format. If a user provides an A/B format ITP, the columns are read as sigma/epsilon, producing completely wrong LJ parameters. This causes guests to fly out of cages during MD (energies off by orders of magnitude).

**Current code evidence (v4.5):**
```python
# gromacs_writer.py lines 445-493 (parse_itp_atomtypes)
# Normalizes 7-col → 8-col by inserting bond_type
# No format detection — assumes sigma/epsilon columns
parts = stripped.split()
if len(parts) >= 7:
    if len(parts) == 7:
        parts.insert(1, parts[0])  # Use name as bond_type
    atomtypes.append(tuple(parts))
```

**Prevention:**
1. Detect A/B format: If column 7 (epsilon or B) has values >1000, it's likely A/B format (B values are typically 10⁶–10¹² kJ·nm¹²/mol, while epsilon values are 0.01–5 kJ/mol)
2. **Reject A/B format with clear error message** — do NOT auto-convert (risk of rounding errors)
3. Tell user to convert: "Your ITP uses A/B LJ format (comb-rule=1 compatible). QuickIce requires sigma/epsilon format (comb-rule=2, AMBER/GAFF2 convention). Convert using: sigma = (B/A)^(1/6), epsilon = A²/(4B)"

**Detection:** Check magnitude of values in columns 7-8 of `[ atomtypes ]`. If both >100, likely A/B format.

**Phase:** Phase 1 — validation must precede any custom guest ITP acceptance

### Pitfall 4: sys.modules Injection Thread Safety in QThread

**What goes wrong:** If v4.7 registers custom guest modules in `sys.modules` (Pitfall 2, Option A), and this happens inside `HydrateWorker.run()` (which runs in a QThread), there's a race condition with the main thread's import system. Python's `importlib.import_module()` and `sys.modules` modifications are NOT thread-safe — two threads can import the same module simultaneously, leading to partially-initialized module objects.

**Why it happens:** The current `_genice_lock` in `hydrate_generator.py` (line 31) only protects the initial GenIce2 lazy-load import. Custom guest registration would happen AFTER this initial import, outside the lock scope.

**Current code evidence (v4.5):**
```python
# hydrate_generator.py line 31
_genice_lock = threading.Lock()

# Lines 50-79: _ensure_genice_import uses _genice_lock
# BUT: Custom guest registration (future v4.7) would happen
# in _run_via_api() line 148+, AFTER _ensure_genice_import()
# which is outside the lock scope
```

**Prevention:**
1. Extend `_genice_lock` scope to cover ALL `sys.modules` modifications
2. Or: Register custom guest modules in the MAIN THREAD before `HydrateWorker.start()`
3. Or: Use `importlib.util.find_spec()` + manual module creation (avoids `import_module` race)
4. Simplest: Create custom guest Molecule class in main thread, pass it to worker via config

**Detection:** Intermittent `ImportError` or `AttributeError` during parallel generation. Extremely hard to reproduce deterministically.

**Phase:** Phase 1 — thread safety must be designed into the registration mechanism from the start

### Pitfall 5: `detect_guest_type_from_atoms` Returns None for Custom Guests

**What goes wrong:** The `detect_guest_type_from_atoms()` function (gromacs_writer.py line 1127-1178) only recognizes: ch4 (C+H), thf (O+CA/CB), co2 (C+O, no H), h2 (H only), Me (united-atom). For any custom guest, it returns `None`, causing fallback to "UNK" residue name in GRO files and "GUE" in .top files.

**Why it happens:** The detection is based on atom composition heuristics that only match known molecule types. There's no mechanism to pass guest type information from the generation step to the export step.

**Current code evidence (v4.5):**
```python
# gromacs_writer.py line 1127-1178
def detect_guest_type_from_atoms(atom_names: list[str]) -> str | None:
    # Only detects ch4, thf, co2, h2, Me
    ...
    return None  # Unknown guest type

# Line 1036: When None, falls back to guest_res_name from ITP
# But for custom guests, no ITP file path is carried in the data structures
```

**Prevention:**
1. **Option A (Recommended):** Add `guest_type: str` to `MoleculeIndex` so each molecule carries its type identity. Set this during generation, not detected during export.
2. **Option B:** Add a `guest_type_map: dict[str, str]` to `InterfaceStructure` and `IonStructure` that maps residue names to guest types.
3. **Option C:** Extend `detect_guest_type_from_atoms` to accept a known-type hint parameter.

**Detection:** GRO files with "UNK" residue names for custom guests. `grompp` will fail because "UNK" doesn't match any moleculetype in .top.

**Phase:** Phase 1 — must be resolved before custom guests reach the export pipeline

## Moderate Pitfalls

Mistakes that cause delays, incorrect topologies, or limited functionality.

### Pitfall 6: GUEST_MOLECULES Dict Rejects Custom Guest Types

**What goes wrong:** `HydrateConfig.__post_init__()` validates `guest_type not in GUEST_MOLECULES` (types.py line 368). The `GUEST_MOLECULES` dict (line 117-132) only contains "ch4" and "thf". Any custom guest type is rejected at configuration time with `ValueError: Unknown guest type`.

**Current code evidence (v4.5):**
```python
# types.py line 117-132
GUEST_MOLECULES: dict[str, dict[str, Any]] = {
    "ch4": {...},
    "thf": {...},
}

# types.py line 368
if self.guest_type not in GUEST_MOLECULES:
    raise ValueError(f"Unknown guest type: {self.guest_type}")
```

**Prevention:** Extend `GUEST_MOLECULES` to support a "custom" entry, or add a separate validation path for custom guest types. Custom guests need: name, atom count, .itp path, .gro path, but not all fields in `GUEST_MOLECULES` are applicable.

**Phase:** Phase 1 — gating validation must be updated

### Pitfall 7: MoleculetypeRegistry Creates New Instance Per Export

**What goes wrong:** `HydrateGROMACSExporter.export_hydrate()` creates a fresh `MoleculetypeRegistry()` per call (hydrate_export.py line 146). This means registrations from the generation step (where guest type was known) are NOT carried forward to the export step. The export step must re-register everything from scratch, creating a fragile information gap.

**Current code evidence (v4.5):**
```python
# hydrate_export.py line 146-151
registry = MoleculetypeRegistry()  # Fresh instance per export
registry.register_hydrate_guest(guest_upper)  # Re-register
```

**Prevention:** Carry the registry from generation through to export. Store the registry in `HydrateStructure` (similar to how `SoluteStructure.registry` already works).

**Phase:** Phase 1 — consistent registry lifecycle

### Pitfall 8: Custom Guest ITP [atomtypes] Section Not Pre-Commented

**What goes wrong:** QuickIce requires all `[ atomtypes ]` to be defined in the main .top file, not in individual .itp files. Bundled ITPs have `[ atomtypes ]` pre-commented (e.g., ch4_hydrate.itp lines 5-8). User-provided custom guest ITPs will NOT have this pre-processing, causing duplicate atomtypes errors in GROMACS.

**Current code evidence (v4.5):**
```python
# gromacs_writer.py line 496-540 (comment_out_atomtypes_in_itp)
# This function EXISTS but is only used for custom molecule ITPs
# It is NOT applied to guest ITPs during export

# hydrate_export.py line 182-183:
shutil.copy(guest_itp_path, guest_dest_path)
# Direct copy — no transformation applied to guest ITPs
```

**Prevention:** Apply `comment_out_atomtypes_in_itp()` to ALL ITP files during export, including guest ITPs. Or: Validate that user-provided ITPs already have `[ atomtypes ]` commented out, and reject if not.

**Phase:** Phase 1 — export pipeline must handle unprocessed ITPs

### Pitfall 9: Moleculetype Name vs Residue Name Mismatch in Custom ITPs

**What goes wrong:** GROMACS requires the residue name in .gro file to match the residue name in the `[ atoms ]` section of the .itp file. For bundled ITPs, this is consistent (e.g., `ch4_hydrate.itp` has moleculetype `CH4_H`, residue `CH4_H`). But user-provided ITPs may have different moleculetype names and residue names (e.g., moleculetype `MOL` but residue `LIG`).

**Current code evidence (v4.5):**
```python
# ch4_hydrate.itp line 12: [ moleculetype ] CH4_H 3
# ch4_hydrate.itp line 16: residue name = CH4_H  ← consistent

# But a user-provided ITP might have:
# [ moleculetype ] METHANE 3
# [ atoms ] residue name = MET  ← different!
```

**Prevention:** After parsing the ITP, use the residue name from `[ atoms ]` section (not moleculetype name) as the GRO residue name. Current `parse_itp_residue_name()` already extracts this. Apply it consistently for custom guests.

**Phase:** Phase 1 — ITP residue name must drive GRO naming

### Pitfall 10: HydrateStructure.to_candidate() Loses Custom Guest Identity

**What goes wrong:** When a hydrate is converted to `Candidate` for interface generation, the `to_candidate()` method (types.py line 736-800) builds `MoleculeIndex` entries with `mol_type` from the original molecule_index. For unknown/custom guests, `mol_type` is "unknown" (hydrate_generator.py line 586), which breaks:
- `detect_guest_type_from_atoms()` → returns None
- `get_hydrate_guest_residue_name()` → returns "UNK_H"
- GRO/TOP export → wrong residue names, GROMACS rejection

**Current code evidence (v4.5):**
```python
# hydrate_generator.py line 580-587
# Unknown atom type → mol_type = "unknown"
molecule_index.append(MoleculeIndex(i, 1, "unknown"))

# types.py line 753-762 (to_candidate)
# Preserves mol_type from original molecule_index
# "unknown" mol_type propagates downstream
```

**Prevention:** When a custom guest is registered, store the guest type name in `MoleculeIndex.mol_type` (not "unknown"). This requires the generator to know the guest type before building molecule_index.

**Phase:** Phase 1 — guest identity must flow through the data pipeline

## Minor Pitfalls

Mistakes that cause annoyance but are fixable.

### Pitfall 11: GenIce2 Guest Rotation for Custom Molecules

**What goes wrong:** GenIce2 places guests at cage centers and applies random rotations using the guest molecule's `Molecule.rotation()` method. Built-in guests (ch4, thf) implement this correctly. Custom guest modules created for `sys.modules` injection (Pitfall 2, Option A) must also implement `rotation()`. If not, GenIce2 may use identity rotation (no random orientation), creating physically unrealistic non-rotated guests.

**Current code evidence:** GenIce2 internal — not in QuickIce code. Verified by testing:
```python
# GenIce2 molecule plugin contract:
class Molecule:
    def rotation(self, rot):  # Required method
        ...
    atoms = [...]  # Required attribute
    name = "..."   # Required attribute
```

**Prevention:** When creating synthetic molecule modules for `sys.modules`, implement `rotation()` using `scipy.spatial.transform.Rotation`. The built-in `ch4` module shows the pattern.

**Detection:** Custom guests all have identical orientation in the structure (visually suspicious).

**Phase:** Phase 2 — when custom guest placement is implemented

### Pitfall 12: Mixed Occupancy UI — Two Guests Per Cage

**What goes wrong:** GenIce2 supports mixed cage occupancy (e.g., `12=co2*0.6+me*0.4`), but QuickIce's `HydrateConfig` only supports a single `guest_type`. The current `_run_via_api` (hydrate_generator.py lines 171-205) builds a single-guest `guests` dict. Mixed occupancy requires parsing a different UI configuration and translating to GenIce2's multi-guest format.

**Current code evidence (v4.5):**
```python
# hydrate_generator.py line 173
guest_type = config.guest_type  # Single guest type
# Lines 178-183: Only one guest_name mapped
```

**Prevention:** Extend `HydrateConfig` with `guest_type_large` and `occupancy_large_guest2` fields, or use a list-of-guests structure. Translate to GenIce2's `parse_guest()` format in `_run_via_api`.

**Phase:** Phase 2 (per SYNTHESIS.md Phase 1 "free wins")

### Pitfall 13: No ITP Validation for Custom Guest Residue Name Length

**What goes wrong:** When a user provides a custom ITP file, `parse_itp_file()` extracts the moleculetype name but does NOT validate that the residue name in `[ atoms ]` is ≤5 characters. Combined with Pitfall 1 (no GRO name validation), a custom guest with a 6+ char residue name creates a GRO file that GROMACS cannot read.

**Current code evidence (v4.5):**
```python
# itp_parser.py lines 36-160 (parse_itp_file)
# Extracts molecule_name, atom_count, atom_types, atom_names, charges
# NO validation of residue name length
# has_atomtypes_section check exists but no name length check
```

**Prevention:** Add validation in `parse_itp_file()` or in the GUI file-picker step: reject ITPs with residue names >5 chars. Provide clear error message explaining GRO format limit.

**Phase:** Phase 1 — validation at ITP intake

### Pitfall 14: CO₂ and H₂ Guests Have No Bundled Hydrate ITPs

**What goes wrong:** GenIce2 supports CO₂ and H₂ as built-in guests (verified: `genice2.molecules.co2` and `genice2.molecules.H2`). QuickIce's `MOLECULE_TO_GROMACS` mapping (gromacs_writer.py line 211-220) already maps "co2" and "h2" to .itp files. But the actual ITP files `co2.itp` and `h2.itp` exist in `quickice/data/`, and `co2_hydrate.itp` and `h2_hydrate.itp` do NOT exist yet. The `get_hydrate_guest_residue_name()` function has fallback names ("CO2_H", "H2_H") but no actual ITP files to copy during export.

**Current code evidence (v4.5):**
```
# Existing ITP files in quickice/data/:
#   tip4p-ice.itp, ch4_hydrate.itp, thf_hydrate.itp, ch4.itp, thf.itp
#   ch4_liquid.itp, thf_liquid.itp
# MISSING:
#   co2_hydrate.itp, h2_hydrate.itp
#   co2_liquid.itp, h2_liquid.itp
```

**Prevention:** Create `co2_hydrate.itp` (Zhang-Sprik CO₂ parameters) and `h2_hydrate.itp` (Vehkamäki H₂ parameters). Follow the ch4_hydrate.itp template: moleculetype with `_H` suffix, atomtypes commented out, comb-rule=2 compatible.

**Phase:** Phase 2 (per SYNTHESIS.md Phase 1 "free wins")

## Recommended Phase-1 Validation Checklist

Before any custom guest enters the export pipeline, these MUST be validated:

1. **Residue name ≤5 chars** — Reject at ITP parsing time, before any GRO write
2. **atomtypes format is sigma/epsilon (not A/B)** — Check column magnitudes; reject A/B
3. **[ atomtypes ] section commented out** — If not, apply `comment_out_atomtypes_in_itp()` during export
4. **Residue name in [ atoms ] matches moleculetype name** — Warn if mismatch (GROMACS allows it but it's confusing)
5. **Custom guest module implements rotation()** — Required for GenIce2 `safe_import` contract
6. **Guest type stored in MoleculeIndex.mol_type** — Not "unknown", must carry identity downstream
7. **sys.modules registration in main thread** — Before `HydrateWorker.start()`, not inside `run()`
