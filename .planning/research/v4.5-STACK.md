## Executive Summary

**No new external libraries required.** The existing Python stack is sufficient for both solute insertion and custom molecule upload features. The key finding is that **no mature Python library exists for parsing GROMACS .itp files** - MDTraj and other MD libraries focus on coordinate files (.gro, .xtc, .pdb) but not topology files (.itp, .top). We must implement a simple .itp parser using standard Python file I/O and regex, which is straightforward given the well-defined GROMACS topology format.

The recommended approach:
1. Build custom .itp parser (50-80 lines) to extract molecule name, atom types, and atom count
2. Reuse existing `scipy.spatial.cKDTree` for overlap detection (proven in ion_inserter.py)
3. Extend existing `gromacs_writer.py` to handle custom molecules via `itp_files` parameter

### Why No External ITP Parsing Library?

**Research findings:**
- **MDTraj**: Reads trajectories and structures (.gro, .xtc, .pdb) but not .itp files (verified from source code analysis)
- **ParmEd**: Can read GROMACS topologies but heavy dependency (~50MB) for simple parsing task
- **OpenMM**: Can parse GROMACS files but overkill (requires C++ libraries)
- **GROMACS Python wrappers** (`gmx` package): Thin wrappers around GROMACS tools, not standalone parsers

**Recommendation:** Build custom parser. The GROMACS .itp format is:
- Well-documented (GROMACS 2026 manual, section "File formats")
- Line-based with clear section markers (`[ moleculetype ]`, `[ atoms ]`, etc.)
- Fixed-width columns in atom records
- 50-80 lines of Python is sufficient for our needs (extract molecule name, atom types, atom count)

## Implementation Details

### 1. ITP Parser Module

**Purpose:** Extract essential information from user-provided .itp files

**Required extraction:**
- Molecule name from `[ moleculetype ]` section
- Atom types from `[ atoms ]` section (column 2)
- Atom count (number of lines in `[ atoms ]` section)

**Format reference (from GROMACS 2026 manual):**
```
[ moleculetype ]
; name  nrexcl
ch4     3

[ atoms ]
; nr  type  resi  res  atom  cgnr  charge  mass
   1   c3     1   CH4    C      1   -0.466  12.01
   2   hc     1   CH4    H      2    0.116   1.008
```

**Implementation approach:**
```python
def parse_itp_file(filepath: Path) -> dict:
    """Extract molecule name, atom types, and count from .itp file."""
    with open(filepath) as f:
        content = f.read()
    
    # Extract molecule name from [ moleculetype ] section
    mol_name = extract_molecule_name(content)
    
    # Extract atom types and count from [ atoms ] section
    atom_types, atom_count = extract_atoms_section(content)
    
    return {
        'molecule_name': mol_name,
        'atom_types': atom_types,  # List of unique atom types
        'atom_count': atom_count,
    }
```

**Complexity:** LOW - GROMACS format is stable, well-documented, line-based

### 2. Molecule Validator

**Purpose:** Ensure uploaded .gro and .itp files are consistent

**Validation checks:**
1. `.gro` atom count matches `.itp` `[ atoms ]` section count
2. Atom names in `.gro` match expected patterns (optional, warns only)
3. No missing required sections in `.itp` (`[ moleculetype ]`, `[ atoms ]`)

**Implementation:**
```python
def validate_molecule_files(gro_path: Path, itp_path: Path) -> tuple[bool, str]:
    """Validate .gro and .itp file consistency."""
    # Parse .gro file
    positions, atom_names, cell = parse_gro_file(gro_path)
    gro_atom_count = len(positions)
    
    # Parse .itp file
    itp_data = parse_itp_file(itp_path)
    itp_atom_count = itp_data['atom_count']
    
    # Check consistency
    if gro_atom_count != itp_atom_count:
        return False, f"Atom count mismatch: .gro={gro_atom_count}, .itp={itp_atom_count}"
    
    return True, "Valid molecule files"
```

**Complexity:** LOW - Reuses existing `parse_gro_file()` from gro_parser.py

### 3. Solute Inserter (THF/CH₄ in Liquid)

**Purpose:** Insert THF or CH₄ molecules at specified concentration in liquid phase

**Key difference from ion insertion:**
- Ions: Single atoms, replace entire water molecules (4 atoms → 1 ion)
- Solutes: Multi-atom molecules, need to place entire molecule with rotation

**Algorithm (adapted from ion_inserter.py):**
1. Calculate number of solute molecules from concentration and liquid volume
2. Identify liquid region water molecules (from molecule_index)
3. For each solute molecule:
   - Select random water molecule position as COM target
   - Load solute structure from bundled .gro/.itp files
   - Apply random rotation to solute
   - Translate solute COM to target position
   - Check overlap with existing atoms using cKDTree
   - If overlap detected, try different water molecule or different rotation
4. Remove replaced water molecules
5. Update molecule_index with new solute entries

**Overlap detection:** Reuse `scipy.spatial.cKDTree` from ion_inserter.py (proven approach)

**VDW radii for overlap check:**
- Use conservative radii (largest atom in molecule) for quick reject
- Or use atom-specific radii for accurate check

**Complexity:** MEDIUM - Similar to ion insertion, but need molecule rotation/translation logic

### 4. Custom Molecule Inserter

**Purpose:** Place user-uploaded custom molecules with optional COM/rotation specification

**User workflow:**
1. Upload .gro + .itp files via file dialog
2. Validate files (atom count match, required sections)
3. Extract atom types from .itp for GROMACS export
4. User specifies:
   - Concentration-based random placement, OR
   - Specific COM coordinates + rotation angles
5. Place molecules with overlap checking

**Integration with GROMACS export:**
- Existing `write_multi_molecule_top_file()` has `itp_files` parameter for custom molecules
- Copy user .itp file to export directory
- Add atom types to `[ atomtypes ]` section in .top file

**Complexity:** MEDIUM - Combines file parsing, validation, placement logic

### Atom Type Extraction for GROMACS Export

**Challenge:** User-provided .itp files reference atom types that must be defined in main .top file

**Two approaches:**

**Approach A: Require user .itp to include [ atomtypes ] section** (RECOMMENDED)
- User provides complete .itp with `[ atomtypes ]` section
- QuickIce copies this section to main .top file
- Simpler, more flexible, user has full control
- Downside: User must manage atom types

**Approach B: QuickIce extracts atom types and adds to .top**
- Parse `[ atoms ]` section to get atom type names
- Look up VDW parameters from force field database (GAFF2, etc.)
- Problem: Requires comprehensive force field database
- Problem: Cannot know VDW parameters for arbitrary atom types
- Not recommended: Too complex, requires force field expertise

**Recommendation:** Use Approach A - require user to provide complete .itp with `[ atomtypes ]` section

**Note:** Existing bundled CH4/THF .itp files already have `[ atomtypes ]` commented out because they're defined in main .top. For custom molecules, we should require users to provide the atom types in their .itp.

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| ParmEd library | Heavy dependency (~50MB), overkill for simple parsing | Custom 80-line Python parser |
| GROMACS command-line tools | Requires GROMACS installation, breaks portability | Pure Python parsing |
| MDTraj for .itp parsing | MDTraj doesn't support .itp files (only .gro, .pdb, trajectories) | Custom parser |
| ACD/Labs, ChemDraw libraries | Commercial, requires license, overkill | User provides .gro + .itp files |
| Automatic atom type inference | Cannot reliably infer VDW parameters from atom names | Require user to provide complete .itp |
