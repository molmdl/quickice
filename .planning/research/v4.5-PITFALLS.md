## Executive Summary

v4.5 introduces **multi-atom molecule insertion** into the liquid phase, which fundamentally differs from v4.0's single-atom ion insertion. The key architectural changes required are:

1. **Molecule rotation during placement** — Ions have no orientation; multi-atom molecules do
2. **Overlap detection complexity** — Must check ALL atoms, not just center-of-mass
3. **Residue naming conventions** — Must distinguish hydrate guests (CH4_HYD) from liquid solutes (CH4_LIQ)
4. **Custom molecule validation** — User-provided .gro/.itp files require extensive validation
5. **Tab reordering integration** — Ion moves from Tab 4 to Tab 6, changing cross-tab data flow

**Critical finding:** The existing `ion_inserter.py` assumes 1-atom molecules and cannot be trivially extended. A new `solute_inserter.py` is needed with rotation matrices, all-atom overlap checking, and water-molecule-replacement logic.

## Critical Pitfalls

Mistakes that cause rewrites, simulation failures, or silent wrong results.

### Pitfall 1: Multi-Atom Molecule Overlap Detection Uses Center-of-Mass Only

**What goes wrong:**
When placing THF (14 atoms) or CH₄ (5 atoms), checking overlap at the center-of-mass position misses clashes between the molecule's atoms and existing water molecules. A solute can be placed such that its center has no overlap, but its outer atoms penetrate water molecules.

**Why it happens:**
Ion insertion (v4.0) works with single-atom molecules. Overlap checking at the ion position is sufficient. Naively extending this to multi-atom molecules by checking COM overlap produces structures that fail GROMACS energy minimization with "LINCS warning" or "Shake constraint failure."

**Consequences:**
- GROMACS grompp may succeed, but energy minimization fails or produces NaN energies
- Downstream MD simulations crash or produce unphysical trajectories
- Silent errors: The structure looks valid but contains overlapping atoms

**Prevention:**
1. Load the solute's atom positions relative to its COM (from bundled .gro files)
2. For each placement attempt:
   - Generate random rotation matrix (or use identity for simple cases)
   - Apply rotation to all solute atoms
   - Translate all atoms to target COM position
   - Build cKDTree with ALL solute atoms and ALL existing system atoms
   - Check pairwise distances against VDW radii sum × scale factor (default 0.57)
3. Reject placement if ANY solute atom has overlap with existing atoms

**Detection:**
After insertion, verify:
- All interatomic distances > 0.15 nm (minimum VDW contact)
- Energy minimization with GROMACS converges (energy < 0 kJ/mol)
- No "constraint failure" warnings during initial EM steps

**Phase to address:**
Tab 4 (Solute Insertion) — Phase 1 (overlap detection implementation)

### Pitfall 2: Solute Replaces Ice Water Molecules, Not Liquid Water

**What goes wrong:**
Concentration-based solute insertion calculates the number of molecules to insert and selects water molecules to replace. If the selection logic doesn't respect the ice-liquid boundary (`ice_atom_count`), solutes replace ice crystal water molecules, destroying the crystalline structure.

**Why it happens:**
The liquid phase water molecules start at index `ice_atom_count` in the positions array. Random selection for replacement must be constrained to `[ice_atom_count, total_atoms)`. Off-by-one errors or using `total_atoms` as the lower bound causes ice replacement.

**Consequences:**
- Silent data corruption: Crystal structure has missing water molecules
- Interface structure no longer physically valid
- Downstream analysis produces wrong results (e.g., density profiles, hydrogen bonding)
- No crash, no error message — just wrong science

**Prevention:**
1. Use explicit boundary assertions: `assert water_to_replace_idx >= iface.ice_atom_count`
2. Build a "liquid water molecules" index list: `[i for i in range(ice_nmolecules, total_nmolecules) if mol_index[i].mol_type == "water"]`
3. Random sample from this list, NEVER from all molecules
4. After replacement, verify ice region atom count unchanged

**Detection:**
- After insertion, check `ice_atom_count` positions still contain ice molecules
- Calculate ice region density: should match pre-insertion density
- Visual inspection: ice region should have no gaps or solute molecules

**Phase to address:**
Tab 4 (Solute Insertion) — Phase 1 (replacement logic)

This is a variant of Pitfall #4 in v4.0-PITFALLS.md, now extended to multi-atom molecules.

### Pitfall 3: Hydrate Guest vs. Liquid Solute Naming Confusion

**What goes wrong:**
Tab 2 (Hydrate) places CH₄ in hydrate cages with residue name "CH4". Tab 4 (Solute) places CH₄ in liquid water with residue name "CH4". GROMACS output has duplicate `[ moleculetype ]` sections or ambiguous residue naming, causing grompp failures or simulation analysis errors.

**Why it happens:**
Both tabs use the same molecule (CH₄), but in different contexts. GROMACS allows multiple copies of the same moleculetype, but requires:
1. Unique residue names if the molecules have different topological properties
2. Consistent naming between .gro and .top files
3. Proper ordering in `[ molecules ]` section

**Consequences:**
- GROMACS grompp error: "Duplicate moleculetype definition"
- Ambiguous analysis: Can't distinguish hydrate guests from liquid solutes in trajectory
- Force field issues: Hydrate guests may have different parameters (constraint algorithms) than freely floating solutes

**Prevention:**
1. Use distinct residue names:
   - Hydrate guests: `CH4_HYD`, `THF_HYD`
   - Liquid solutes: `CH4_LIQ`, `THF_LIQ`
2. Update bundled .itp files to use these names in `[ moleculetype ]` directive
3. Map internal molecule types to GROMACS names:
   ```python
   SOLUTE_TO_GROMACS = {
       "ch4_liquid": {"res_name": "CH4_LIQ", "itp_file": "ch4.itp"},
       "thf_liquid": {"res_name": "THF_LIQ", "itp_file": "thf.itp"},
       "ch4_hydrate": {"res_name": "CH4_HYD", "itp_file": "ch4.itp"},
       "thf_hydrate": {"res_name": "THF_HYD", "itp_file": "thf.itp"},
   }
   ```
4. GROMACS export order:
   ```
   SOL (water)
   CH4_HYD (hydrate guests from Tab 2)
   THF_HYD
   CH4_LIQ (liquid solutes from Tab 4)
   THF_LIQ
   CUSTOM_MOL (Tab 5)
   NA, CL (ions from Tab 6)
   ```

**Detection:**
- Generate a structure with both hydrate guests and liquid solutes
- Export to GROMACS format
- Run `gmx grompp` — should succeed without "duplicate moleculetype" errors
- Check .gro file: all CH4_HYD should appear before any CH4_LIQ

**Phase to address:**
Tab 4 (Solute Insertion) — Phase 2 (GROMACS export)

### Pitfall 4: Custom Molecule .gro/.itp Atom Count Mismatch

**What goes wrong:**
User uploads a custom molecule with `.gro` file containing 15 atoms but `.itp` file defining 14 atoms in `[ atoms ]` section. The validation passes (both files exist), but GROMACS export produces invalid topology that fails with "Number of coordinates does not match topology."

**Why it happens:**
QuickIce validates file existence and parses basic structure, but doesn't cross-check atom counts between the two files. Users frequently make mistakes:
- Editing one file but not the other
- Using files from different sources (e.g., .gro from Avogadro, .itp from CGenFF)
- Removing atoms from one file but not updating the other

**Consequences:**
- GROMACS grompp fails with cryptic error message
- User confusion: Both files loaded successfully, why does export fail?
- No clear guidance on which file is wrong

**Prevention:**
1. Parse .gro file to count atoms (line 2 is atom count, verify against actual atom lines)
2. Parse .itp file to count atoms in `[ atoms ]` section
3. Compare counts BEFORE any other processing:
   ```python
   def validate_molecule_files(gro_path: Path, itp_path: Path) -> tuple[bool, str]:
       gro_atoms = parse_gro_atom_count(gro_path)
       itp_atoms = parse_itp_atom_count(itp_path)
       
       if gro_atoms != itp_atoms:
           return False, f"Atom count mismatch: .gro has {gro_atoms} atoms, .itp defines {itp_atoms} atoms"
       
       return True, "Valid molecule files"
   ```
4. Provide specific error message showing both counts
5. Optional: Check residue names match (GRO resname == ITP moleculetype name)

**Detection:**
Unit test with mismatched file pairs. Verify error message clearly states the mismatch.

**Phase to address:**
Tab 5 (Custom Molecule) — Phase 1 (validation)

### Pitfall 5: Custom Molecule Atom Type Conflicts with Existing Force Field

**What goes wrong:**
User uploads custom molecule with atom types `C`, `H`, `O` in the `.itp` file. These generic names conflict with force field atom types already defined for water (TIP4P-ICE uses `OW`, `HW`, `MW`). GROMACS grompp fails with "Atom type X already defined" or uses wrong parameters silently.

**Why it happens:**
GROMACS requires all atom types in the system to be unique OR defined in a shared `atomtypes.itp` file. Custom molecules often use generic atom type names that collide with standard force field definitions. The error is caught by grompp, not by QuickIce's validation.

**Consequences:**
- GROMACS grompp fails with "duplicate atom type" error
- Silent errors: If atom types are redefined, the later definition overwrites the earlier one, leading to wrong VDW parameters
- Users don't understand why validation passed but export failed

**Prevention:**
1. **Option A (Recommended):** Require users to provide atom type definitions in their .itp file:
   ```
   [ atomtypes ]
   ; name  mass  charge  ptype  sigma  epsilon
   custom_C  12.01  0.0  A  0.34  0.36
   custom_H  1.008  0.0  A  0.25  0.13
   ```
   QuickIce copies this section to the main .top file's `[ atomtypes ]` section.

2. **Option B (More complex):** Check for conflicts and auto-rename:
   - Extract atom types from custom .itp
   - Compare against reserved types: `OW`, `HW`, `MW`, `NA`, `CL`, plus any types from bundled .itp files
   - If conflicts found, auto-rename with prefix (e.g., `C` → `user_C`)
   - Update all references in .itp file

3. **Validation:** After parsing custom .itp, check:
   ```python
   RESERVED_ATOM_TYPES = {"OW", "HW", "MW", "NA", "CL", "c3", "hc", ...}  # From bundled .itp files
   custom_types = parse_atom_types_from_itp(itp_path)
   conflicts = custom_types & RESERVED_ATOM_TYPES
   if conflicts:
       raise ValidationError(f"Atom types {conflicts} conflict with reserved types. "
                             "Please rename or provide [ atomtypes ] section.")
   ```

**Detection:**
Unit test with custom molecule using generic atom type names. Verify error or auto-rename behavior.

**Phase to address:**
Tab 5 (Custom Molecule) — Phase 1 (validation + GROMACS export)

### Pitfall 6: Missing Bonded Parameters in Custom Molecule .itp

**What goes wrong:**
User uploads custom molecule .itp file with `[ atoms ]` section defining atoms and charges, but missing `[ bonds ]`, `[ angles ]`, or `[ dihedrals ]` sections. QuickIce accepts the file, but GROMACS energy minimization fails with "No default angle types" or "Function not found" errors.

**Why it happens:**
QuickIce validates basic structure (atom count, moleculetype name) but doesn't check for completeness of bonded parameters. Users may generate .itp files from different sources (manual editing, external tools) that don't include all required sections.

**Consequences:**
- GROMACS energy minimization fails
- User confusion: Validation passed, but simulation won't run
- No clear guidance on what's missing

**Prevention:**
1. Parse .itp file and check for required sections:
   ```python
   REQUIRED_ITP_SECTIONS = ["atoms"]  # At minimum
   OPTIONAL_ITP_SECTIONS = ["bonds", "angles", "dihedrals", "impropers"]
   
   def validate_itp_completeness(itp_path: Path) -> tuple[bool, str]:
       sections = parse_itp_sections(itp_path)
       
       if "atoms" not in sections:
           return False, "Missing required [ atoms ] section"
       
       if "bonds" not in sections:
           # Molecules with >1 atom must have bonds
           atom_count = parse_itp_atom_count(itp_path)
           if atom_count > 1:
               return False, "Multi-atom molecule missing [ bonds ] section"
       
       return True, "ITP file has required sections"
   ```

2. Provide warning if molecule has >3 atoms but no `[ angles ]` section (most molecules need angle terms)

3. Better approach: Require users to validate their .itp files with GROMACS before uploading:
   - Add documentation: "Test your .itp file with `gmx grompp` before uploading"
   - Optional: Run `gmx grompp` internally during validation (requires temporary files)

**Detection:**
Unit test with incomplete .itp files (missing bonds, angles). Verify error messages.

**Phase to address:**
Tab 5 (Custom Molecule) — Phase 1 (validation)

### Pitfall 7: Charge Neutrality Violated by Custom Molecule

**What goes wrong:**
User uploads custom molecule with net charge +2e (e.g., a divalent ion or charged organic molecule). Inserting this molecule into a previously neutral system creates a net charge. GROMACS PME (Particle Mesh Ewald) simulations require charge neutrality or produce artifacts.

**Why it happens:**
QuickIce doesn't check custom molecule charge during validation. The user may not realize their .itp file defines a charged species. Or they may upload an ion (e.g., Ca²⁺) as a "custom molecule" without counter-ions.

**Consequences:**
- GROMACS grompp warns: "System has non-zero total charge"
- PME simulations produce artifacts: artificial electric fields, incorrect energies
- Non-physical results that are hard to diagnose

**Prevention:**
1. Calculate net charge from .itp `[ atoms ]` section:
   ```python
   def calculate_net_charge(itp_path: Path) -> float:
       atoms = parse_itp_atoms(itp_path)
       charges = [atom['charge'] for atom in atoms]
       return sum(charges)
   ```

2. If net charge != 0:
   - Warn user: "Custom molecule has net charge of +Xe. System may not be electroneutral."
   - Suggest: "Add counter-ions in Tab 6 (Ion) to neutralize the system."
   - For large net charges (>2e), require explicit user acknowledgment

3. Track cumulative charge after all insertion tabs:
   ```python
   # After Tab 4 (Solute): charge should still be 0
   # After Tab 5 (Custom): charge may be non-zero
   # After Tab 6 (Ion): warn if charge != 0
   total_charge = sum([mol.charge for mol in inserted_molecules])
   if abs(total_charge) > 1e-6:
       logger.warning(f"System has net charge {total_charge:.2f}e. "
                      "GROMACS may require counter-ions for PME simulations.")
   ```

**Detection:**
Unit test with charged custom molecules. Verify warning messages.

**Phase to address:**
Tab 5 (Custom Molecule) — Phase 1 (validation + charge tracking)

### Pitfall 8: PBC Boundary Crossing for Large Molecules During Rotation

**What goes wrong:**
A large custom molecule (e.g., 2 nm diameter) is placed near a box edge. Random rotation causes some atoms to end up outside the periodic box boundaries. The molecule is "broken" across PBC, with some atoms on one side and others on the opposite side.

**Why it happens:**
Rotation around the center-of-mass doesn't account for box boundaries. An atom at position `(box_x - 0.5, y, z)` rotated 180° around COM moves to `(com_x + 0.5, ...)` which may be > `box_x`. GROMACS expects molecules to be whole (all atoms in same periodic image) for bonded interactions.

**Consequences:**
- GROMACS grompp may succeed, but simulation produces LINCS errors
- Bonded interactions calculated across PBC produce huge energies
- Trajectory visualization shows "broken" molecules

**Prevention:**
1. After rotation and translation, check all atom positions:
   ```python
   def wrap_molecule(positions: np.ndarray, cell: np.ndarray) -> np.ndarray:
       """Wrap molecule atoms into [0, box_size) for each dimension."""
       wrapped = positions.copy()
       for dim in range(3):
           # Detect atoms that crossed boundary
           # Shift entire molecule to keep atoms together
           # Use molecule COM as reference
           ...
       return wrapped
   ```

2. Use existing `wrap_molecules_into_box()` from `gromacs_writer.py` (v4.0)

3. Alternative: Reject placements where any atom would cross boundary:
   ```python
   # Before placement, check if molecule fits
   molecule_radius = np.max(np.linalg.norm(solute_atoms_rel, axis=1))
   if (com_x + molecule_radius > box_x or 
       com_x - molecule_radius < 0 or ...):
       reject_placement("Molecule too close to box edge")
   ```

4. For custom molecules, check all internal bonds after wrapping:
   ```python
   for bond in molecule_bonds:
       dist = distance(atom_positions[bond[0]], atom_positions[bond[1]])
       if dist > max_bond_length * 1.5:
           raise ValidationError("Molecule broken across PBC after rotation")
   ```

**Detection:**
Unit test with large molecules placed near box edges. Verify wrapping behavior.

### Pitfall 9: Concentration Calculation Wrong for Multi-Atom Solute Replacement

**What goes wrong:**
The formula `n_solute = concentration × liquid_volume × NA` calculates the number of solute molecules to insert. But each solute molecule replaces ONE water molecule (not multiple). For large solutes (THF has 14 atoms vs water's 4), the concentration may be higher than intended if not accounting for the volume occupied by the solute.

**Why it happens:**
Concentration in mol/L is defined as moles of solute per liter of solution. When replacing water molecules with solutes, the solution volume changes slightly. For dilute solutions (< 0.1 M), this effect is negligible. For higher concentrations, it matters.

**Consequences:**
- Actual concentration differs from user input
- May exceed solubility limits, causing GROMACS instability
- User confusion: "I asked for 1 M but density measurement shows 1.2 M"

**Prevention:**
1. For low concentrations (< 0.5 M), use simple formula:
   ```python
   n_solute = int(round(concentration_mol_L * liquid_volume_nm3 * 1e-24 * AVOGADRO))
   ```

2. For high concentrations, iterate to account for volume change:
   ```python
   # Each solute replaces 1 water molecule
   # Volume change ≈ (solute_volume - water_volume) per molecule
   # For THF in water: volume change ≈ 0.02 nm³ per molecule
   ```

3. Provide feedback: "Calculated N molecules for concentration X M. Actual concentration may differ slightly due to molecular volume."

4. Warn if concentration > 1 M: "High concentration. Consider energy minimization before production MD."

**Detection:**
Unit test with high concentrations. Compare calculated vs. expected values.

**Phase to address:**
Tab 4 (Solute Insertion) — Phase 1 (concentration calculation)

### Pitfall 10: Rotation Matrix Not Orthogonal for Custom Molecule Placement

**What goes wrong:**
User provides a rotation matrix for custom molecule placement (Tab 5) that is not orthogonal (det ≠ 1). This distorts the molecule geometry, stretching bonds and angles. GROMACS energy minimization fails or produces unrealistic structures.

**Why it happens:**
Users may construct rotation matrices incorrectly (e.g., from non-orthogonal vectors, by scaling a valid rotation matrix, or by inputting Euler angles incorrectly). The validation doesn't check matrix orthogonality.

**Consequences:**
- Molecule geometry corrupted
- GROMACS energy minimization diverges
- Unphysical bond lengths and angles

**Prevention:**
1. Validate rotation matrix orthogonality:
   ```python
   def is_valid_rotation_matrix(R: np.ndarray, tol: float = 1e-6) -> bool:
       """Check if R is a valid 3D rotation matrix (orthogonal, det=1)."""
       if R.shape != (3, 3):
           return False
       # Check orthogonality: R^T * R = I
       should_be_identity = R.T @ R
       if not np.allclose(should_be_identity, np.eye(3), atol=tol):
           return False
       # Check determinant = 1 (not -1, which is reflection)
       if abs(np.linalg.det(R) - 1.0) > tol:
           return False
       return True
   ```

2. If matrix invalid, reject with error: "Rotation matrix must be orthogonal (det = 1)."

3. Alternative: Accept Euler angles instead of rotation matrix (more user-friendly):
   ```python
   def euler_to_rotation_matrix(phi: float, theta: float, psi: float) -> np.ndarray:
       """Convert Euler angles (radians) to rotation matrix."""
       # ZXZ convention or other standard convention
       ...
       return R
   ```

**Detection:**
Unit test with non-orthogonal matrices. Verify rejection with clear error.

**Phase to address:**
Tab 5 (Custom Molecule) — Phase 2 (custom placement mode)

### Pitfall 11: GRO File Parser Breaks on Non-Standard Formatting

**What goes wrong:**
User uploads .gro file with non-standard formatting (e.g., extra spaces, different decimal places, missing velocities, different residue number width). The parser fails or produces wrong coordinates.

**Why it happens:**
GRO format has fixed-width columns, but variations exist:
- Residue numbers may be > 99999 (wraps to 0 or uses > 5 columns)
- Atom names may have leading/trailing spaces
- Coordinates may have different precision (3 vs 4 decimal places)
- Velocities may be omitted

**Consequences:**
- Parser crashes or returns wrong atom positions
- User confusion: "My .gro file works in GROMACS but not QuickIce"
- Validation rejects valid files

**Prevention:**
1. Use robust parsing with fallbacks:
   ```python
   def parse_gro_line(line: str) -> dict:
       """Parse GRO atom line with flexible formatting."""
       # Standard GRO format:
       # resid(5) resname(5) atomname(5) atomid(5) x(8) y(8) z(8) [vx(8) vy(8) vz(8)]
       
       # Try standard fixed-width parsing first
       try:
           resid = int(line[0:5].strip())
           resname = line[5:10].strip()
           atomname = line[10:15].strip()
           x = float(line[20:28])
           ...
       except (ValueError, IndexError):
           # Fallback: split by whitespace
           parts = line.split()
           ...
   ```

2. Reference GROMACS source code for parsing edge cases

3. Test with various GRO file formats from different sources (Avogadro, PACKMOL, etc.)

**Detection:**
Unit test with non-standard .gro files (large residue numbers, missing velocities, etc.)

**Phase to address:**
Tab 5 (Custom Molecule) — Phase 1 (validation)

### Pitfall 12: ITP Parser Fails on Force Field Variations

**What goes wrong:**
User uploads .itp file from a different force field (e.g., CHARMM instead of OPLS-AA) with different section formats, parameter conventions, or directive names. The parser fails or produces wrong topology.

**Why it happens:**
ITP format is standardized by GROMACS, but force fields have variations:
- Section names may differ (`[ bondtypes ]` vs `[ bonds ]`)
- Parameter ordering may differ
- Some sections may be missing or have different names
- Comment styles may vary

**Consequences:**
- Parser crashes or returns incomplete topology
- Missing bonded parameters cause simulation failures
- User confusion about force field compatibility

**Prevention:**
1. Support common ITP variations:
   - GROMACS standard format (OPLS-AA, AMBER, CHARMM)
   - Both `[ bonds ]` and `[ bondtypes ]` naming
   - Flexible column ordering (key-based parsing instead of position-based)

2. Validate against GROMACS documentation:
   - Required sections: `[ moleculetype ]`, `[ atoms ]`
   - Optional sections: `[ bonds ]`, `[ angles ]`, `[ dihedrals ]`, `[ impropers ]`, `[ pairs ]`

3. Provide warning if force field detected: "ITP file appears to use CHARMM format. Ensure compatibility with TIP4P-ICE water model."

4. Recommend: "Use GAFF2 (General Amber Force Field) for organic molecules with TIP4P-ICE water."

**Detection:**
Unit test with .itp files from different force fields.

**Phase to address:**
Tab 5 (Custom Molecule) — Phase 1 (validation)

### Pitfall 13: Tab Reordering Breaks Cross-Tab Data Flow

**What goes wrong:**
v4.5 moves Ion insertion from Tab 4 to Tab 6. Code that accesses "the ion tab" via index `3` (0-based) now points to the wrong tab. Data flow dependencies between tabs are broken.

**Why it happens:**
Tab indices were hardcoded in v4.0. The reordering requires updating all references to tab positions and data flow.

**Consequences:**
- UI shows wrong content in wrong tabs
- Data from Ion tab (now Tab 6) doesn't flow to export correctly
- Tests fail that expect specific tab order

**Prevention:**
1. Use named constants for tab indices:
   ```python
   class TabIndex:
       ICE = 0
       HYDRATE = 1
       INTERFACE = 2
       SOLUTE = 3      # NEW
       CUSTOM = 4      # NEW
       ION = 5         # MOVED from 3
   ```

2. Never hardcode tab indices in logic:
   ```python
   # WRONG
   if current_tab == 3:
       process_ion_data()
   
   # RIGHT
   if current_tab == TabIndex.ION:
       process_ion_data()
   ```

3. Update all ViewModel signals and slots to use new tab order

4. Test cross-tab data flow explicitly:
   - Interface (Tab 3) → Solute (Tab 4): liquid volume must be available
   - Solute (Tab 4) → Custom (Tab 5): molecule count affects placement
   - Custom (Tab 5) → Ion (Tab 6): total system charge tracking

**Detection:**
Integration test: Generate structure using all tabs in order, export, verify correct output.

**Phase to address:**
Architecture (Phase 0) — before any Tab 4/5/6 implementation

### Pitfall 14: GROMACS Export Order Changes Break Analysis Tools

**What goes wrong:**
v4.0 exports in order: SOL, hydrate_guests, NA, CL. v4.5 adds solutes and custom molecules, changing order to: SOL, hydrate_guests, liquid_solutes, custom_molecules, NA, CL. Downstream analysis tools that expect specific ordering produce wrong results.

**Why it happens:**
Analysis scripts often assume fixed ordering of molecules in the topology. Adding new molecule types shifts indices, breaking assumptions.

**Consequences:**
- Analysis tools select wrong atom groups
- Trajectory visualization shows wrong colors/representations
- User scripts break when upgrading from v4.0 to v4.5

**Prevention:**
1. Document export order clearly in user guide:
   ```
   GROMACS Export Order (v4.5):
   1. SOL — All water molecules (ice + liquid)
   2. Hydrate guests — CH4_HYD, THF_HYD (from Tab 2)
   3. Liquid solutes — CH4_LIQ, THF_LIQ (from Tab 4)
   4. Custom molecules — User-defined (from Tab 5)
   5. Ions — NA, CL (from Tab 6)
   ```

2. Generate GROMACS index file (`.ndx`) with named groups:
   ```
   [ Ice ]
   1-5000
   [ Liquid ]
   5001-10000
   [ Solutes ]
   10001-10050
   ```

3. Maintain backward compatibility: Provide v4.0-style export option if no new molecule types used

**Detection:**
Compare GROMACS export output between v4.0 and v4.5 for same structure.

**Phase to address:**
GROMACS Export (Phase 2) — after all insertion tabs complete

### Pitfall 15: Residue Number Wrapping in GRO Export

**What goes wrong:**
Large systems (>99,999 atoms) have atom indices that exceed GRO format's 5-digit limit. Residue numbers wrap or overflow, causing parsing errors in GROMACS tools.

**Why it happens:**
GRO format uses 5 positions for residue number. Values > 99999 wrap to 0 or cause formatting issues.

**Consequences:**
- GROMACS tools fail to read .gro file
- Visualization tools display wrong residue numbers

### Pitfall 16: No Validation for Custom Molecule Bond Geometry

**What goes wrong:**
User uploads .gro file with unrealistic bond lengths (e.g., 0.5 nm C-H bond instead of 0.11 nm). QuickIce accepts the file, but GROMACS energy minimization diverges.

**Why it happens:**
QuickIce validates file format, not chemical realism. Users may generate .gro files incorrectly.

**Consequences:**
- Energy minimization fails or takes many steps
- User doesn't know the initial structure is problematic

**Prevention:**
1. Optional: Check bond lengths against reasonable ranges:
   ```python
   TYPICAL_BOND_LENGTHS = {
       ("C", "H"): (0.10, 0.12),  # nm
       ("C", "C"): (0.13, 0.16),
       ("C", "O"): (0.12, 0.15),
       ...
   }
   
   def validate_bond_geometry(positions, atom_names, bonds):
       for i, j in bonds:
           dist = distance(positions[i], positions[j])
           atom_pair = tuple(sorted([atom_names[i], atom_names[j]]))
           if atom_pair in TYPICAL_BOND_LENGTHS:
               min_len, max_len = TYPICAL_BOND_LENGTHS[atom_pair]
               if dist < min_len or dist > max_len:
                   logger.warning(f"Unusual bond length: {atom_names[i]}-{atom_names[j]} = {dist:.3f} nm")
   ```

### Pitfall 17: User Confusion About Random vs. Custom Placement Modes

**What goes wrong:**
Tab 5 offers two modes: "Random" (random position + random orientation) and "Custom" (user-specified COM + rotation). Users misunderstand which mode they're using, leading to unexpected results.

**Why it happens:**
The UI doesn't clearly communicate the difference. Users may think "Custom" means they can choose the molecule type, not the placement.

**Consequences:**
- Users get random placement when they wanted specific positions
- Users provide COM coordinates but don't realize they need to also specify rotation
- Support questions and frustration

**Prevention:**
1. Clear UI labels:
   - "Random placement: Insert molecules at random positions in liquid"
   - "Custom placement: Specify exact position and orientation for each molecule"

2. Provide default rotation matrix (identity) for custom mode:
   ```python
   if custom_mode and rotation_matrix is None:
       rotation_matrix = np.eye(3)  # No rotation
       logger.info("Using default orientation (no rotation)")
   ```

3. Add tooltip: "Rotation matrix: 3×3 orthogonal matrix defining molecule orientation. Use identity matrix [[1,0,0],[0,1,0],[0,0,1]] for no rotation."

4. Add visual preview (stretch goal): Show where molecule will be placed before committing

**Detection:**
User testing with clear mode selection feedback.
