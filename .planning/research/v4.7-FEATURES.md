## Feature Details: Verified Behavior

### 1. Filled Ice Structures

**What filled ices are scientifically:** Dense ice phases where guest molecules occupy **channels** (not cages) in the water lattice. Formed at high pressures (>0.5 GPa). The water framework is a dense ice lattice (Ih, Ic, or II-like) with guests squeezed into interstices. **Filled ices have NO polyhedral cages** вЂ” GenIce2's `assess_cages` correctly returns "no cages detected."

**Verified generation results (Test 26):**

| Lattice | Water Mols | Cell Type | Box Values | Guest Placement Method |
|---------|------------|-----------|------------|----------------------|
| c0te | 6 | Triclinic (Оі=120В°) | 9 values | `spot_guests={0: 'H2', 1: 'H2', 2: 'H2'}` |
| c1te | 36 | Triclinic | 9 values | `spot_guests={0: 'H2'}` |
| c2te | 32 | Orthorhombic | 3 values | `spot_guests={0: 'H2'}` |
| ice1hte | 16 | Orthorhombic | 3 values | `spot_guests={0: 'H2'}` |
| sTprime | 24 | Orthorhombic | 3 values | No `cagepos` attribute вЂ” `spot_guests` fails silently |

**Key finding: Filled ices use `spot_guests` (GenIce constructor parameter), NOT `-g` flag (generate_ice parameter).** The `-g` flag references cage type numbers (12, 14, 16), which filled ices don't have. The `spot_guests` parameter takes `{cage_index: 'guest_name'}` where `cage_index` maps to the lattice's `cagepos` list.

**Critical constraint: `sTprime` has no `cagepos` attribute.** Its attributes are only `['bondlen', 'cell', 'coord', 'density', 'waters']`. Guest placement requires either:
1. Post-hoc manual placement (extract channel centers from water framework geometry)
2. CIF import with `assess_cages=True` (but filled ices have no cages в†’ returns nothing)
3. Adding `cagepos`/`cagetype` to the sTprime lattice module

**Recommendation:** For v4.7, generate `sTprime` without guests (empty framework). Users who need Hв‚‚ in sT' channels must use the CLI directly.

### 2. COв‚‚ Guest Molecule

**Verified generation (Test 6):** sI hydrate with COв‚‚ in all cages produces 208 atoms:
- 46 water molecules (184 atoms: OWГ—46, HW1Г—46, HW2Г—46, MWГ—46)
- 8 COв‚‚ molecules (24 atoms: CГ—8, OГ—16)
- Residue name: "CO2"
- Cage mapping: `parse_guest(guests, "12=co2")` + `parse_guest(guests, "14=co2")`

**Scientific use case:** COв‚‚-CHв‚„ swap (inject COв‚‚ into CHв‚„ hydrate to produce CHв‚„ gas + COв‚‚ storage). This is the #1 industrial application of clathrate hydrates.

**Molecule details:**
- GenIce2 molecule: `safe_import('molecule', 'co2').Molecule()`
- `labels_` = `['C', 'O', 'O']` (3 atoms)
- `sites_` shape = (3, 3)
- `name_` = `'CO2'` (3 chars вЂ” within 5-char GRO limit)

**QuickIce code impact:**
- `_build_molecule_index` must recognize "CO2" residue (like THF)
- `CO2_ATOMTYPE_NAMES` already defined in `gromacs_writer.py` as `['c_2', 'o_2']`
- `MOLECULE_TO_GROMACS` already has `co2` entry with `itp_file='co2.itp'`
- **MISSING:** `co2.itp` and `co2_hydrate.itp` files in `quickice/data/`

### 5. Mixed Cage Occupancy (Binary Clathrates)

**Verified generation (Test 12):** sI with CHв‚„+COв‚‚ mixed occupancy:
- `parse_guest(guests, "12=co2*0.5+me*0.5")` в†’ 50% COв‚‚ + 50% CHв‚„ in small cages
- Result: 7 COв‚‚ molecules + 1 Me molecule in small cages (stochastic)
- Large cages: 100% COв‚‚ (24 atoms from 8 COв‚‚ molecules)

**Occupancy math verification (Test 32):** 2Г—2Г—2 supercell sI with 50% small cage occupancy:
- Expected: 16 small cages Г— 50% = 8 CHв‚„, 48 large cages Г— 100% = 48 CHв‚„, total = 56 CHв‚„
- **Actual: 56 CHв‚„** вЂ” exact match вњ…

**GenIce2 syntax reference:**
| Syntax | Meaning |
|--------|---------|
| `-g 12=ch4` | 100% CHв‚„ in type-12 cages |
| `-g 12=ch4*0.5` | 50% CHв‚„ in type-12 cages |
| `-g 12=co2*0.6+me*0.4` | 60% COв‚‚ + 40% CHв‚„ in type-12 cages |
| `-g 14=co2` | 100% COв‚‚ in type-14 cages |
| `spot_guests={0: 'me', 2: 'co2'}` | CHв‚„ in cage #0, COв‚‚ in cage #2 (per-cage, not per-type) |

**sH cage type complication:** sH uses non-standard cage type naming:
- `12` = 5В№ВІ small cages (6 per UC)
- `12_1` = 4Ві5вЃ¶6Ві medium cages (4 per UC)  
- `20` = 5В№ВІ6вЃё large cages (2 per UC)

The `_1` suffix differentiates the medium cage from the small cage (both are 12-hedra but with different face topology). The `-g` flag syntax must use the full cage type name: `12_1`.

### 6. Water Model Selector

**Verified water models (Test 3):**

| Model | GenIce2 Name | Sites | Atom Labels | Atoms/Water | Impact on Output |
|-------|-------------|-------|-------------|-------------|-----------------|
| TIP3P | `tip3p` / `3site` | 3 | O, H, H | 3 | 178 atoms per sI UC (vs 224 for TIP4P) |
| SPC/E | `spce` | 3 | Ow, Hw, Hw | 3 | 178 atoms; different naming convention |
| TIP4P | `tip4p` / `4site` | 4 | OW, HW1, HW2, MW | 4 | 224 atoms per sI UC (current default) |
| **TIP4P/Ice** | `ice` | 4 | OW, HW1, HW2, MW | 4 | Same geometry as tip4p; QuickIce's force field |
| TIP5P | `tip5p` / `5site` | 5 | OW, HW1, HW2, MW1, MW2 | 5 | 270 atoms per sI UC |
| 6-site (NvdE) | `NvdE` / `6site` | 6 | O, H, H, M, L, L | 6 | 306 atoms per sI UC |
| 7-site | `7site` | 7 | OW, HW1, HW2, MW1, MW2, LW1, LW2 | 7 | 342 atoms per sI UC |

**Critical impact on downstream code:**
1. **`_build_molecule_index`** вЂ” currently only handles TIP4P (OW, HW1, HW2, MW) and 3-site (O, H, H). Adding TIP5P (MW1, MW2), 6-site (M, L, L), SPC/E (Ow, Hw, Hw) requires new patterns.
2. **GROMACS export** вЂ” each water model needs its own `.itp` file. Currently only `tip4p-ice.itp` is bundled.
3. **Interface pipeline** вЂ” water_filler.py uses TIP4P geometry. Switching models breaks filler placement.
4. **Atom count validation** вЂ” `WATER_ATOMS_PER_MOLECULE = 4` hardcoded. Must be per-model.

**Recommendation for v4.7:** Start with TIP4P-ICE (default, current), TIP4P/2005, TIP3P, SPC/E. These 4 cover the most common hydrate MD studies. Defer 5/6/7-site models.

### 7. Depol Mode (Empty Cages)

**Verified generation (Test 9):** Empty sI hydrate with different depol modes:

| Depol Mode | Result | Notes |
|-----------|--------|-------|
| `strict` | 184 atoms (46 water mols) | Forces net dipole to near zero; standard for bulk hydrates |
| `optimal` | 184 atoms (46 water mols) | Relaxes dipole constraint; needed for semiclathrates with ions |
| `None` | 184 atoms (46 water mols) | No depolarization; random H orientations |

**Ice XVI (empty sII):** 544 atoms = 136 water Г— 4 sites. Verified.
**Ice XVII (empty C0):** 48 atoms = 12 water Г— 4 sites. Verified.

**Scientific use cases for depol mode:**
- `strict`: Standard for bulk clathrates without ions
- `optimal`: Required for semiclathrates where ions prevent strict depolarization; also useful for structures with fixed edges (hydrogen-ordered ices)
- Empty cages (depol mode): Ice XVI and XVII studies вЂ” mechanical stability of empty frameworks, guest removal simulations

**QuickIce change:** Currently hardcoded `depol='strict'` in `_run_via_api()`. Add a `depol_mode` field to `HydrateConfig` and pass it through.

### 8. Custom Guest in Hydrate

**Verified end-to-end (Test 16):** Custom ethanol molecule injected via `sys.modules`:
- Created `EthanolMolecule` subclass of `genice2.molecules.Molecule`
- Injected into `sys.modules["genice2.molecules.eto"]`
- Generated sII hydrate with custom ethanol in 8 large cages
- Result: 136 water + 8 ethanol = 568 total atoms вњ…

**Two methods available (from R1-SAFE-IMPORT-FEASIBILITY.md):**

| Method | How | Pros | Cons |
|--------|-----|------|------|
| `sys.modules` injection | Create `types.ModuleType` + `Molecule` subclass, register before GenIce call | Full programmatic control; handles multi-char atom labels (CA, CB) | Must clean up `sys.modules`; name collision risk |
| `mol[file=path]` syntax | Write `.mol` file, pass in `guests` dict | GenIce-native; auto Г…в†’nm conversion | Limited to element symbols; no multi-char atom labels |

**Recommendation:** Use `sys.modules` injection as primary (matches user's `.gro`/`.itp` atom naming). Fall back to `mol[file=path]` for simple molecules.

**User workflow expectations:**

```
1. User uploads .gro + .itp files for custom guest molecule
2. QuickIce validates:
   - GRO file is parseable (atom names + positions)
   - ITP file is parseable (moleculetype name + atom types + charges)
   - Residue name в‰¤ 3 chars (for _H suffix: total в‰¤ 5 chars GRO limit)
   - comb-rule = 2 (Lorentz-Berthelot, AMBER/GAFF2 convention)
3. QuickIce converts GRO в†’ Molecule (sites_ centered at origin in nm)
4. QuickIce registers Molecule in sys.modules
5. User selects target cage type (small/large) and occupancy %
6. QuickIce generates hydrate with custom guest
7. QuickIce exports GROMACS files:
   - Custom guest ITP with _H suffix
   - atomtypes commented out (moved to main .top)
   - Main .top includes all guest ITPs
```

**GRO 5-char residue name constraint:**
- GenIce2 outputs guest residue names as-is (no truncation)
- GROMACS 2023.5 REJECTS files with >5-char residue names (fatal error)
- With QuickIce convention `_H` suffix: base name must be в‰¤3 chars
- Example: "CO2" в†’ "CO2_H" (5 chars вњ…), "ETHANOL" в†’ "ETHANOL_H" (8 chars вќЊ)
- **Validation required:** QuickIce must check and warn/truncate before generation
